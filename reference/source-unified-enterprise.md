# ステップ1：レポート全体のアジェンダ

## 1.1 中心命題

エンタープライズにAIエージェントを組み込む中心課題は「**AIを賢くすること**」ではなく、「**企業の既存のID・権限・責任・業務プロセス・監査・データ境界・組織構造の中に、新しい実行主体を安全に参加させること**」である。

> エンタープライズAIエージェントとは、単なるチャットUIではなく、**組織の権限構造を忠実に投影し、既存システムの真実（System of Record）を壊さずに束ね、すべての行為を企業横断で監査・統治できる形に閉じ込めた、管理可能・監査可能・権限制御された「デジタル業務主体」**である。エージェントの知能は前提条件にすぎず、勝負は「誰の権限で・どのデータを・どう守って・誰の責任で」動かすかにある。

## 1.2 統合の評価サマリと方針

## 1.3 AIエージェントは「企業内の実行主体」である

一般的なAIチャットは「回答主体」だが、エンタープライズエージェントは「業務実行主体」であり、企業システム上の**一級オブジェクト**として定義・管理する。

```text
EnterpriseAgent
- agent_id / owner_department / business_purpose
- allowed_users / allowed_projects / allowed_tools / allowed_data_domains
- risk_tier / approval_policy / memory_scope
- audit_policy / cost_budget / incident_owner
- model_version / prompt_version / policy_version
```

### エージェント分類学（役割の型）

| 分類 | 役割 | 例 |
|---|---|---|
| Employee Copilot | 従業員個人の業務支援 | メール下書き、資料作成、予定調整 |
| Department Agent | 部門業務の実行支援 | HR / Sales / Finance Agent |
| Project Agent | プロジェクト単位の作業支援 | PMO Agent、Issue Triage Agent |
| Process Agent | 業務プロセスの自動実行 | 稟議、請求、返金、オンボーディング |
| Customer-facing Agent | 顧客との対話・サポート | CS Agent、EC Agent |
| Governance Agent | 監査・リスク・品質管理 | Compliance / Security Review Agent |
| Platform Agent | 社内開発・運用支援 | SRE / Data / Dev Agent |

## 1.4 企業構造をアーキテクチャに反映する（組織グラフとスコープ）

企業はフラットなユーザー集合ではない。権限・メモリ・ログ・評価・コストは、この階層に紐づけて設計する。

```text
Company > Business Unit > Department > Section/Group > Team > Project > Subproject > Work Item
                                                                 └ Daily Operations
```

| スコープ | 対象 | 共有範囲 |
|---|---|---|
| User | 個人の嗜好・作業スタイル | 本人のみ |
| Team | チームルール・定例・タスク | チーム |
| Project | 決定事項・背景・成果物 | プロジェクト＋上位 |
| Department | 業務標準・KPI・手順・予算 | 部門 |
| Company | 全社規程・経営情報・全社ナレッジ | 全社 |
| Customer | 顧客別契約・問い合わせ・利用履歴 | 担当者・許可者 |

この構造を、Workday（組織・職位・レポートライン）、Okta/Entra（グループ）、Linear/Asana/Jira/Notion（プロジェクト）から名寄せした単一の**組織グラフ（Org Graph）**として持ち、すべての面がスコープ・委譲・共有・承認の根拠にする。

## 1.5 全体アーキテクチャ：7面と2つの横断軸

```text
[面1] 体験・ゲートウェイ        従業員チャネル | 顧客チャネル（信頼境界で分離）
   ↓                          Enterprise Agent Gateway（認証/分類/レート/監査入口）
[面2] 制御・ガバナンス          レジストリ | モデルGW | ポリシー/憲法 | 評価 | コスト | 事故対応
   ↓
[面3] アイデンティティ・信頼 ★   IdP連携 | OBO/トークン交換 | ワークロードID | PDP(ABAC/ReBAC)/PEP
   ↓                          Permission Mirror | 権限忠実(最小合成) | JIT資格情報 | ゼロトラスト
[面4] 実行・オーケストレーション   Hub&Spoke | RACI | Risk-Tier自律 | 承認チェーン | Saga | 永続WF
   ↓                          Command Envelope | SoR書き込み境界 | 業務キュー | イベント駆動 | Project Twin
[面5] 知識・メモリ・コンテキスト   権限認識RAG | Context Mesh | 正規オブジェクト/知識グラフ
   ↓                          スコープ記憶 | 目的限定コンテキスト | DLP | 揮発セキュアバス
[面6] 統合・ツール              Tool/MCP Gateway | SaaS Adapter | レート調停 | 既存iPaaS再利用
   ↓
[System of Record]  Salesforce/ServiceNow/Workday/Slack/MS365/Box/Jira/Zendesk/Shopify/バクラク/独自
   ↓
[面7] 観測・評価・監査          Observability Lake | 統一監査・系譜（人＋エージェント＋システム）
─ 横断軸1：組織グラフ ─ 全面がスコープ・委譲・承認を組織構造から一貫して導く
─ 横断軸2：ゼロトラスト/監査 ─ 全呼び出しを「人＋エージェント＋システム」の三者で認可・記録
```

## 1.6 標準・フレームワークとの整合

エンタープライズでは、これらを「アプリ設計の指針」でなく「**企業アーキテクチャ設計の制約**」として扱う。

- **NIST AI RMF（Generative AI Profile）**：生成AI固有リスクの特定と管理アクション設計の枠組み。
- **OWASP Top 10 for LLM Applications**：Prompt Injection / Sensitive Information Disclosure / Excessive Agency / Unbounded Consumption 等を主要リスクとして整理。
- **NIST SP 800-207 Zero Trust Architecture**：境界でなく主体・資産・リソース中心の保護。
- **OIDC / SCIM**：既存ID標準（認証・プロビジョニング）の上に乗る。独自ID管理を乱立させない。
- **OAuth 2.0 Token Exchange（RFC 8693）**：委譲・代理実行（OBO）の標準。
- **OPA/Rego・Cedar**：Policy-as-Code による決定論的認可。
- **MCP（Model Context Protocol）**：ツール接続の標準（ただし企業ではGateway経由に統制）。
- **CloudEvents**：SaaS/社内イベントの共通記述。
- **OpenTelemetry GenAI semantic conventions**：エージェント・モデル・ツール呼び出しの標準観測。

# ステップ2：各アーキテクチャパターンの項目設計

## 2.1 共通記述スキーマ

各パターンを以下で統一記述する。エンタープライズでは「落とし穴（アンチパターン）」が事故に直結するため、明示項目とする。

- **概要**：何であるかの一文要約。
- **設計**：構造・データフロー・状態遷移・実装上の要点。
- **解決する企業課題**：どのエンタープライズ固有の力（漏洩・サイロ・動的文脈・監査・コスト）に応えるか。
- **向き／不向き**：採用が効く条件と、害・過剰になる条件。
- **要素技術・既存システム連携**：代表技術・標準・対象SaaS。
- **落とし穴／選定の勘所**：典型的な失敗と回避の指針。

## 2.2 面（カテゴリ）設計

| 面 | テーマ | 主眼 | パターン数 |
|---|---|---|---|
| 面1 体験・ゲートウェイ (EX) | 入口と提供面 | 仕事のある場所に届け、入口で統制する | 3 |
| 面2 制御・ガバナンス (GV) | 統治・統制 | 一元レジストリ・モデル統制・評価・コスト・事故対応 | 10 |
| 面3 アイデンティティ・信頼 (ID) | 権限の忠実な伝播 ★最難関 | 誰の権限で動くかを保証する | 8 |
| 面4 実行・オーケストレーション (RT) | 分業・実行・自動化 | 責任分担・自律度・副作用・長尺・イベント | 11 |

# ステップ3：アーキテクチャパターン本体

# 面1：体験・ゲートウェイ（EX）

## EX-1. Enterprise Agent Gateway（統一フロントドア）
- **概要**：すべてのエージェント要求が通る単一の入口。テナント/ユーザー認証、要求分類、リスク分類、レート制御、監査エントリを一元適用する。
- **設計**：`従業員/顧客 → Gateway（認証・分類・リスク判定・レート・監査入口）→ 実行面`。チャネル（Slack/Web/SaaS埋め込み）を吸収し、本人IDと相関IDを後段へ伝播。
- **解決する企業課題**：入口が分散し統制・監査・容量管理が崩れる問題。数万人規模のバースト吸収。
- **向き／不向き**：複数チャネル・大規模なら必須／単一PoCには過剰。
- **要素技術・連携**：API Gateway、OIDC、相関ID、リスク分類器。
- **落とし穴**：Gatewayを素通しプロキシにして認可・監査を後段任せにすること。入口は統制点である。

## EX-2. 業務埋め込み vs 独立ポータル
- **概要**：仕事のある場所（Slack/Teams/Salesforce/ServiceNow画面）に埋め込むか、横断・探索用の独立ワークベンチにするかを使い分ける。
- **設計**：日常文脈に近いタスクは埋め込み、横断・長尺・承認付きタスクは独立ワークベンチ（計画・進捗・根拠・承認・差分を一画面）。
- **解決する企業課題**：ツールスイッチの摩擦で使われない問題。
- **向き／不向き**：現場定着を狙う全社展開に有効／単機能には不要。
- **要素技術・連携**：Slack/Teamsアプリ、Salesforce/ServiceNow拡張、Webワークベンチ。
- **落とし穴**：独立ポータルだけ作り誰も開かないこと。

## EX-3. チャネル非依存フロントドア
- **概要**：同一エージェントを複数チャネルから呼べるようにし、本人ID・スコープ・履歴・監査を一貫させる。
- **設計**：チャネルアダプタで入口差を吸収し、背後は同一の実行・権限・メモリ・監査。
- **解決する企業課題**：チャネル別作り込みで挙動・権限・履歴が分裂する問題。
- **向き／不向き**：マルチチャネル提供／単一チャネルには不要。
- **要素技術・連携**：チャネルアダプタ、統一セッション。
- **落とし穴**：チャネル間で本人IDの引き継ぎが切れ権限が緩むこと。

# 面2：制御・ガバナンス（GV）

## GV-1. Enterprise Agent Control Plane（レジストリ／ライフサイクル）
- **概要**：社内の全エージェントを登録・所有者明示・審査・版管理・廃止まで一元統制する制御プレーン。
- **設計**：各エージェントに owner / purpose / allowed_users / allowed_tools / data_domains / risk_tier / approval_policy / audit_policy / cost_budget を付与（1.3のオブジェクト）。新規・変更は審査（セキュリティ・法務・データ保護）を経て公開。未登録（シャドーAI）は実行基盤・モデルGWで遮断。
- **解決する企業課題**：野良エージェント乱立、責任者不明、部門重複、過剰権限、変更の監査不能。
- **向き／不向き**：エージェントが3個を超え複数チームが使い始めたら必須／個人PoCには不要。
- **要素技術・連携**：Agent Registry、Policy-as-Code、CMDB(ServiceNow)、サービスカタログ。
- **落とし穴**：台帳止まりにし実行制御と結びつけないこと。登録＝実行許可のゲートにする。

## GV-2. Agent Catalog & Marketplace（社内カタログ）
- **概要**：審査済みエージェント/スキル/ツールを社内アプリストアとして提供し、発見・申請・利用・複製を可能にする。
- **設計**：レジストリ(GV-1)を土台に、目的・所有者・データアクセス・リスク階層・コスト・品質スコアを表示。部署はテンプレ(GV-3)から派生。
- **解決する企業課題**：どんなエージェントがあるか不明、部門重複、無審査利用、申請の属人化。
- **向き／不向き**：複数部門展開／単一機能には不要。
- **要素技術・連携**：カタログ、利用申請、Usage Analytics、品質レーティング。
- **落とし穴**：審査を緩め「何でも公開」にし品質・安全がばらつくこと。

## GV-3. Department Agent Factory（役割テンプレート工場）
- **概要**：部門/役割ごとのエージェントを、標準テンプレート（ポリシー＋コネクタ＋評価のパック付き）から安全に量産する。
- **設計**：HR/Sales/CS/Finance/Legal/Eng/Security の各テンプレに、役割相応のツール・データ・権限・ポリシーを既定化。入社・異動で役割が変わると付与/剥奪が追従。
- **解決する企業課題**：都度開発による設計ばらつきとスケール不能。数万人への個別設定の非現実性。
- **向き／不向き**：AI CoE/Platformがあり複数部門展開／部門固有要件が薄い小規模には過剰。
- **要素技術・連携**：テンプレート、ローコードビルダー、Policy/Connector/Eval Pack、Okta/Workdayの役割。
- **落とし穴**：テンプレが粗く過剰権限になること。最小権限(ID-4)で削る。

## GV-4. Industry Policy Pack（業界ポリシーパック）
- **概要**：業界特有の規制・慣習・監査要件を、再利用可能なポリシーパックとして組み込む。
- **設計**：金融/医療/人事/法務/公共/製造 等のパックを定義し、policy-as-code(ID-7)・eval(GV-7)・データ分類・保持・承認ルールへ展開。
- **解決する企業課題**：業界規制をプロンプトで管理して形骸化・属人化する問題、導入審査の遅延。
- **向き／不向き**：規制産業・大企業・グローバル／規制影響が小さい内部支援AIには過剰。
- **要素技術・連携**：Policy Pack、Compliance Rule、GRCツール。
- **落とし穴**：個別エージェントのプロンプトに規制を埋め込むこと。

## GV-5. Central Model Gateway（モデル・ベンダー統制）
- **概要**：全LLM呼び出しを単一GW経由にし、ベンダー承認・データ所在地・PII・コスト・監査を集中制御する。
- **設計**：承認済みモデルのみ許可。データ分類に応じルーティング（極秘＝VPC内/社内推論、一般＝外部API）。DPA・リージョン・保持を強制。本文はストレージ退避＋メタのみ監査。
- **解決する企業課題**：各チームが勝手に外部LLMへ機密送信、ベンダー乱立、コスト不可視。
- **向き／不向き**：全社基盤に必須／単一アプリには軽量化可。
- **要素技術・連携**：LiteLLM/Portkey型GW、Bedrock/Azure OpenAI（リージョン）、社内推論基盤。
- **落とし穴**：迂回ルートを放置すること。egress制御で塞ぐ。

## GV-6. Version Registry（モデル/プロンプト/ツール/ポリシー/索引の版管理）
- **概要**：モデル・プロンプト・ツール・ポリシー・RAG索引・スキーマをバージョン管理し、変更をPR・eval・カナリア・ロールバックの対象にする。
- **設計**：各実行に model/prompt/tool/policy/retrieval_index/schema の各versionを記録。「デプロイ＝挙動変更」をWebデプロイ並みの規律で扱う。
- **解決する企業課題**：小さな変更で挙動が変わる、プロバイダのサイレント更新、監査時の再現不能。
- **向き／不向き**：継続運用する全エージェント／使い捨て実験には不要。
- **要素技術・連携**：Registry、Git、Feature Flag、Canary、Eval Dataset、Rollback。
- **落とし穴**：コードだけ版管理し、プロンプト・モデル・索引を野放しにすること。

## GV-7. Evaluation & Governance Pipeline（評価CI/CD）
- **概要**：品質・安全性・業務適合性を、テストでなく継続評価としてCIと本番の双方で測り続ける。
- **設計**：`変更要求 → オフラインeval → セキュリティeval → ポリシーeval → シャドー → カナリア → 本番監視 → フィードバック`。完全一致でなくルーブリック・LLM-as-judge・特性アサーション・人手レビュー・レッドチーミング。
- **解決する企業課題**：非決定論ゆえ単体テストで品質劣化・挙動変化を検出できない。
- **向き／不向き**：本番AI全般／一時PoCには軽量手動評価。
- **要素技術・連携**：Golden Dataset、LLM-as-a-judge、promptfoo/DeepEval/Braintrust、CI/CD。
- **落とし穴**：技術指標のみで合否判定し業務適合・安全を測らないこと。

## GV-8. Cost Quota & Chargeback（コスト配賦）
- **概要**：トークン・ツール・実行コストをテナント/部門/プロジェクト/ユーザー/エージェント/モデル/タスク種別で計測し、予算・上限・按分を行う。
- **設計**：全呼び出しに cost_center を付与し計上。部門別予算・アラート、上限到達時は縮退（簡易モデル・キャッシュ・人間）。経営向けROIダッシュボード。
- **解決する企業課題**：LLMコストのブラックボックス化、部門の使い過ぎ、マルチエージェントの推論爆発、顧客別採算不明。
- **向き／不向き**：数千〜数万人規模・AI機能を有料提供する企業／小規模PoCには不要。
- **要素技術・連携**：Token Meter、Cost Attribution、Budget/Quota、FinOps、組織グラフ（按分軸）。
- **落とし穴**：コストを「インフラ費」と捉え業務成果に紐づけないこと（→GV-10と対）。

## GV-9. Incident Response & Kill Switch（事故対応・停止）
- **概要**：誤送信・漏洩・誤更新・コスト暴走・インジェクション・ツール暴走に備えた検知・停止・調査・復旧の仕組み。
- **設計**：`検知 → 封じ込め → エージェント/ツール/モデル/テナント単位で停止 → トレース保全 → 影響評価 → 通知 → 是正 → ポストモーテム`。
- **解決する企業課題**：事故時に止められない・原因究明できない・影響範囲が分からない。
- **向き／不向き**：本番AI全般に必須。
- **要素技術・連携**：Kill Switch、Circuit Breaker、Runbook、Audit Snapshot、Replay、Access Revocation、SIEM。
- **落とし穴**：全体停止しかできないこと。粒度別（モデル/エージェント/ツール/テナント）に止められるよう設計。

## GV-10. Two-Layer Value Measurement（生産性×経営KPI）
- **概要**：「従業員は効率化、経営は企業価値」の二目的を別々の指標で計測し、投資対効果を可視化する。
- **設計**：個人/チーム層＝処理時間短縮・自己解決率・満足度。経営層＝リードタイム・売上/コスト影響・採用/離職効率。利用ログと業務成果を紐づけROIを示す。
- **解決する企業課題**：導入したが価値を説明できず展開が止まる。
- **向き／不向き**：経営承認を要する全社展開／PoCには簡易計測。
- **要素技術・連携**：Salesforce(売上)、Zendesk(サポートKPI)、Workday/Talentio(人事KPI)、BI。
- **落とし穴**：トークン数等の技術指標だけで成功を語ること。

# 面3：アイデンティティ・信頼（ID）★最難関

成否はこの面で決まる。中心問題は「**エージェントは誰の権限で動くのか**」。

## ID-1. Workforce/Customer 二面分離
- **概要**：従業員向けと顧客向けで、IdP・データ・エージェント・実行環境・監査を物理的・論理的に完全分離する。
- **設計**：従業員面＝Okta/Entra/Google Workspace、顧客面＝Auth0/Okta CIAM。面をまたぐデータ移動は明示ゲート（分類・承認・マスキング）でのみ。顧客向けは「顧客本人の情報＋公開情報のみ」「社内推論過程を露出しない」「高リスク時は人間移譲」。
- **解決する企業課題**：顧客向けが社内データに到達する（逆も）という最重大の漏洩クラスを構造的に排除。別顧客情報の混入防止。
- **向き／不向き**：顧客接点を持つ全企業／社内専用のみなら片面でよい。
- **要素技術・連携**：Tenant Isolation、CIAM(Auth0)、Output Guardrail、PII Filter、Human Handoff、Shopify(顧客面)。
- **落とし穴**：社内AIの一部機能を外に出して顧客向けにすること。別境界として設計する。

## ID-2. Identity Federation & On-Behalf-Of（OBO委譲）
- **概要**：エージェントは万能特権IDを持たず、依頼者本人の権限（または明示委譲）に縮退したトークンを下流ごとに取得して動く。
- **設計**：`User → IdP → Gateway（トークン交換 RFC 8693）→ Runtime → SaaS/内部API（OBO）`。委譲チェーン（user→agent→tool）をトークンに刻み、各SaaSの監査ログで本人に帰責。サービスアカウント利用時も実行主体（actor）と依頼者（subject）を分離記録。
- **解決する企業課題**：横断アクセス時の権限集約・混乱代理、SaaS監査での「誰がエージェント経由で触れたか」の追跡不能。
- **向き／不向き**：複数SaaS横断・監査要件が厳しい業務／完全公開情報のみなら不要。
- **要素技術・連携**：OIDC/SAML、SCIM、OAuth Token Exchange(RFC 8693)、Okta/Auth0、MCP。
- **落とし穴**：万能サービスアカウント1個で全SaaSを叩き、アプリ層だけで「見せない」と判定すること（判定バグ＝漏洩）。

## ID-3. Workload / Agent Identity（エージェント自身のID）
- **概要**：人の代理でない自律実行（定時バッチ・システム起点）のため、エージェント/ワークロードに検証可能な機械IDを与える。
- **設計**：SPIFFE/SVIDやクラウドのワークロードIDで短命の機械IDを付与。自律アクションは限定権限で動き、人の代理とは別系統で認可・監査。全呼び出しは「人ID（あれば）＋ワークロードID」の二重表現。
- **解決する企業課題**：操作主体が人かエージェントか曖昧、自律ジョブの過剰権限。
- **向き／不向き**：自律・バッチ系／対話のみなら軽量。
- **要素技術・連携**：SPIFFE/SPIRE、AWS IAM Roles Anywhere/IRSA、Entra Workload Identity、mTLS。
- **落とし穴**：自律エージェントに管理者権限を渡すこと。自律ほど最小権限を厳格に。

## ID-4. Permission Mirror & Least-of Faithful Access（権限忠実アクセス）
- **概要**：各SaaS/独自システムの権限を基盤側で再現・参照し、実効権限を常に「**エージェント能力 ∩ 本人権限 ∩ ポリシー**」の最小に縮退させる。
- **設計**：SaaSの users/groups/roles/ACL/共有設定を同期した Permission Mirror を持ち、RAG・ツール実行前にアクセス可否を判定。委譲(ID-2)が使える系は下流が本人権限で弾く。委譲不可の独自/旧式系は、本人エンタイトルメントを再現したフィルタを必ず通し「高リスク」に分類。
- **解決する企業課題**：「検索できたから答える」状態、RAGが本来見えない文書を返す、権限を超えた要約。
- **向き／不向き**：文書/チケット/CRM/チャットを横断検索する全社AI／権限が極単純な小環境には過剰。
- **要素技術・連携**：ACL/SCIM Group Sync、SaaS Admin API、Zanzibar系/ReBAC、ABAC、PDP。
- **落とし穴**：エンタイトルメントのコピーが源と乖離し剥奪済みアクセスが残る（遅延失効）。再同期＋短TTLで抑える。

## ID-5. JIT Scoped Credentials（最小・短命・用途限定）
- **概要**：長命の広いトークン/APIキーを廃し、ツール呼び出し直前に用途・対象・時間を限定した資格情報を都度発行する。
- **設計**：シークレットマネージャから動的に短命クレデンシャルを払い出し、スコープを最小化（「この顧客レコード読み取り、5分」）。コネクタは資格情報を保持せずブローカー経由で都度取得。
- **解決する企業課題**：鍵の散在・長期露出、乗っ取り時の被害時間窓。
- **向き／不向き**：機密・副作用ツール／低リスク参照のみなら軽量化可。
- **要素技術・連携**：HashiCorp Vault、AWS STS、各SaaSのスコープ付きトークン。
- **落とし穴**：「遅い」とキャッシュを広げ結局長命化すること。TTLは用途リスクで決める（→ステップ4）。

## ID-6. Zero-Trust Runtime + 中央PDP/分散PEP（ABAC/ReBAC）
- **概要**：社内起動でも信頼せず、すべての行為を毎回「誰が・どのエージェントで・どのテナント/プロジェクト・どのデータ・どの目的・どのリスク・今許されるか」で検証する。
- **設計**：認可判断を中央PDPに集約し、各実行点（Gateway・コネクタ・ランタイム）がPEPとして強制。ABAC/ReBABで主体×資源×コンテキスト×アクションを評価し、組織グラフを属性源に。判断は監査へ。NIST SP 800-207 準拠。
- **解決する企業課題**：「社内だから安全」の誤解、内部権限の横展開、インジェクション経由の内部API悪用。
- **向き／不向き**：機密・マルチクラウド・マルチSaaS／完全閉域実験には過剰。
- **要素技術・連携**：OPA/Cedar(PDP)、mTLS、Workload Identity、短命トークン、Network Policy、Runtime Sandbox。
- **落とし穴**：PDPを単一障害点/ボトルネックにすること。判断キャッシュとフェイルセーフ（不明なら拒否）を設計。

## ID-7. Policy-as-Code Guardrail（決定論的行動可否）
- **概要**：行動可否を自然言語プロンプトでなくPolicy-as-Codeで判定する。LLMは判断材料を整理し、Policy Engineが許可/拒否/承認要求/マスキングを決める。
- **設計**：`Agent提案 → Policy入力(actor/agent/action/resource/data_classification/risk_tier/purpose/project) → Engine → allow/deny/require_approval/redact`。Industry Policy Pack(GV-4)・憲法を展開先にする。
- **解決する企業課題**：プロンプトはセキュリティ境界にならない、規制・社内ルールの散在、承認基準の属人化、「なぜ許可したか」の説明不能。
- **向き／不向き**：規程・権限・ルールが複雑／単純文章生成のみなら不要。
- **要素技術・連携**：OPA/Rego、Cedar、PDP/PEP、Policy Versioning、Approval Workflow。
- **落とし穴**：高リスク領域でLLMに最終判断させること。判断は決定論ポリシーに委ねる。

## ID-8. Consent & Access Transparency（同意・透明化）
- **概要**：エージェントが本人代理でどのシステムに何のためアクセスしてよいかを、本人が把握・同意・取り消しできるようにする。
- **設計**：初回・高リスク時に委譲同意を取り、付与済みスコープを一覧・失効できるダッシュボードを提供。アクセスは本人にも可視。
- **解決する企業課題**：「いつの間にかエージェントが自分の権限で何でもしている」不信、同意原則のコンプラ。
- **向き／不向き**：従業員・顧客の信頼が重要な全用途／閉域バッチには軽量。
- **要素技術・連携**：IdPの同意画面、社内ポータル。
- **落とし穴**：同意を一度で恒久化しスコープがなし崩しに広がること。

# 面4：実行・オーケストレーション（RT）

## RT-1. Org-Hierarchical Hub & Spoke（意図ルーティング＋ドメインスポーク）
- **概要**：従業員は全社ポータル（Hub）とだけ対話し、Hubが意図を解釈して部署のドメイン特化エージェント（Spoke）に委譲・統合する。
- **設計**：Hubが意図分類 → HR Spoke(Workday/Talentio) / Eng Spoke(Jira/Linear/GitHub) / Sales Spoke(Salesforce/Sansan) 等へ委譲。委譲時に権限を減衰（親より強い権限を子が持てない）。Spokeは独自コンテキスト窓を持ち要約のみ返す。
- **解決する企業課題**：巨大プロンプトによるコンテキスト枯渇・誤解釈、部署SaaSのAPI変更が他部署に波及する問題（影響局所化）。
- **向き／不向き**：部署ごとに利用ツールが明確な大企業／単一業務の小規模には過剰。
- **要素技術・連携**：セマンティックルーター、LangGraph/AutoGen、部署別ベクトルDB、能力レジストリ。
- **落とし穴**：単一の万能エージェントに全ツール・全ポリシーを詰め込むこと（誤作動・権限集約・障害集中）。

## RT-2. RACI-based Multi-Agent Orchestration
- **概要**：マルチエージェントを、思考を深めるためでなく「**企業の責任分担（RACI）を反映する**」ために用いる。
- **設計**：R(実行)=ドメインエージェント、A(最終責任)=人間/オーナー、C(相談)=専門エージェント、I(通知)=ステークホルダ。例—契約レビュー：R=Legal Agent、C=Sales/Finance/Security Agent、A=Legal Manager、I=AE/CS。
- **解決する企業課題**：マルチエージェントの責任境界が曖昧、最終責任者不明、専門エージェント同士の議論発散。
- **向き／不向き**：複数部門関与・専門分化・最終責任が重い業務／単純Q&A・低遅延要件には不向き。
- **要素技術・連携**：Multi-agent Orchestrator、RACI Matrix、Handoff Protocol、Approval Workflow、Decision Log。
- **落とし穴**：「複雑だから」マルチ化すること。基準は「責任分担が複数に分かれるか」。

## RT-3. Risk-Tiered Autonomy（自律度の階層）
- **概要**：業務リスクに応じてエージェントの自律度を段階化する。
- **設計**：Tier0 回答/要約のみ → Tier1 下書き → Tier2 社内記録作成 → Tier3 外部通知/顧客影響 → Tier4 金銭/契約/人事/権限変更 → Tier5 禁止または二者承認必須。参照系は自動、更新系は承認付きに分離。
- **解決する企業課題**：全自動は危険・全承認は遅い、部門ごとのリスク感覚差、スケールと統制の両立。
- **向き／不向き**：高リスク業務を含む全社基盤／単純生成ツールには過剰。
- **要素技術・連携**：Risk Scoring、Policy Engine(ID-7)、Approval Workflow、Segregation of Duties、Audit。
- **落とし穴**：Tier境界を一律で決めること。データ機密度・不可逆性・職責で可変に（→ステップ4）。

## RT-4. Human Approval Chain（組織解決型承認）
- **概要**：高リスク・不可逆アクションの直前に人間承認チェーンを挟む。エージェントは「提案」まで、「実行」は承認後。
- **設計**：`提案 → リスク分類 → 承認者解決（組織グラフから上長/所管責任者/コスト所有者）→ 承認/却下/修正/エスカレーション → 実行 → 監査`。不在時は代理（delegation of authority）。SLAタイマと期限切れ処理、却下理由の学習まで設計。承認は既存ツール(Slack/ServiceNow/Workday)で。
- **解決する企業課題**：取り返しのつかない誤実行、承認者の属人化、監査証跡不足。
- **向き／不向き**：組織階層に基づく承認が要る業務／大量低リスク処理には摩擦。
- **要素技術・連携**：Approval Engine、Org Chart(Workday)、Delegation/Escalation、SLA Timer、Digital Signature。
- **落とし穴**：承認者をハードコードして異動で宛先不明になること。常に組織グラフから動的解決。

## RT-5. Intent-to-Enterprise Command Envelope（構造化コマンド封筒）
- **概要**：自然言語の依頼を、企業システムで実行可能な構造化コマンドに変換してから実行する。自然言語はUIであり内部プロトコルではない。
- **設計**：`自然言語 → intent → Command Envelope（actor/agent/target_system/resource/action/risk_tier/requires_approval/reason）→ ポリシーチェック(ID-7) → 承認(RT-4) → SaaS Adapter(IN-2)`。
- **解決する企業課題**：自然言語から直接API実行する危険、業務操作の監査不能、SaaS仕様依存、曖昧判断。
- **向き／不向き**：書き込み・複数SaaS連携／回答・要約だけなら不要。
- **要素技術・連携**：JSON Schema、Command Bus、Domain Command、Policy Engine、Audit。
- **落とし穴**：自然言語のままAPIに渡すこと。必ず構造化コマンドへ。

## RT-6. System-of-Record Write Boundary（書き込み境界）
- **概要**：基幹データを直接更新せず、決定論的ドメインサービス（SoRの正規手続き）を経由して更新する。
- **設計**：エージェントは「何をしたいか」を提案（Command Envelope）、ドメインサービスが検証・認可・業務ルール・トランザクションを経て SoR へ書き込む。下書き・提案は SoE(Slack/Notion/メール)に留め、人が確定。
- **解決する企業課題**：LLM誤判断による基幹破壊、不整合更新、業務ルール違反、トランザクション境界の混乱。
- **向き／不向き**：SoRを持つ整合性重視業務／下書き・一時メモには不要。
- **要素技術・連携**：DDD、Domain Service、Command Handler、Validation、Audit Trail。SoR=Workday/Salesforce/バクラク/Shopify。
- **落とし穴**：エージェントに SoR への直接write権限を与えて「速い」と喜ぶこと。

## RT-7. Enterprise Saga Agent（補償トランザクション）
- **概要**：複数SaaS/内部システムにまたがる副作用処理を、分散トランザクションでなくSagaとして管理し、途中失敗時に補償で巻き戻す。
- **設計**：各ステップ（Salesforce更新→Slack通知→Jira作成→契約ドラフト→承認→顧客メール）に補償（取り消し・訂正・差し戻し）を定義。長セッション中にDBトランザクションを開き続けない。冪等キーで二重実行防止。
- **解決する企業課題**：複数システム更新の途中失敗、長時間処理と整合性の両立、ロールバック不能。
- **向き／不向き**：複数システムに書き込む業務（オンボーディング/退職/契約/返品返金）／単一DB完結には不要。
- **要素技術・連携**：Saga Orchestration、Durable Workflow、Idempotency Key、Outbox、Compensation。
- **落とし穴**：1セッションをDBトランザクションにすること。ステップごとに小さく確定。

## RT-8. Durable Enterprise Agent Workflow（永続ワークフロー）
- **概要**：長時間処理を、同期APIでなく永続ワークフローとして実行し、承認待ち・障害・リトライ・キャンセルから再開する。
- **設計**：状態 `requested→planning→retrieving_context→waiting_approval→executing_tools→validating→completed/failed/cancelled`。LLM結果はアクティビティ境界外で固定し二度引かない（リプレイ安全性）。予算/時間/ステップ上限で暴走防止。
- **解決する企業課題**：数分〜数十分の処理、数時間〜数日の承認待ち保持、ワーカー障害耐性、冪等性。
- **向き／不向き**：10秒超・多段階・承認付き業務／数秒の単純回答には不要。
- **要素技術・連携**：Temporal/Step Functions/Durable Functions、LangGraph persistence、Queue、State Machine。
- **落とし穴**：長尺処理を同期HTTPに載せタイムアウト・コネクション占有を招くこと。

## RT-9. Enterprise Work Queue Agent（業務キュー参加）
- **概要**：エージェントを人間と同じ業務キュー（チケット・問い合わせ・申請・レビュー）に参加する「ワーカー」として設計する。
- **設計**：エージェント/人間/ハイブリッドワーカーがキューを処理。完了不能タスクは人間へ引き継ぎ、SLA・負荷分散・エスカレーションを管理。
- **解決する企業課題**：AI処理と人間業務の分断、引き継ぎ困難、負荷分散・SLA管理不能。
- **向き／不向き**：チケット駆動・SLAあり・人とAIの協働業務／完全対話型個人支援には不向き。
- **要素技術・連携**：Queue、Ticket System(ServiceNow/Zendesk/Jira)、SLA、Assignment Policy、Human Handoff。
- **落とし穴**：「チャットボット」として設計すること。「業務キューを処理するワーカー」と捉えると定着しやすい。

## RT-10. Event-Driven Enterprise Orchestrator（イベント駆動）
- **概要**：自然言語UIを持たず、SaaS/社内システムのイベント（Webhook）起点で裏側で自律稼働し、判断してAPIで別システムへアクションする。経営が求める抜本的自動化の本丸。
- **設計**：`Event Source（例:Workday入社完了/契約Boxアップロード）→ Event Bus(Kafka/EventBridge) → Agent判断 → 各API実行 → Slack通知`。RPA（決定論）では扱えない例外・揺らぎを含むプロセスを自動化。更新系は非同期Saga＋HitL（Slack/ServiceNowのApprove/Reject）を必須に。
- **解決する企業課題**：ユーザーが呼ばないと動かない受動性、システム間の人手コピペ業務、Webhook乱立。
- **向き／不向き**：業務イベントが明確／創造的作業のみには不向き。
- **要素技術・連携**：Event Bus、CloudEvents、CDC、Pub/Sub、Temporal/Step Functions、iPaaS(Workato)。
- **落とし穴**：イベントの嵐でコスト暴走・暴走実行。トリガー条件・頻度制限・デバウンス・リスク分類・予算上限・キャンセルを必ず設計。

## RT-11. Project Workspace / Digital Twin Agent（プロジェクト・デジタルツイン）
- **概要**：個人でなく「プロジェクト/チーム」に紐づく共有エージェントを構築し、文脈・成果物・意思決定・タスクを継続管理する“チームの一員”にする。
- **設計**：プロジェクト起動時に専用ワークスペース（共有メモリ=GraphRAG、Slackチャンネル、Asana/Jiraボード、Boxフォルダへの動的RBAC）をプロビジョニング。メンバーは `@Project-X-Agent` で対話。サブプロジェクトは親の非機密文脈を継承。毎朝Asana進捗とSlack議論を突合し「仕様不整合リスク」を自律警告するなど能動的に振る舞う。終了時はメモリをアーカイブ/失効。
- **解決する企業課題**：文脈がSlack/Notion/Jira/会議録/メールに分散、新メンバーのキャッチアップ遅延、意思決定理由の消失、情報の属人化、「言った言わない」。
- **向き／不向き**：マトリクス組織・アジャイル・長期大型・サブプロジェクトを持つ案件／個人作業・短期単発には不向き。
- **要素技術・連携**：GraphRAG(Neo4j)、Slack Bot、動的RBAC、Decision Log、RACI、Task API。
- **落とし穴**：終了後もメモリ・権限が残り異動者経由で漏れること。ライフサイクル失効を設計。

# 面5：知識・メモリ・コンテキスト（KM）

「横断データを組み合わせて文脈にしたいが安全上難しい」緊張をどう解くかが主題。

## KM-1. Access-Controlled Enterprise RAG（権限認識RAG）
- **概要**：検索/RAGの結果を、データ源のアクセス制御を保ったまま、依頼者が見てよいものだけに絞る。
- **設計**：`クエリ → ユーザー/エージェント/プロジェクト文脈 → Permission Filter(ID-4) → ハイブリッド検索(BM25＋ベクトル) → リランク → 引用付き回答`。取り込み時にソースACL・分類・鮮度を同梱し、検索時点の最新エンタイトルメントで評価（剥奪を即反映）。
- **解決する企業課題**：「インデックスにコピーした瞬間に源のアクセス制御が無効化される」企業RAG最大の落とし穴、古い文書参照、根拠不明回答。
- **向き／不向き**：文書/チケット/CRM/チャット横断検索／権限制御不能なデータ源・リアルタイムDB正本には不向き。
- **要素技術・連携**：Hybrid Search、Vector DB、Reranker、ACL Filter、Citation、Freshness Ranking。Box/Drive/Notion/Confluence。
- **落とし穴**：ACLを取り込み時固定し再同期しないこと（退職者・異動者が見続ける）。

## KM-2. Access-Controlled Context Mesh（フェデレーテッド文脈）
- **概要**：各データ源を一箇所に集約せず、権限制御を維持したまま分散問い合わせ（フェデレーション）で文脈を供給する。
- **設計**：`クエリ → Context Router → 各 Context Provider(Salesforce/Slack/Drive/Jira/DWH/独自) → ACL-aware retrieval → Context Package`。機密データは本人トークンでJIT取得（集約しない）。
- **解決する企業課題**：全社データレイクに集めると権限が壊れる、RAG索引に機密混入、最新権限を反映できない、コピー増で監査困難。
- **向き／不向き**：権限維持重視・データ所在地/規制が重要／権限不要の公開データ・極端な低レイテンシ要件には不向き。
- **要素技術・連携**：Federated Search、Context Router、Retrieval Proxy、Embedding Index per Scope、Just-in-time Retrieval。
- **落とし穴**：レイテンシを嫌い結局コピーに戻りACL同梱を怠ること。

## KM-3. Canonical Enterprise Object Model & Knowledge Graph（正規オブジェクト／知識グラフ）
- **概要**：多様なSaaSデータを、エージェントが業務文脈を理解するための共通業務オブジェクト（語彙）に正規化し、エンティティを名寄せして関係を張る。
- **設計**：Employee/Customer/Account/Opportunity/Contract/Project/Task/Ticket/Document/Invoice 等の正準オブジェクトを定義。完全統合でなく「意味的統合」（各システムへの参照を持つ）。エンティティ解決で同一顧客・同一人物をシステム横断で名寄せし、関係（所属・担当・参照・共有）とエンタイトルメント・エッジを張る。
- **解決する企業課題**：SaaSごとに同概念の名前が違う、顧客/案件/契約/請求の分断、横断文脈の統合不能、部門間の語彙差。
- **向き／不向き**：システムが多くデータ分散・経営/部門横断AI／単一SaaS完結業務には不要。
- **要素技術・連携**：Canonical Data Model、Enterprise Knowledge Graph、GraphRAG/Neo4j、MDM、Entity Resolution、Sansan(人物名寄せ)。
- **落とし穴**：全社データを単一グラフDBにコピーし巨大な漏洩資産を作ること。no-copy(KM-2)＋権限フィルタ(KM-1)前提で。

## KM-4. Scoped Memory Hierarchy（スコープ記憶階層）
- **概要**：メモリを個人/チーム/プロジェクト/部門/全社/顧客/エージェント内部に分離し、共有範囲を組織グラフに従わせる。
- **設計**：個人=Personal Enclave（本人のみ）、共有=Project Workspace（メンバー＋上位）等に物理的・論理的分離。サブプロジェクトは親の非機密のみ継承。承認者は種別ごと（PM/部門責任者/顧客情報管理者）。書き込みはゲート（分類・重複検知・承認）を通す。
- **解決する企業課題**：個人メモリのチーム漏れ、別プロジェクト/別顧客情報の混入、過去文脈の誤用。
- **向き／不向き**：継続利用・複数部署/プロジェクト・顧客情報を扱うAI／完全ステートレス単発には不要。
- **要素技術・連携**：Memory Store、Vector DB、ACL、Namespace、TTL、Consent、Memory Review UI。
- **落とし穴**：すべてを「全社共有メモリ」にし機密と雑多を混在させること。

## KM-5. Purpose-Bound Context Package（目的限定コンテキスト）
- **概要**：エージェントに渡す文脈を「使えるデータ」でなく「その業務目的に必要な最小データ」に限定してパッケージ化する。
- **設計**：目的（sales_followup / contract_review / support_response / onboarding / security_investigation 等）ごとに、許可データ型・許可システム・保持・マスキング・引用要件を定義。再現可能なpackとしてversion・トークン数・機密度を付与。
- **解決する企業課題**：何でも文脈に入れる、目的外利用、顧客/人事情報の過剰共有、プロンプト肥大化。
- **向き／不向き**：機密データを扱う・目的別権限管理が要る／公開情報のみには不要。
- **要素技術・連携**：Purpose Policy、Data Classification、DLP、Context Builder、Token Budget、Retention。
- **落とし穴**：関連性で絞らず詰め込み、精度低下（lost in the middle）とコスト増を招くこと。

## KM-6. DLP & Redaction Boundary（DLP・マスキング）
- **概要**：LLM・RAG・ログ・ツール実行の前後で機密情報を検出・マスキング・分類する。
- **設計**：`入力 → DLP/Secret Scan → Redaction/Tokenization → LLM/Tool → Output DLP → 応答/ログ`。分類はモデルGW(GV-5)と連動し、極秘は外部LLMに出さない経路へ。
- **解決する企業課題**：PII・秘密鍵・契約情報の外部送信、出力への秘密混入、ログ基盤への機密混入。
- **向き／不向き**：機密を扱う全AI／完全公開情報のみには不要。
- **要素技術・連携**：Microsoft Purview/Google DLP/Presidio、PII Detector、Secret Scanner、Tokenization、Output/Log Filter。
- **落とし穴**：入力前だけ検査し、RAG結果・ツール結果・出力・ログ保存前を素通しすること。

## KM-7. Ephemeral Secure Context Bus（揮発・機密計算）
- **概要**：情報漏洩防止とコンプラに極振りし、コンテキスト（プロンプト/記憶）を揮発・暗号化し、隔離領域で処理してセッション終了と同時に物理破棄する。
- **設計**：`要求 → 各SaaS収集 → DLP Proxyでマスキング → 隔離領域(機密計算)内でLLM処理(VPC内/学習オプトアウト済) → 応答 → コンテキスト破棄`。プロンプト/レスポンス本文はログ基盤に一切送らず、レイテンシ/トークン数等のメタのみ送信（トレースの程度の究極系）。
- **解決する企業課題**：SaaS横断結合で生じる予期せぬ機密（M&A/インサイダー情報等）の生成、外部LLMへの流出、ログ平文残存。
- **向き／不向き**：人事評価・極秘・規制データ（絶対にログ/キャッシュに平文を残せない処理）／低機密の大量処理には過剰。
- **要素技術・連携**：AWS Nitro Enclaves/Confidential Computing、Presidio(DLP)、Azure OpenAI(VPC統合)、揮発インメモリ(Redis No-Persistence)。
- **落とし穴**：性能のため隔離を緩めたり、デバッグ目的で本文をログに残すこと。極振り用途では一貫して破棄。

# 面6：統合・ツール（IN）

## IN-1. Enterprise Tool / MCP Gateway
- **概要**：エージェントがSaaS/内部API/MCP/DB/RPAを直接呼ばず、企業管理のGateway経由で呼ぶ。AI時代のEnterprise Service Busに近い。
- **設計**：認証・認可(ID-6)・スキーマ検証・レート制御・DLP・監査・冪等性・ドライランをGatewayで一元適用。ツールカタログ（スキーマ・権限・コスト）を管理し、有効化/無効化/版を運用制御。MCPサーバ群を信頼境界ごとに分離して束ねる。
- **解決する企業課題**：MCP/ツールの野良接続、APIキー漏洩、過剰権限、SaaSごとの監査差、インジェクションによるツール悪用。
- **向き／不向き**：ツール連携が多い・複数エージェントが共通ツールを使う／単一LLMチャットには不要。
- **要素技術・連携**：MCP Gateway、API Gateway、Tool Registry、JSON Schema、Secret Manager、DLP、Idempotency Key。
- **落とし穴**：「接続できること」を優先し「接続してよいこと」の統制を欠くこと。

## IN-2. SaaS Connector Adapter（腐敗防止）
- **概要**：各SaaSの独自API仕様・認証・データモデル・レート・エラー形式を、エージェントが扱う共通インターフェイス（業務語彙）に変換する。
- **設計**：`Agent Command → Canonical Tool Interface → SaaS Adapter（Salesforce/Workday/ServiceNow/Slack/Google/独自）`。SaaS固有差をアダプタに閉じ込め上流に漏らさない。スキル/プロンプトは業務語彙で書く。
- **解決する企業課題**：SaaS仕様差・変更がプロンプトやロジックに染み出す保守地獄、SaaS差し替え時の影響。
- **向き／不向き**：複数SaaS横断・将来差し替えの可能性／単一SaaS深依存には不要。
- **要素技術・連携**：Adapter Pattern、OpenAPI、GraphQL Federation、Connector SDK、Rate Limit Handler、Error Normalization。
- **落とし穴**：共通モデルを作り込みすぎ実態と乖離すること。薄く必要分だけ翻訳。

## IN-3. Rate / Quota Broker（レート/クォータ調停）
- **概要**：数万人が同じSaaSを叩くため、各SaaSのAPIレート上限を企業全体で公平配分・保護する。
- **設計**：システム別トークンバケットで優先度（対話＞バッチ）・テナント公平性・バックプレッシャを適用。上限接近時はキャッシュ/縮退/遅延通知へ。429を吸収しリトライを集中管理。
- **解決する企業課題**：一部署の大量バッチが全社のSaaS連携を枯渇させる輻輳。
- **向き／不向き**：大規模・共有SaaS／低トラフィックには軽量。
- **要素技術・連携**：Token Bucket、API Gatewayのレート機能、各SaaS API制限。
- **落とし穴**：個々のエージェントが各自リトライしSaaSを叩き潰すこと。リトライはブローカに集約。

## IN-4. Existing iPaaS Reuse（既存統合資産の再利用）
- **概要**：連携をゼロから作らず、既存の統合基盤（MuleSoft/Workato/Boomi/社内ESB）に乗る。
- **設計**：既存の連携・変換・認証資産をコネクタGWから呼ぶ。新規はMCP化、既存はiPaaS経由のハイブリッド。
- **解決する企業課題**：車輪の再発明による重複投資と保守分裂。
- **向き／不向き**：既存統合資産が豊富／新規グリーンフィールドには不要。
- **要素技術・連携**：MuleSoft/Workato、API管理(Apigee/Kong)。
- **落とし穴**：iPaaSの認可粒度が粗く権限忠実性(ID-4)が崩れること。検証してから乗る。

# 面7：観測・評価・監査（OB）

## OB-1. Enterprise Agent Observability Lake
- **概要**：実行ログ・トレース・コスト・ツール呼び出し・取得文脈・承認・評価結果を統合する観測基盤。
- **設計**：run_id/session_id/user_id/agent_id/model/prompt_version/tool_calls/retrieved_context/approval_status/token_usage/cost/latency/error/risk_tier を記録。保存は三層に分離（メタ→Trace DB、本文/文脈/成果物→暗号化オブジェクトストレージ、集計指標→DWH）。OpenTelemetry GenAI準拠。
- **解決する企業課題**：なぜその判断か不明、コスト発生源不明、障害調査・監査・品質改善不能。
- **向き／不向き**：本番AI全般（基本不向きなし。保存範囲と機密管理を要する）。
- **要素技術・連携**：OpenTelemetry、Trace Store、Object Storage、DWH、Prompt Store、Replay Tool、Datadog/CloudWatch。
- **落とし穴**：全プロンプトをログ基盤に直接入れること（巨大・高コスト・PIIリスク）。三層分離する（→ステップ4）。

## OB-2. Unified Audit & Lineage（三者帰責）
- **概要**：すべてのエージェント行為を「人（principal）＋エージェント（workload）＋対象システム」で改ざん不能に記録し、後から完全再構成・規制報告できる。
- **設計**：各アクションに principal・workload・tool/system・入出力ハッシュ・ポリシー判断・委譲チェーン・コストを記録。SoRの変更と突合可能に。相関IDでエージェント内監査と各SaaS監査を貫く。インシデント時はリプレイ(GV-9)。
- **解決する企業課題**：インシデント原因究明、規制対応、説明責任、エージェント内と各SaaS監査の分断。
- **向き／不向き**：本番AI全般に必須。
- **要素技術・連携**：SIEM(Splunk/Sentinel)、各SaaS監査ログ、相関ID、Event Store。
- **落とし穴**：エージェントとSaaSの監査が分断され横断追跡できないこと。相関IDで一本化。

# ステップ4：「程度」の選定基準

連続量パラメータは、過小・過大の両極が害になる。本番トレース(OB-1)と評価(GV-7)で継続調整する前提で、出発点と判断軸を示す。

## 4.1 自律度のティア境界（Risk-Tier の引き方）
- **過小（厳しすぎ）**：何でも要承認で承認者がボトルネック化し価値が消える。
- **過大（緩すぎ）**：不可逆な誤実行リスク。
- **基準**：「影響の不可逆性 × 影響額/範囲 × データ機密度 × 本人の職責」で Tier0–5 の境界を決める。参照系は自動、更新系（金銭/契約/人事/権限/外部公開）は承認寄り。導入初期は広めに要承認にし、evalで安全を確認しながら自動範囲を拡大。

## 4.2 タイムアウト・リトライ・予算（コスト上限）
- **基準**：エージェントは遅く高コストなので長め・多めに倒すが上限を設ける。TTFTと全体（無進捗時間）を別設定。リトライは冪等ステップのみ・指数バックオフ＋ジッタ・最大2〜3回、セッションのコスト/時間予算に従属。マルチエージェント(RT-2)は推論N倍を見込み予算上限を厳格に。

## 4.3 プロンプト/トレースのログ粒度（三層分離）
- **基準（程度の代表例）**：メタデータ（モデル/版/トークン/レイテンシ/コスト/相関ID/ツール/成否/risk_tier）→ **Trace DB**。本文・取得文脈・成果物 → **暗号化オブジェクトストレージ**（参照リンクで紐づけ、PIIはマスキング）。集計指標・品質スコア → **DWH**。極秘処理(KM-7)は本文を一切ログに残さずメタのみ。エラー時・低評価時・ランダムN%のみフル保存というサンプリングも併用。

## 4.4 コンテキスト投入量（top-k・トークン予算）
- **基準**：「使えるデータ」でなく「目的に必要な最小データ」(KM-5)。関連度ランキングで上位のみ、リランカーで件数を絞り、目的別トークン予算内に圧縮。詰め込みは精度低下（lost in the middle）とコスト増を招く。

## 4.5 メモリ保持・忘却（TTL・スコープ）
- **基準**：「重要度 × 鮮度 × 参照頻度」で残すものを選別、古い詳細は要約へ。スコープ(KM-4)ごとにTTLと失効を設定。プロジェクト終了・退職・異動でメモリと権限を失効。本人が消去できる権利を設計に含める。

## 4.6 ガードレール強度（誤検知 vs 見逃し）
- **基準**：リスクの高い経路（外部公開・機密・副作用・顧客面）ほど厳格に、低リスクは軽量に。FP率（過剰ブロック）とFN率（見逃し）を計測し業務の許容点で閾値を動かす。レイテンシ予算が厳しい検査は非同期/サンプリング化。

## 4.7 キャッシュ積極度・JIT資格情報TTL
- **基準**：完全一致を一次・セマンティックを二次。誤ヒットの害が大きい領域は類似度しきい値を高く・TTLを短く、パーソナライズ/時系列/機密は無効化。JITクレデンシャル(ID-5)のキャッシュTTLは用途リスクで決め、機密・副作用ほど短命に。

## 4.8 モデルの強さ・データ分類別ルーティング
- **基準**：難易度推定と確信度/検証でカスケード昇格。さらに**データ分類**で経路を分け、極秘＝VPC内/社内推論、一般＝外部API(GV-5)。昇格率・分類別比率を監視して調整。

## 4.9 カナリア段階・イベント駆動の頻度制限
- **基準**：1%→5%→25%→100%の多段で各段の品質/コスト/エラーが基準割れなら自動ロールバック(GV-6)。トラフィックが少なく有意性が出なければオフラインeval重視。イベント駆動(RT-10)はデバウンス・頻度上限・予算上限で嵐を抑える。

# ステップ5：「相反する仕組み」の選定基準

二者択一に見える対立の多くは「どこで線を引くか」が答え。エンタープライズで決定的な軸を示す。

## 5.1 OBO委譲 vs サービスアカウント（vs エージェントID vs ハイブリッド）
| 観点 | User OBO(ID-2) | Service Account | Agent Identity(ID-3) | Hybrid |
|---|---|---|---|---|
| 権限忠実性 | 高 | 低（判定バグ＝漏洩） | 中 | 高 |
| 対応範囲 | 委譲対応SaaSのみ | どのAPIでも | 自律ジョブ | 広い |
| 監査帰責 | 本人に明確 | 曖昧になりがち | エージェントに明確 | 明確 |
| 実装 | 複雑 | 容易 | 中 | 複雑 |

**基準**：個人業務支援＝User OBO。部門代表業務＝Agent Identity＋部門ポリシー。全社バッチ＝Service Account＋厳格監査＋高リスク分類。高リスク操作＝User OBO＋人間承認。最も実務的なのは「**実行主体はAgent、権限上限はUser**」のHybrid。

## 5.2 中央集権データレイク vs フェデレーテッド Context Mesh（＋JIT vs 事前索引）
| 観点 | 中央集権ベクトルDB/レイク | Federated Context Mesh(KM-2) |
|---|---|---|
| 向き | 分析・BI・統計 | 権限付きAI文脈取得 |
| メリット | 高速・集計容易 | 権限を維持しやすい |
| デメリット | 権限のミリ秒同期が事実上不可能→漏洩 | レイテンシ・実装複雑 |

**基準**：権限不要の公開社内規程は中央ベクトルDBへ。機密SaaSデータは本人トークンでJIT取得するフェデレーテッドへ、という**ハイブリッドが必須**。個別ユーザー権限で答えるAIにはMeshを優先。経営分析・統計のみレイク。事前索引する場合もACL同梱(KM-1)を必須に。**「速いから機密も索引化」は禁忌**。

## 5.3 単一エージェント vs RACIマルチエージェント
**基準**：まず単一。マルチ化の基準は「複雑だから」でなく「**企業内の責任分担が複数に分かれるから**」(RT-2)。複数部門関与・専門分化・最終責任が重い業務でマルチ、単純Q&A・低遅延・低コストは単一。マルチはコスト・遅延・複雑性・障害点が増える。

## 5.4 Read-only vs Write-capable（段階的拡張）
**基準**：`Read-only → Draft-only → 承認付きWrite → 低リスク自動Write → 高リスク統制Write` の順で段階拡張。参照系はAutopilot、更新系はSoR経由(RT-6)＋HitL(RT-4)のCopilotに分離。

## 5.5 Copilot（業務支援）vs Autopilot（業務代行）
**基準**：経営はAutopilotのROIを求めるが、確率的挙動による基幹（Workday/SAP/Salesforce）破壊は許容できない。**参照系API＝Autopilot、更新系API＝HitLのCopilot**を明確に分離。Autopilot化はeval・カナリア・監査が揃った領域から段階的に。

## 5.6 個人の記憶 vs プロジェクト/チームの記憶
**基準**：個人効率化は個人メモリ、サイロ化防止は共有メモリが要る。**Personal Enclave（個人領域）と Project Workspace（共有領域）を物理的・論理的に分離**(KM-4/RT-11)し、共有範囲を組織グラフに従わせる。一本化は漏洩・混線の温床。

## 5.7 全プロンプトログ vs 選択的トレースログ
**基準**：再現性のため本文は残したいが、ログ基盤に平文で全件は機密・コストで破綻。**メタはTrace DB、本文は暗号化ストレージ、集計はDWH**の三層分離(4.3/OB-1)。極秘処理(KM-7)はメタのみ。

## 5.8 中央集権プラットフォーム vs 部署フェデレーション
**基準**：認可・監査・モデル統制・コスト・憲法・評価は**中央集権**（GV/ID面）。ドメイン知識・ユースケース・エージェント中身は**各部署にフェデレート**（GV-3のテンプレで権限委譲）。「中央が全部作る」も「各部署が野放し」も失敗。**中央がガードレールと舗装路を、部署が業務ロジックを**持つ二層統治。

## 5.9 コネクタ自前構築 vs 既存iPaaS再利用
**基準**：既存統合資産がある領域は再利用(IN-4)、ただし認可粒度が権限忠実(ID-4)を満たすか検証。新規・独自はMCP化(IN-1)。

## 5.10 内部/オンプレモデル vs 外部API
**基準**：極秘・規制データ・大量定常→VPC内/社内推論。一般機密・最新性能・変動需要→外部API（DPA・リージョン・保持を統制）。モデルGW(GV-5)がデータ分類で自動ルーティング。

## 5.11 同期 vs 非同期
**基準**：数秒で終わる対話は同期＋ストリーミング。10秒超・多段階・承認待ちは非同期＋永続ワークフロー(RT-8)。部分結果を即ストリーム、重い処理は非同期化のハイブリッドが実用的。

## 5.12 プロンプトで守る vs ポリシー/実行基盤で守る
**基準**：答えは明確。**プロンプトはセキュリティ境界にならない**。安全保証は権限・承認・検証・隔離・Policy-as-Code(ID-7)で実行基盤側に置く。プロンプトは品質・ふるまいの調整に使う。

# ステップ6：アーキテクチャパターン全体の統合・選定・組み合わせ

## 6.1 依存関係（積み上げ構造）

層として重ねて使う。代表的依存：

- **OB-1 Observability / OB-2 Audit** は GV-7 評価・GV-9 事故対応・GV-6 版管理の素材（記録なくして評価・再現・調査なし）。
- **ID-2 OBO / ID-4 Permission Mirror** は KM-1 権限認識RAG・KM-2 Context Mesh の前提（権限の伝播なくして安全な横断文脈なし）。
- **ID-6 PDP/PEP・ID-7 Policy-as-Code** は GV-4 Industry Pack・RT-3 Risk-Tier・RT-4 承認の判断基盤。
- **GV-1 Control Plane** は GV-2 カタログ・GV-8 コスト・OB-2 監査の統制点（実行許可のゲート）。

- **横断軸：ゼロトラスト/監査** が全アクションを三者帰責で貫く。

## 6.2 組み合わせレシピ（基盤→入口→実行→自動化）

## 6.6 設計原則（まとめ）

1. **エージェントは本人の権限を超えない**——実効権限は「能力 ∩ 本人権限 ∩ ポリシー」の最小。便利さのための全権化を禁ずる。
2. **従業員面と顧客面を物理的に分ける**——最重大の事故クラスを構造的に排除。
3. **作らず、寄り添う**——SoRを置き換えず、読み、正規手続きで書く。既存統合資産を再利用する。
4. **データはコピーする前に疑う**——no-copy・JIT・ACL同梱を既定にし、集約は目的が明確なときだけ。
5. **組織グラフを唯一の権威に**——スコープ・共有・承認・委譲を組織構造から一貫して導く。
6. **プロンプトでなく、アイデンティティとポリシーで守る**——安全保証は実行基盤側に置く。
7. **すべての行為を三者で帰責する**——人＋エージェント＋システムを相関IDで貫き企業横断で監査。
8. **中央はガードレールと舗装路を、部署は業務ロジックを**——集権と分権の二層統治。
9. **自然言語はUIであり内部プロトコルではない**——作用は必ず構造化コマンドへ変換する。
10. **エージェントは「業務キューを処理する管理されたデジタル業務主体」**——チャットボットでなく、登録・監査・権限制御・評価され続ける実行主体として設計する。
11. **AIを賢くするより、企業の境界内で安全に働けるようにする**——知能は前提、勝負は権限・統合・統治。
12. **やるか/やらないかでなく、どの程度かを設計する**——自律度・ログ・予算・キャッシュ等の連続量を、トレースとevalで継続調整する。

> 最も凝縮すると——**AIエージェントを企業に導入するとは、LLMを業務システムにつなぐことではなく、企業のID・権限・責任・データ・プロセス・監査・組織構造の中に、新しい実行主体を安全に参加させることである。** 確率的な知能を、決定論的な権限・組織・監査の檻の中に閉じ込めたとき、初めて数万人規模の本番に耐えるエンタープライズAIエージェントが成立する。
