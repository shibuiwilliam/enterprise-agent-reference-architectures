# AIエージェント本番組み込み アーキテクチャ統合リファレンス

> 3本のレポート（汎用パターン版・詳細カタログ版・本統合作業）を精読・評価し、重複を排除・再分類して1本に統合したリファレンス。パターンカタログは詳細版の体系（カテゴリA〜L）を骨格に採用し、不足パターンを補追。そのうえで、各レポートで手薄だった「**程度の選定基準**」「**相反する仕組みの選定基準**」を独立章として厚く再構成した。

# ステップ1：レポート全体のアジェンダ

## 1.1 本書の到達目標

AIエージェントは「賢いが信用しきれない・遅く・高コストで・確率的で・騙されうる実行主体」である。本書の目的は、この異質なコンポーネントを**決定論的な殻・契約・権限・検証・予算・観測・統治の中に安全に閉じ込め**、その周囲を従来のソフトウェア工学（状態管理・トランザクション・監査・テスト・デプロイ・可観測性）で固める設計知を、再利用可能なパターン集として提供することにある。

## 1.2 統合にあたっての評価と方針

3レポートを比較評価した結論は以下のとおり。

- **パターン網羅性**：詳細カタログ版（A〜L、12カテゴリ）が最も粒度が細かく実務的であったため、これを**骨格として全面採用**する。
- **独自パターンの補追**：汎用版が持つ「ブラックボード型協調」「投機/ヘッジ実行」を、それぞれカテゴリB・Hに統合する。「意図ルーティング」はB-3・H-1に内包させる。
- **手薄だった2領域の強化**：いずれのレポートも「程度（how much）の決め方」と「相反する選択肢（どちらを採るか）の判断基準」が断片的だった。本統合では**ステップ4・5として独立章化**し、各パターンに紐づけて具体的なしきい値・判断軸を与える。
- **重複排除**：同一概念の別名（例：耐久セッション/チェックポイント、補償トランザクション/Saga、ACL/腐敗防止層）は1パターンに正規化した。

## 1.3 全体アジェンダ（本書の歩き方）

1. **ステップ1（本章）**：到達目標・統合方針・アジェンダ。
2. **ステップ2**：各パターンを記述する共通スキーマと、カテゴリ分類の設計。
3. **ステップ3**：アーキテクチャパターン本体（カテゴリA〜L、計約50パターン）。
4. **ステップ4**：「程度」の選定基準（タイムアウト・リトライ・予算・ログ粒度・コンテキスト量・温度・ガードレール強度など、連続量の決め方）。
5. **ステップ5**：「相反する仕組み」の選定基準（同期/非同期、単一/複数エージェント、抽象化/固有最適化など、二者択一の判断軸）。
6. **ステップ6**：全体統合（依存関係・成熟度ロードマップ・選定ガイド・リファレンスアーキテクチャ・設計原則・組み合わせ方）。

## 1.4 設計を駆動するAIエージェントの性質（force）

すべてのパターンは、以下のエージェント特性が生む「設計圧力」への応答である。これらは独立でなく**連鎖**して制約を強める。

| エージェント特性 | アーキテクチャ上の含意 | 主な対応カテゴリ |
|---|---|---|
| 1リクエストが長い（数秒〜数十分） | 同期RPCでなくJob/Session/Workflowとして扱う | A |
| 1リクエストが高コストかつ可変 | 予算制御・キャッシュ・モデル選択・早期停止 | H |
| 非決定論的（同入力でも出力が変わる） | 再現性・評価・ガードレール・監査ログ | C/F/I |
| ハルシネーション | 根拠確認・検証器・RAG・外部API優先 | F |
| 自然言語インターフェイス | 意図解析・曖昧性解消・契約化・構造化出力 | C |
| メモリをセッション/エージェント間で受け渡す | メモリを一級のデータ管理対象にする | E |
| 可用性が不安定（外部LLM依存） | フォールバック・ルーティング・縮退・再実行 | H |
| ツール/MCPで副作用を起こす | 権限・サンドボックス・冪等性・補償・監査 | D/G |
| LLM差し替えで挙動が変わる | モデル抽象化・評価CI・カナリア・版固定 | I/J |
| プロンプトインジェクション | LLMを「騙されうる代理人」とみなし被害半径を限定 | G |
| 過剰な自律性（excessive agency） | 権限・機能・自律判断を最小化する | D/F/G |
| ループ・暴走の可能性 | ステップ/予算/時間の上限とデッドマンスイッチ | A/H |
| 説明責任（なぜそうしたか） | トレース保存・リプレイ・版記録 | I |

> 整合する公知の枠組み：MCP（Model Context Protocol）、OWASP Top 10 for LLM Applications（Excessive Agency / Prompt Injection / Insecure Plugin Design 等）、NIST AI RMF（生成AIプロファイル）、LangGraphのdurable execution/checkpoint、OpenAIのStructured Outputs・prompt caching・evals、AWS Bedrock AgentCore。

# ステップ2：各アーキテクチャパターンの項目設計

## 2.1 共通記述スキーマ

すべてのパターンを、以下の7項目で統一記述する。横並び比較と選定を可能にするためである。

- **概要**：そのパターンが何であるかの一文要約。
- **設計**：構造・データフロー・状態遷移・実装上の要点。
- **解決する課題**：どのエージェント特性（force）に応えるか。
- **ユースケース**：典型的な適用場面。
- **向き**：採用が効くシステム条件。
- **不向き**：採用がオーバーヘッドや害になる条件。
- **要素技術**：代表的な実装技術・プロダクト・標準。

## 2.2 カテゴリ分類の設計

パターンは「どの設計圧力に応えるか」で12カテゴリに分類する。これは責務境界とも一致し、リファレンスアーキテクチャの層構造に対応する。

| カテゴリ | テーマ | 主眼 |
|---|---|---|
| **A** | 実行・セッション・オーケストレーション | 長く・止まり・再開し・暴走しうる実行を骨格化 |
| **B** | エージェント構成・分担 | 何を決定論に残し、どこをどう分業するか |
| **C** | 入出力・契約化 | 曖昧な自然言語を安全な契約へ変換 |
| **D** | ツール・MCP・外部接続 | 副作用の最重要リスク境界の統制 |
| **E** | メモリ・コンテキスト管理 | 記憶を保存・忘却・権限付きのデータ機能に |
| **F** | 信頼性・検証・ガードレール | 確率的出力を品質保証された出力へ |
| **G** | セキュリティ・マルチテナント | 騙されても被害が小さい設計 |
| **H** | コスト・性能・可用性 | 高コスト・可変・不安定の資源管理 |
| **I** | 観測性・評価・変更管理（LLMOps） | 測定・再現・安全な継続変更 |
| **J** | デプロイ・ベンダー抽象化 | モデル/SDKロックインの構造的隔離 |
| **K** | 人間協調・UI/UX | 制御された自律性と協働点の設計 |
| **L** | 導入・移行・統治 | 後付け統合と段階的な信頼獲得 |

> 読み方：A〜Bは「骨格」、C〜Gは「安全と品質の境界層」、H〜Jは「資源と運用」、K〜Lは「人とプロセス」。後段のステップ6で、これらを積み上げる依存関係と標準構成を示す。

# ステップ3：アーキテクチャパターン本体

# カテゴリA：実行・セッション・オーケストレーション

エージェントの「長く・止まり・再開し・暴走しうる」実行特性を、システムの骨格として扱う。

## A-1. Request-to-Job Gateway（非同期ジョブ受付）
- **概要**：1リクエストを同期APIでなく非同期ジョブとして受け付け、実処理をバックグラウンドのワーカー/ワークフローへ委ねる。
- **設計**：API Gatewayが受理し即座に `agent_session_id`・`job_id` を発行（`202 Accepted`）。実処理はキュー/ワークフローエンジンへ。クライアントはポーリング・WebSocket・SSE・push通知で進捗と結果を取得。状態は `pending → running → (partial) → done/failed/timeout`。
- **解決する課題**：HTTPタイムアウト、コネクション占有、LB切断、バースト吸収、Web層とエージェント層の独立スケール。
- **ユースケース**：リサーチ、レポート/資料生成、コード生成、データ分析、長尺の問い合わせ対応、RPA的処理。
- **向き**：完了まで複数ステップや外部検索を伴うタスク全般。
- **不向き**：数百msの即応が要るオートコンプリート・同期トランザクション。
- **要素技術**：SQS / Pub/Sub / Kafka / Redis Streams、Celery / Sidekiq、Temporal / Step Functions / Cloud Tasks、WebSocket / SSE、ジョブ状態ストア（Postgres / Redis）。

## A-2. Durable Agent Session（耐久セッション／チェックポイント）
- **概要**：実行状態をステップ単位でチェックポイント永続化し、クラッシュ・中断・承認待ちから再開できるようにする。
- **設計**：各ステップの入出力・ツール結果・モデル応答・判断・エラーを `agent_session_state` に保存。再開は直前チェックポイントから。LLM呼び出しは非決定論的なのでアクティビティ境界の外で結果を固定し二度引かない（リプレイ安全性）。
- **解決する課題**：プロセス再起動・LLMタイムアウト・ツール失敗への耐性、数時間〜数日の人間承認待ち保持、冪等性。
- **ユースケース**：承認付き業務処理、複数ツール連携、数十分の自律タスク、長期ワークフロー。
- **向き**：副作用を伴う長尺ワークフロー、SLAの厳しい基幹連携。
- **不向き**：完全ステートレスな単発チャット、低価値な使い捨て処理。
- **要素技術**：Temporal / AWS Step Functions / Azure Durable Functions / Restate / Inngest、LangGraph checkpoint、event sourcing、PostgreSQL / DynamoDB / Redis。

## A-3. Streaming Progress（進捗ストリーミング）
- **概要**：最終結果だけでなく、計画・ツール実行状況・検証・承認待ちなどの進捗を逐次提示する。
- **設計**：「計画中／取得中／検証中／生成中／承認待ち」等の状態を出す。生のchain-of-thoughtでなく、監査可能なステップ名・要約・根拠リンクを表示する。
- **解決する課題**：長時間処理中の不安・ブラックボックス性・重複リクエスト・離脱。
- **ユースケース**：調査AI、生成AIアプリ、コードエージェント、B2B SaaSのエージェント機能。
- **向き**：UXが重要な対話型・作業型エージェント。
- **不向き**：内部バッチ、途中状態を見せる必要のない処理。
- **要素技術**：SSE / WebSocket、event stream、trace summarizer、progress state machine。

## A-4. Interruptible Agent（中断・方針変更）
- **概要**：実行中にユーザー/システムが処理を止め、方針を修正して再開できる。
- **設計**：各ステップ後にチェックポイントを作り、「止める／方針変更／このツールは使うな／ここから再開」を受け付ける。A-2の上に構築。
- **解決する課題**：開始時の指示が不完全、途中で意図が変わった、暴走しかけた、というケース。
- **ユースケース**：資料作成、コード生成、分析、設計レビュー、営業提案。
- **向き**：長時間・探索的なタスク。
- **不向き**：原子的に完了すべき決済・予約確定・契約締結。
- **要素技術**：checkpoint、pause/resume、WebSocket、event log、human feedback API、Temporal signal。

## A-5. Time-Budgeted Agent Loop（予算制御ループ／デッドマンスイッチ）
- **概要**：思考・探索・ツール実行に時間・回数・トークン・金額の予算を課し、暴走と予算超過を止める。
- **設計**：`max_steps` / `max_tool_calls` / `max_cost_usd` / `deadline_at` / `max_retry` をセッションに持たせ、超過時は「部分結果」「追加承認要求」「簡易モデルへ切替」「人間へエスカレーション」へ分岐。
- **解決する課題**：無限ループ、過剰探索、高額API利用、外部ツール連打、月末の請求事故。
- **ユースケース**：Deep Research、複数API連携、コード修正、マルチテナントSaaS。
- **向き**：高コストLLM・自律性の高いエージェント。
- **不向き**：途中打ち切りが許されない単発ミッションクリティカル処理（事前見積もり重視）。
- **要素技術**：token accounting、cost meter、budget manager、deadline propagation、circuit breaker、トークンバケット。

## A-6. Agent Saga（補償トランザクション）
- **概要**：副作用付きツールの連鎖を分散トランザクションでなくSagaとして管理し、途中失敗時に補償処理で巻き戻す。
- **設計**：各ツール実行を `step` とし、成功時の次処理と失敗時の補償を定義（例：返金→在庫更新→通知の途中失敗で返金取消・管理者通知）。長セッション中にRDBトランザクションを開き続けない。
- **解決する課題**：非決定論的主体が外部世界に副作用を起こす「中途半端な状態」、長時間処理と整合性の両立。
- **ユースケース**：注文変更、返金、予約、請求、契約、在庫、顧客通知。
- **向き**：副作用が金銭・物理に直結する業務。
- **不向き**：強いACIDが必須で補償・承認では許容できない処理、不可逆操作。
- **要素技術**：Saga orchestration、outbox pattern、idempotency key、compensation handler、workflow engine。

# カテゴリB：エージェント構成・分担

「何を決定論的に保ち、どこをエージェントに委ね、どう分業するか」を決める。

## B-1. Deterministic Backbone, Probabilistic Edge（決定論的バックボーン×確率論的エッジ）
- **概要**：ワークフローの骨格と中核業務ロジックは決定論的コード/ワークフローエンジンが管理し、曖昧処理・生成・探索だけをエージェントノードに委ねる。
- **設計**：「受付/検証/承認/実行/通知」や料金計算・権限判定・在庫引当・状態遷移は通常コードで実装。エージェントは「問い合わせ解釈」「文章生成」「候補選定」「例外処理」に限定。エージェント出力は決定論的な殻（入力正規化＋出力スキーマ検証）で包み、システムはLLM生出力に直接触れない。
- **解決する課題**：全委任で失う再現性・監査性・法的説明責任・安全性、フォーマット不安定さとハルシネーションの下流流出。
- **ユースケース**：金融・医療・法務・人事・EC・在庫・契約などの基幹業務、B2B SaaS。
- **向き**：制御性・監査性が重要なほぼすべての本番組み込み。
- **不向き**：フロー自体が毎回変わる探索的研究・自由創作（殻を緩める）。
- **要素技術**：Temporal / Step Functions / BPMN、domain service、rule/policy engine、JSON Schema、Pydantic / Zod / Instructor、構造化出力。

## B-2. Planner–Executor–Reviewer（計画・実行・検証の分離）
- **概要**：計画する頭脳、実行する手足、検証する目を別ロールに分け、必要に応じ修正ループを回す。
- **設計**：Plannerが構造化計画（DAG/ステップ列）を出力 → Executorが安価モデル/決定論コードで実行 → Reviewerが成果物・根拠・安全性を検証。想定外はPlannerへ差し戻し（re-plan）。計画を機械可読アーティファクト（JSON/DSL）として固定するのが鍵。
- **解決する課題**：単一エージェントは自分の誤りに気づきにくい。高コストモデル呼び出し回数削減、テスト容易化、説明性向上。
- **ユースケース**：コード生成、文書生成、分析、意思決定支援、設計レビュー。
- **向き**：複数ステップ・複数ツールを要する複雑タスク。
- **不向き**：低レイテンシ必須・コスト制約が強い処理、計画が無意味なほど短いタスク。
- **要素技術**：multi-agent orchestration、Plan-and-Execute / ReWOO、role prompt、structured critique、LLM-as-a-judge、JSON DSL。

## B-3. Supervisor & Specialist（階層委譲・意図ルーティング）
- **概要**：Supervisorが全体を統括し意図を分類、専門サブエージェントへ委譲（handoff）して結果を統合する。入力種別による経路振り分け（Intent Router）もここに内包する。
- **設計**：Supervisorがintent・制約・進捗を管理し、Legal/SQL/Coding/Search/Support等の専門エージェントへ分配。第1段は軽量分類器（embedding/小型モデル/ルール）で意図判定し、確信度が低ければ安全な既定経路へフォールバック。サブエージェントは独自コンテキスト窓を持ち要約のみ親へ返す（窓の節約）。失敗サブタスクは再割当・再計画。
- **解決する課題**：単一プロンプトに全知識・全ツール・全ポリシーを詰め込む複雑化と誤作動、コンテキスト窓・能力限界、全件最上位モデル処理の過剰コスト。
- **ユースケース**：複数業務領域・複数ツール・複数権限の企業向けエージェント、リサーチ、大規模コード改修、多機能アシスタント。
- **向き**：広範囲・多専門にまたがる複雑タスク、入力種別が多様なシステム。
- **不向き**：単純なFAQ・分類・要約（オーケストレーション過剰）、極端な低レイテンシ/低コスト要件、誤分類コストが致命的な領域。
- **要素技術**：agent router、handoff、task decomposition、specialized prompts、capability registry、Semantic Router、LangGraph / CrewAI / AutoGen / OpenAI Agents SDK。

## B-4. Agent Ensemble & Debate（合議・討論）
- **概要**：複数モデル/プロンプト/戦略に同じ問題を解かせ、投票・採点・討論で品質と頑健性を高める。
- **設計**：N個の独立サンプルを生成し、(a) 一致度の高い答えを採用（self-consistency）、(b) 検証器でbest-of-N選抜、(c) 提案者と反対者・審判による討論。コストはN倍なので難問のみNを増やす適応制御が現実的。
- **解決する課題**：確率性・モデル依存・単発回答の不安定さ、単一視点バイアスと見落とし。
- **ユースケース**：高価値な意思決定、難しい推論、コードレビュー、リスク判定、安全性レッドチーミング。
- **向き**：正誤が判定可能で誤りのコストが高い判断。
- **不向き**：コスト・レイテンシが厳しい処理、正解が一意に定まらない創作。
- **要素技術**：self-consistency、multi-sample decoding、LLM-as-a-judge、ranker、multi-agent debate、model ensemble。

## B-5. Blackboard Coordination（ブラックボード型協調）
- **概要**：複数エージェントが中央の共有作業領域（ブラックボード）を介して非同期に協調する。各エージェントは盤面を読み、自分の専門に基づき書き込む。直接通信を排し疎結合化する。
- **設計**：中央コントローラが盤面状態から「次に起動する専門家」を決める。エージェント間通信はすべて盤面経由とし、楽観ロック/バージョニングで競合制御。漸進的に解を組み立てる探索タスクに向く。
- **解決する課題**：エージェント間の通信トポロジ複雑化、どの専門家がいつ必要かが動的に変わるタスク。
- **ユースケース**：複雑な診断・設計問題、漸進的に解を構築する探索的タスク。
- **向き**：解の構築が漸進的で、必要な専門家が動的に決まる場合。
- **不向き**：単純な逐次フロー（盤面のオーバーヘッドが無駄）、競合制御が複雑化しすぎる規模。
- **要素技術**：共有ステート（Redis/DB）、イベント駆動（盤面更新イベント）、楽観ロック、controller。

# カテゴリC：入出力・契約化

自然言語という曖昧で攻撃面の広い入出力を、システムが安全に扱える契約へ変換する。

## C-1. Natural Language Boundary Adapter（自然言語境界アダプタ）
- **概要**：自然言語をそのまま業務ロジックに渡さず、境界で構造化（intent / slots / constraints）してから渡す。
- **設計**：入力を `intent`, `entities`, `constraints`, `risk_level`, `missing_information` に変換。業務ロジックは構造化データのみ受け取る。
- **解決する課題**：自然言語の曖昧さ（命令・希望・条件・雑談・攻撃の混在）。
- **ユースケース**：チャットUI、社内AI、問い合わせ受付、業務依頼受付。
- **向き**：自然言語入口を持つ業務システム全般。
- **不向き**：自由対話そのものが目的のアプリ。
- **要素技術**：intent classifier、slot filling、JSON Schema、Pydantic / Zod、function calling。

## C-2. Structured Output Contract & Self-Correction（構造化出力契約＋自己修正）
- **概要**：LLM出力をスキーマで契約化し、検証失敗時はエラーをフィードバックして自己修正させる。
- **設計**：出力は必ず `schema_version` 付きJSON。パーサ（Pydantic等）がスキーマ＋ビジネスルール（例「割引率20%以下」「実在商品IDか」）を検証。Passは下流へ、Failはエラー内容を返して再生成。上限到達時は安全フォールバック。
- **解決する課題**：フォーマット崩れ、存在しないenum、不完全JSON、ビジネスルール逸脱、下流パースエラー。
- **ユースケース**：構造化データ（JSON/SQL）生成、基幹データ入力、ワークフロー遷移、UI生成。
- **向き**：正確性・型整合が求められる連携処理。
- **不向き**：創作・雑談（リトライ遅延がUXを損なう）。
- **要素技術**：JSON Schema、Pydantic / Instructor / Zod、OpenAI Structured Outputs、Outlines（制約付きデコーディング）、schema registry。

## C-3. Inverted Structured Output（意思決定オブジェクト）
- **概要**：LLMに最終アクションを実行させず、「通常コードが使う中間判断」を出させる。
- **設計**：LLMは `recommended_action`, `confidence`, `evidence`, `missing_inputs`, `requires_human_approval` を返す。最終実行は決定論的サービスがポリシー検証のうえ行う。
- **解決する課題**：LLMが直接アクション実行することによる誤判断の被害拡大（excessive agency）。
- **ユースケース**：審査、承認、分類、ルーティング、リスク判定。
- **向き**：判断は任せたいが実行責任はシステムに残したい処理。
- **不向き**：自由生成自体が成果物の場合。
- **要素技術**：decision object、policy validation、confidence threshold、approval workflow。

## C-4. Ambiguity Negotiation（曖昧性交渉）
- **概要**：曖昧な依頼を即実行せず、不足情報と前提を明示して確認する。
- **設計**：`missing_information` と `assumptions` を抽出し、安全・コスト・業務影響が大きい場合は実行前に確認（clarification loop）。
- **解決する課題**：意図と異なる実行、誤発注・誤送信。
- **ユースケース**：予約、発注、メール送信、データ更新、契約処理。
- **向き**：誤実行の影響が大きい処理。
- **不向き**：多少間違っても問題ない探索的タスク（確認が摩擦になる）。
- **要素技術**：clarification loop、risk scoring、confirmation UI、assumption logging。

# カテゴリD：ツール・MCP・外部システム接続

エージェントが外部世界へ作用する最重要リスク境界。ここを甘くするとシステム全体が脆くなる。

## D-1. Tool Gateway（ツールゲートウェイ）
- **概要**：エージェントが直接社内API/DBを叩かず、すべてのツール呼び出しをゲートウェイに集約する。
- **設計**：認証・認可・rate limit・引数スキーマ検証・監査ログ・idempotency・dry-runをGatewayで一元適用。ツールカタログ（スキーマ・権限・コスト）を管理し追加/廃止/版を運用から制御。MCPサーバ群もここに束ねる。
- **解決する課題**：直接の強権限付与による誤実行・過剰実行・情報漏洩、ツール乱立のガバナンス崩壊、認証情報の散在。
- **ユースケース**：企業システム連携、複数MCP/外部API、複数エージェントでのツール共有。
- **向き**：多数のツール・副作用ツールを持つ本番エージェント。
- **不向き**：読み取り専用の軽量ツール1〜2個の小規模PoC。
- **要素技術**：API Gateway、MCP server/client、OAuth / サービスアカウント、OPA、rate limiter、audit log、idempotency key。

## D-2. Least-Privilege Tool Binding（最小権限ツールバインディング）
- **概要**：エージェント/ユーザー/セッションごとに利用可能ツールと権限を最小化し、スコープ・期限付きで動的付与する。
- **設計**：Tool Registryに `read_only`/`write`/`destructive`/`external_network`/`pii_access` 等の属性を付与。セッション開始時にポリシーで利用可能ツールだけをbinding。委譲時は権限を減衰（attenuate、親より強い権限を子が持てない）。
- **解決する課題**：OWASPの excessive agency / insecure plugin design / sensitive information disclosure、乗っ取り時の被害範囲（blast radius）。
- **ユースケース**：社内AI、SaaS管理画面、顧客データを扱うエージェント、マルチエージェント。
- **向き**：副作用ツールを多数持つ環境、機密データを扱う環境。
- **不向き**：完全に閉じたsandbox内の実験。
- **要素技術**：RBAC / ABAC、OPA/Rego / Cedar、ケイパビリティトークン、短命クレデンシャル、MCP permissions、scope token。

## D-3. Dry-Run First Execution（ドライラン先行実行）
- **概要**：副作用のあるツールは、まず影響をシミュレートしてから本実行する二段階方式。
- **設計**：`refund_order` 等に対し最初に `simulate_*` を実行。差分・影響範囲・リスクをユーザー/ポリシーエンジンに提示し承認後に本実行（two-phase execution）。
- **解決する課題**：AIの誤判断による不可逆操作。
- **ユースケース**：削除、送金、返金、発注、契約、メール一斉送信。
- **向き**：不可逆・高影響の書き込み操作。
- **不向き**：読み取り専用・低リスクな生成処理。
- **要素技術**：dry-run API、diff engine、approval workflow、policy check。

## D-4. Sandboxed Tool Runtime（サンドボックス実行）
- **概要**：コード実行・ブラウザ操作・ファイル操作などを隔離環境で実行し、ネットワーク・認証情報・ファイルを制限する。
- **設計**：生成コードやブラウザ操作はcontainer / micro-VM / browser sandbox内で実行。ephemeralクレデンシャル、network policy、seccompで制限。
- **解決する課題**：プロンプトインジェクションやツール誤用による任意コード実行・情報漏洩・破壊的操作。
- **ユースケース**：コードインタープリタ、ブラウザエージェント、データ分析、ファイル変換。
- **向き**：副作用や任意コードを伴うツール。
- **不向き**：低レイテンシ・低コスト最優先の単純処理、読み取り専用検索のみの構成。
- **要素技術**：Docker / Firecracker / gVisor / WASM、Kubernetes Job、seccomp、network policy、ephemeral credentials。

## D-5. MCP Adapter Isolation（MCP境界分離）
- **概要**：MCPサーバを信頼境界ごとに分離し、エージェントから直接フルアクセスさせない。
- **設計**：社内データ・外部SaaS・ローカルファイル・ブラウザ・コード実行を別MCPサーバ/namespace/network segmentに分け、Tool GatewayがMCP clientとして接続しポリシー適用後に公開。
- **解決する課題**：MCP導入で増える攻撃面・権限管理・監査の複雑化。
- **ユースケース**：企業内MCP基盤、複数チーム共用のエージェント環境。
- **向き**：多数のMCP接続を持つ組織基盤。
- **不向き**：個人ローカル用途の軽量MCP利用。
- **要素技術**：MCP server/client、network isolation、tool registry、secret manager、audit trail。

# カテゴリE：メモリ・コンテキスト管理

メモリを「便利機能」でなく、保存・忘却・権限・鮮度・分離を伴うデータ管理機能として設計する。

## E-1. Layered Memory（階層メモリ）
- **概要**：記憶を短期（推論用）・セッション（進行ジョブ用）・長期（嗜好/事実）・組織共有（ナレッジ/ルール）に分離する。
- **設計**：短期はコンテキスト窓内、窓が溢れる前に要約して長期へ固定化（consolidation）。検索時は関連メモリだけを呼び戻し注入。複数エージェントが同一メモリを参照しコンテキストを共有。
- **解決する課題**：コンテキスト窓の有限性、セッションを跨いだ記憶喪失、長期パーソナライズ、エージェント間サイロ化。
- **ユースケース**：パーソナルアシスタント、長期プロジェクト伴走AI、マルチエージェント。
- **向き**：継続的に同じユーザー/プロジェクトと関わるエージェント。
- **不向き**：一問一答で完結する使い捨てツール。
- **要素技術**：Redis / PostgreSQL、Vector DB（Qdrant / Milvus / pgvector）、GraphRAG（Neo4j）、Mem0 / Zep / Letta、embedding、TTL、namespace、access control。

## E-2. Context Pack（コンテキストパック）
- **概要**：エージェントに渡すコンテキストを、その場で組み立てる再現可能な「パッケージ」として扱う。
- **設計**：`system instruction`, `user profile`, `task state`, `retrieved docs`, `tool list`, `policy`, `budget`, `output schema` をpackとして構成し、version・作成根拠・トークン数・機密度を付与。関連度でランキングしトークン予算内に圧縮・整形。
- **解決する課題**：プロンプトの肥大化・属人化、「何を渡したか」の再現不能、コンテキスト窓の非効率。
- **ユースケース**：本番エージェント全般、特にマルチテナントSaaS。
- **向き**：渡す情報が動的で多様なシステム。
- **不向き**：毎回ほぼ同一の単純LLM呼び出し。
- **要素技術**：prompt template、context builder、RAG、token counter、prompt registry、context hash。

## E-3. Memory Write Gate（書き込みゲート）
- **概要**：エージェントが勝手に長期メモリへ保存しないよう、保存前に検査・承認を通す。
- **設計**：長期書き込みは `candidate_memory` として出力し、ポリシー・重複検知・機密判定・（必要なら）ユーザー承認を通してから保存。
- **解決する課題**：誤った/古い記憶の固定化、プライバシー侵害、意図しない保存。
- **ユースケース**：個人化AI、社内ナレッジAI、顧客対応AI。
- **向き**：長期メモリを書き込むエージェント。
- **不向き**：一時セッションのみのAI。
- **要素技術**：memory classifier、PII detector、deduplication、approval UI、retention policy。

## E-4. Forgetting & Expiration（忘却・失効）
- **概要**：メモリにTTL・失効条件・鮮度を持たせ、古い情報を使い続けないようにする。
- **設計**：`expires_at`, `confidence`, `source`, `last_verified_at`, `sensitivity` を持つmemory itemとして管理。古い情報は検索順位を下げるか再確認を要求。
- **解決する課題**：AIが古い情報をもっともらしく使い続ける問題。
- **ユースケース**：人事・組織情報、価格、契約条件、顧客状態、プロジェクト状態。
- **向き**：時間で変化する事実を扱うメモリ。
- **不向き**：不変のドメイン知識。
- **要素技術**：TTL index、temporal retrieval、source freshness scoring、memory compaction。

# カテゴリF：信頼性・検証・ガードレール

確率的出力を「品質保証された出力」へ引き上げる層。

## F-1. Evidence-First Answer（根拠優先／RAGグラウンディング）
- **概要**：回答より先に根拠取得を強制し、回答に出典・confidenceを紐づける。
- **設計**：まず検索/DB/API/ドキュメントからevidenceを取得。最終回答にevidence id・source・confidenceを付ける。引用必須化・ハイブリッド検索・リランクで検索品質を独立改善。
- **解決する課題**：ハルシネーション、根拠不明な断定、古い記憶の利用。
- **ユースケース**：社内検索、法務、技術調査、顧客サポート、経営レポート。
- **向き**：事実性・鮮度が重要な用途。
- **不向き**：外部知識が不要な純推論・変換、創作・ブレスト。
- **要素技術**：RAG、hybrid search（BM25＋ベクトル）、reranker、knowledge graph、citation store、context compression。

## F-2. Guardrail Sidecar（入出力ガードレール）
- **概要**：入出力の安全検査（PII漏洩、毒性、ジェイルブレイク、トピック逸脱、機密語）を本体から独立した層として配置する。
- **設計**：入力ガード（インジェクション検知・PIIマスキング・ポリシー違反検知）と出力ガード（機密流出・ハルシネーション疑い・フォーマット違反）の二段。サイドカー/プロキシ化で複数エージェント横断再利用・一元管理。
- **解決する課題**：自然言語入力の広い攻撃面、機密漏洩の下流流出、コンプラ要件の集中管理。
- **ユースケース**：規制業種、外部ユーザー向け対話システム。
- **向き**：外部に開かれたエージェント全般。
- **不向き**：レイテンシ予算が極端に厳しい用途（軽量ルールに絞る）。
- **要素技術**：NeMo Guardrails / Guardrails AI / Llama Guard / Prompt Shields、分類器、サービスメッシュのサイドカー。

## F-3. Verifier Agent（検証エージェント／LLM-as-Judge／自己修正）
- **概要**：生成とは別に検証専用エージェント/サービスを置き、採用前にチェックする。生成は難しく検証は容易、という非対称性を活用。
- **設計**：最終出力をfactuality・policy・format・security・business rule・toxicity・PIIの観点で検査。結果により修正・ブロック・人間確認へ分岐。検証器は生成器と独立に、できれば決定論的に作る（自己採点は甘い）。反復は必ず上限つき。
- **解決する課題**：LLMの自己過信、業務ルール違反や危険出力の出荷前検出、ハルシネーション・要件未達。
- **ユースケース**：外部ユーザー向けAI、事実性が重要なRAG、コード生成（テスト実行で検証）。
- **向き**：誤りのコストが高い用途、出力の正しさを機械検証できる用途。
- **不向き**：検証コストが生成コストを大きく超える用途、検証も主観頼みで収束しない用途。
- **要素技術**：LLM-as-a-judge、ユニットテスト/型チェック/コンパイラ、rule engine、moderation、PII detection、Reflexion / Self-Refine。

## F-4. Policy-as-Code Guardrail（ポリシー・アズ・コード）
- **概要**：行動制約をプロンプトでなくコードとして管理し、LLMの判断とは独立に実行前後で判定する。
- **設計**：「誰が・どのエージェントで・どのデータに・どのツールを・どの条件で」をコード化し、policy decision point / enforcement pointで強制。
- **解決する課題**：プロンプトはセキュリティ境界にならないという根本問題、NIST AI RMFが求めるリスク管理の体系化。
- **ユースケース**：企業AI基盤、マルチテナントSaaS、規制業界。
- **向き**：明確な統制が必要な組織。
- **不向き**：個人PoC。
- **要素技術**：OPA/Rego、Cedar、AWS IAM、ABAC、PDP/PEP。

## F-5. Human Approval Checkpoint（人間承認ゲート／HITL）
- **概要**：高リスク・不可逆アクションの直前に人間承認を挟む。エージェントは「提案」まで、「実行」は承認後。
- **設計**：フローをDAG化し、クリティカルノード（決済・本番デプロイ・契約送信）で実行を一時停止しコンテキストをシリアライズ。人間が計画・影響範囲・根拠・リスク・代替案をレビューし「承認/却下/修正」。状態を復元し再開。リスク分類器で「自動可/要承認/禁止」に振り分け。
- **解決する課題**：取り返しのつかない誤実行、ブラックボックス化、信頼の段階的獲得、規制対応。
- **ユースケース**：金融取引、インフラ管理、法的文書送信、人事評価、重要データ更新。
- **向き**：失敗コストの高い領域、導入初期。
- **不向き**：人間介在で価値が消える高速・大量処理、承認者がボトルネックになる用途。
- **要素技術**：耐久ワークフロー（A-2）、approval queue、Slack/Teams/email approval、根拠提示UI、リスクスコアリング、監査ログ。

# カテゴリG：セキュリティ・マルチテナント

LLMを「騙されうる代理人」とみなし、騙されても被害が小さい設計を目指す。

## G-1. Confused-Deputy Damage Limitation（混乱代理被害限定／Dual-LLM）
- **概要**：プロンプトインジェクションの完全防止でなく被害半径の限定を目標とし、信頼できない入力を扱うLLMと特権操作を行うLLMを分離する。
- **設計**：隔離LLM（Quarantined）が信頼できないデータ（Web・メール・文書）を処理するがツール権限を持たず構造化結果だけ返す。特権LLM（Privileged）はツールを実行できるが信頼できない生データを直接見ない（不透明ハンドル経由）。ツール側でユーザー権限・データ所有者・操作目的・承認状態を検証し、LLMの判断だけで高権限操作を許可しない。
- **解決する課題**：直接/間接プロンプトインジェクション、悪意あるWeb/メール/文書による操作誘導、権限奪取・情報漏洩。
- **ユースケース**：外部コンテンツを読むエージェント（メール・ブラウザ・社内文書アシスタント）。
- **向き**：信頼できない入力を扱い、かつ副作用ツールを持つエージェント。
- **不向き**：外部入力を一切扱わない閉域処理。
- **要素技術**：Dual-LLM、CaMeL系の制御/データフロー分離、capability token、user-bound / tool-side authorization、content trust labeling、sandbox。

## G-2. Data Boundary Firewall（データ境界ファイアウォール／DLP）
- **概要**：LLMに渡す前後でデータを検査・マスキング・分類し、過剰送信と機密流出を防ぐ。
- **設計**：入力時にPII・機密・契約・認証情報を検出し、渡してよい情報だけをredaction/tokenizationして渡す。出力時もegress filterで機密混入を検査。
- **解決する課題**：LLMへの過剰データ送信、出力への秘密混入、越境データ問題。
- **ユースケース**：個人情報・顧客情報・営業秘密・ソースコードを扱うAI。
- **向き**：機密データを扱うすべてのエージェント。
- **不向き**：公開情報のみを扱うAI。
- **要素技術**：DLP、PII detector、redaction、tokenization、data classification、egress filter。

## G-3. Tenant-Isolated Runtime（テナント分離ランタイム）
- **概要**：テナントごとに実行環境・メモリ・ツール権限・トレースを分離する。
- **設計**：`tenant_id` をsession・memory・tool call・trace・vector indexすべてに付与。高セキュリティ顧客はdedicated runtime / dedicated storageへ分離。
- **解決する課題**：他テナントのデータ混入、メモリ漏洩、ツール誤実行。
- **ユースケース**：B2B SaaS、エンタープライズAI、BYOC。
- **向き**：複数テナントを抱えるサービス。
- **不向き**：単一ユーザーアプリ。
- **要素技術**：tenant namespace、row-level security、dedicated VPC、KMS、vector index per tenant。

# カテゴリH：コスト・性能・可用性

高コスト・可変・不安定という資源特性を、設計対象として明示的に扱う。

## H-1. Cost-Aware Model Router（コスト考慮モデルルーター／カスケード）
- **概要**：タスクの難易度・リスク・予算に応じてモデルを動的選択し、失敗時のみ上位モデルへエスカレーション（カスケード）。
- **設計**：軽量ゲートウェイが入力を分類（難易度推定・キャッシュヒット判定）し、単純検索は軽量モデル/従来API、複雑推論は高性能モデルへ。低信頼/検証不合格時は上位へescalate。大半を安価層で捌き難問のみ高コスト層へ。
- **解決する課題**：全リクエスト最上位モデル処理の過剰コストと過剰遅延、ベンダーロックイン、新モデル移行時のA/B。
- **ユースケース**：カスタマーサポート、社内FAQ、難易度のばらつきが大きいシステム、大量の分類/抽出/要約。
- **向き**：リクエスト難易度の分布が広いサービス。
- **不向き**：常に最高水準の推論が要る専門特化システム、難易度判定自体が困難な領域。
- **要素技術**：RouteLLM等のルーター、難易度分類器、LiteLLM / OpenRouter、confidence score、fallback chain。

## H-2. Semantic Result Cache（セマンティック結果キャッシュ）
- **概要**：完全一致でなく意味的に近いリクエストの結果を再利用してLLM/検索/ツール処理を省く。
- **設計**：入力を埋め込み化しベクトルストアで類似検索、閾値超なら過去回答を返す（必要なら軽量再検証）。完全一致キャッシュを一次、セマンティックを二次に置く多層構成。TTL・無効化でキャッシュ汚染を防ぐ。
- **解決する課題**：高コスト・高レイテンシ、レート制限、よくある質問への過剰処理。
- **ユースケース**：FAQ、社内問い合わせ、技術サポート、レポート生成。
- **向き**：質問分布に偏りがある用途。
- **不向き**：最新性・個別性・厳密性が重要な処理、再利用率が低い一意なリクエスト。
- **要素技術**：埋め込みモデル、Vector DB（pgvector / Redis / Qdrant / Pinecone）、GPTCache、TTL・無効化、cache validation。

## H-3. Prompt-Cache Optimized Context（プロンプトキャッシュ最適化）
- **概要**：共通プロンプトやツール定義をキャッシュしやすい構造にして、入力トークンコストとレイテンシを下げる。
- **設計**：system prompt・tool definitions・policy・共通知識を安定したprefixとして固定し、ユーザー固有情報は後方に置く。プロバイダのprompt cachingを効かせる。
- **解決する課題**：長いsystem prompt・ツール定義・RAGコンテキストによる高コスト・高レイテンシ。
- **ユースケース**：共通プロンプトが大きいエージェントプラットフォーム、MCPツールが多い環境。
- **向き**：反復的に同一prefixを使う構成。
- **不向き**：毎回まったく異なる短いプロンプト。
- **要素技術**：prompt prefix design、provider prompt caching、context hashing、tool lazy loading。

## H-4. Graceful Degradation & Fallback（縮退・フォールバック／サーキットブレーカー）
- **概要**：LLMプロバイダ障害・高負荷・予算超過時に段階的に機能を落としてサービスを継続する。サーキットブレーカーで切替を自動化。
- **設計**：呼び出しをサーキットブレーカーで包みエラー率/レイテンシ閾値で回路を開く。高性能モデル不可→小型、生成不可→テンプレート、ツール不可→人間エスカレーション、RAG不可→検索リンク提示。半開状態で復旧確認。機能ごとにレート枠/プールを分離（bulkhead）。
- **解決する課題**：単一プロバイダ障害の即停止リスク、レート制限、コスト上限への巻き込まれ、輻輳の連鎖。
- **ユースケース**：顧客向けSaaS、24/7社内AI、ミッションクリティカル補助機能。
- **向き**：可用性SLAを約束するサービス。
- **不向き**：品質低下が一切許されない処理（フォールバックよりエラー返却が誠実）。
- **要素技術**：複数プロバイダ抽象（LiteLLM / OpenRouter）、Resilience4j / Polly、fallback model、rate limit、queue backpressure、指数バックオフ＋ジッタ。

## H-5. Speculative / Hedged Execution（投機・ヘッジ実行）
- **概要**：複数の処理経路を投機的に並列起動し、最初に成功した（または最良の）結果を採用する。レイテンシをコストで買う。
- **設計**：例—複数プロバイダへ同時投入し最速応答を採用（hedged request）、複数解法を並列に試行。コストがN倍になるため、レイテンシがクリティカルな箇所に限定。遅延した経路はタイムアウト連動でキャンセル。
- **解決する課題**：テールレイテンシ（遅いプロバイダ応答）、可用性。
- **ユースケース**：レイテンシ厳守のリアルタイムUI、可用性最優先の重要パス。
- **向き**：レイテンシ価値がコスト増を上回るクリティカルパス。
- **不向き**：コスト最優先の大量バッチ（無駄なN倍課金）。
- **要素技術**：hedged request、並列実行、最速勝者採用、タイムアウト連動キャンセル、deadline propagation。

# カテゴリI：観測性・評価・変更管理（LLMOps）

非決定論的システムを「測定し、再現し、安全に変更し続ける」ための運用基盤。

## I-1. Agent Trace & Observability（トレース・可観測性）
- **概要**：各ステップ・LLM呼び出し・ツール呼び出し・判断・エラーをspanツリーとして記録する。
- **設計**：`agent_session_id`, `trace_id`, `span_id`, `model`, `prompt_version`, `tool_name`, `latency`, `tokens`, `cost`, `input_hash`, `output_hash` を記録。エラー率・コスト・レイテンシ・品質をダッシュボード化しアラート。全プロンプト本文はログ基盤でなくオブジェクトストレージへ退避し参照リンクで紐づける（程度はステップ4参照）。トレースはeval・リプレイの素材になる。
- **解決する課題**：ブラックボックス化、「なぜその判断か」「どこで失敗・課金されたか」が不明、インシデント原因究明。
- **ユースケース**：本番エージェント全般。
- **向き**：すべての本番運用。
- **不向き**：（基本的になし。機密はhash化・redactionで対応）。
- **要素技術**：OpenTelemetry（GenAI semantic conventions）、LangSmith / Langfuse / Arize Phoenix、Datadog / CloudWatch、trace store、object storage、redaction。

## I-2. Evaluation CI/CD（評価CI/CD）
- **概要**：品質をテストでなく継続的評価（eval）として、CIと本番の双方で測り続ける。
- **設計**：ユースケース別ゴールデンデータセットにオフラインevalをCIで実行（完全一致でなくルーブリック採点・LLM-as-Judge・特性アサーション）。本番はオンラインeval（サンプリング採点・ユーザーフィードバック・やり直し率/エスカレーション率）。モデル/プロンプト/ツール/RAG変更時の回帰ゲートにする。
- **解決する課題**：非決定論ゆえ通常の単体テストで品質劣化・挙動変化を検出できない。
- **ユースケース**：継続改善する本番AI全般。
- **向き**：品質責任を負うチーム。
- **不向き**：評価基準が定義困難な探索的PoC初期（まず手動評価）。
- **要素技術**：golden dataset、LLM-as-a-judge、scenario/regression eval、promptfoo / DeepEval / Braintrust / LangSmith、CI/CD。

## I-3. Production Replay（本番リプレイ／差分テスト）
- **概要**：本番ログを匿名化・固定化し、モデル/プロンプト変更時に新旧で再実行して差分比較する。
- **設計**：実リクエスト・取得コンテキスト・ツール結果を匿名化保存し、新旧エージェントでreplay。品質・コスト・レイテンシ・安全性で差分評価。event sourcing（A-2）と組み合わせると任意時点へ巻き戻せる。
- **解決する課題**：開発データでは再現できない本番の曖昧さ・ノイズ・攻撃・例外。
- **ユースケース**：大規模SaaS、顧客サポート、社内AI基盤。
- **向き**：トラフィックが多くログを活用できる運用。
- **不向き**：ログ保存が規制上難しい場合（匿名化前提）。
- **要素技術**：log redaction、replay harness、shadow evaluation、differential testing、event store。

## I-4. Version Pinning & Change Management（版固定・変更管理・カナリア）
- **概要**：プロンプト・モデル・ツール・スキーマ・ポリシー・RAG indexをバージョン管理し、変更をPR・eval・カナリア・ロールバックの対象にする。「デプロイ＝挙動変更」をWebデプロイ並みの規律で扱う。
- **設計**：各実行に `prompt_version` / `model_version` / `tool_schema_version` / `policy_version` / `retriever_version` を記録。新版はシャドー（複製トラフィックで事前観測）→カナリア（1%→5%→…、基準割れで自動ロールバック）で投入。
- **解決する課題**：小さなプロンプト変更でも挙動が大きく変わる、プロバイダ側のサイレントなモデル更新。
- **ユースケース**：監査が必要な業務、継続改善するSaaS機能。
- **向き**：成熟した運用、十分なトラフィック。
- **不向き**：トラフィックが少なくカナリアの統計的有意性が出ない小規模（オフラインeval重視）。
- **要素技術**：prompt/model/schema registry、feature flag、semantic versioning、GitOps、A/B基盤、自動ロールバック。

# カテゴリJ：デプロイ・ベンダー抽象化

「LLMが変わると挙動が変わる」「特定SDK/ベンダーに縛られる」リスクを構造的に隔離する。

## J-1. Agent Runtime Abstraction（実行基盤の抽象化）
- **概要**：エージェント実行基盤を抽象化し、SDK/ランタイムを差し替え可能にする。
- **設計**：`AgentRuntime` interfaceに `start_session` / `resume_session` / `invoke_tool` / `checkpoint` / `stream_events` / `cancel` を定義し、実装をprovider adapterとして切替（OpenAI Agents SDK、LangGraph、ADK、Bedrock AgentCore、自社基盤）。
- **解決する課題**：特定ベンダー/SDKへのロックイン、移行時の大規模改修。
- **ユースケース**：エンタープライズAI基盤、BYOC、マルチクラウドSaaS。
- **向き**：長期運用・複数基盤併用。
- **不向き**：単一プロバイダ前提の小規模アプリ。
- **要素技術**：adapter / ports-and-adapters、provider registry、capability matrix。

## J-2. Model Behavior Compatibility Layer（モデル挙動互換レイヤー）
- **概要**：モデルごとの癖（tool calling形式、structured output、context length、安全拒否、streaming形式、token会計）を吸収する互換レイヤー。
- **設計**：モデル別差異をadapterで吸収し、上位アプリは統一インターフェイスを使う。
- **解決する課題**：同じプロンプトでもモデルが変わると挙動が変わる問題。
- **ユースケース**：OpenAI / Anthropic / Gemini / OSS LLMを併用する環境。
- **向き**：複数モデルを使い分ける/乗り換える運用。
- **不向き**：単一モデル固定で長期運用する場合。
- **要素技術**：model adapter、prompt transpiler、schema normalizer、capability detection。

## J-3. Agent Capability Registry（能力レジストリ）
- **概要**：各エージェント/ツール/モデルが何をできるかをレジストリで管理し、選定の根拠にする。
- **設計**：`capability` / `required_permissions` / `cost_profile` / `latency_profile` / `risk_level` / `supported_models` / `owner` / `version` を登録。Supervisor（B-3）が参照して委譲先を選ぶ。
- **解決する課題**：エージェント/ツールが増えると「誰に何を任せるか」が不明になる。
- **ユースケース**：企業内エージェント基盤、複数部署利用、MCPマーケットプレイス。
- **向き**：多数のエージェント/ツールを抱える組織。
- **不向き**：単一エージェントの小規模アプリ。
- **要素技術**：service catalog、tool registry、metadata DB、capability ontology、discovery API。

# カテゴリK：人間協調・UI/UX

エージェントは「制御された自律性」で運用する。人間との協調点を設計する。

## K-1. Agent Workbench（エージェント・ワークベンチ）
- **概要**：計画・進捗・成果・根拠・承認を一画面で管理する作業台型UI。
- **設計**：チャットに加え、タスクボード・進捗タイムライン・根拠パネル・成果物プレビュー・承認ボタン・差分表示を持つ。
- **解決する課題**：チャットUIだけでは長時間・多段階・副作用付き作業を管理しにくい。
- **ユースケース**：資料作成、コード修正、調査、業務自動化、社内AI。
- **向き**：作業型・長尺タスク。
- **不向き**：単純FAQ。
- **要素技術**：task UI、timeline、artifact preview、approval panel、trace viewer。

## K-2. Editable Plan（編集可能な計画）
- **概要**：実行前に計画を提示し、人間が編集してから実行する。
- **設計**：計画を `steps[]` として構造化し、ユーザーが順序変更・削除・追加・制約付与できる。編集後のplanを実行（B-2のPlannerと連携）。
- **解決する課題**：計画が意図とずれたまま長時間実行される問題。
- **ユースケース**：設計、調査、コード生成、営業資料、プロジェクト計画。
- **向き**：計画品質が成果を左右するタスク。
- **不向き**：即時応答が必要な処理。
- **要素技術**：plan schema、UI editor、workflow compiler、human feedback。

## K-3. Agent-to-Human Escalation（人間へのエスカレーション）
- **概要**：自信不足・権限不足・リスク過大のとき、人間へ自然に引き継ぐ。
- **設計**：confidence・risk・policy違反・missing data・tool failureをもとにescalation。人間には要約・試行済み手順・根拠・推奨アクションを渡す。
- **解決する課題**：AIが無理に完了しようとして誤回答・誤実行する。
- **ユースケース**：カスタマーサポート、社内ヘルプデスク、運用監視、法務・経理。
- **向き**：人間オペレーターと協働する運用。
- **不向き**：人間が存在しない完全自動サービス。
- **要素技術**：handoff queue、case management、Slack/Teams integration、confidence threshold。

# カテゴリL：導入・移行・統治

既存システムへの後付けと、信頼を段階的に獲得する移行戦略。

## L-1. Shadow Mode & Progressive Autonomy（シャドー運用→段階的自律）
- **概要**：本番処理は人間/既存システムが行い、裏でエージェントに同じタスクを解かせて比較。十分な品質の領域から段階的に自動化を広げる（ストラングラー的移行）。
- **設計**：エージェント出力はユーザーに出さず既存結果と差分評価。フロントのルーターが対象ケースを「従来処理/エージェント処理」に振り分け、品質ゲートを満たすたびに担当範囲を拡大、最終的に従来処理を退役。常にロールバック可能を保つ。
- **解決する課題**：ビッグバン置換のリスク、実データでの安全な性能検証、組織的な信頼形成。
- **ユースケース**：業務自動化の導入初期、PoCから本番への移行。
- **向き**：リスク許容度が低い組織の漸進的移行。
- **不向き**：裏側実行やログ取得が許されない処理、置換対象のない新規機能。
- **要素技術**：shadow traffic、feature flag、offline eval、difference analysis、human benchmark、品質ゲート。

## L-2. Anti-Corruption Layer（アンチコラプション層）
- **概要**：確率的・自然言語的なエージェントの世界と、決定論的・厳格なレガシー基幹システムの世界の間に翻訳層を置き、互いの概念汚染を防ぐ。
- **設計**：エージェントの曖昧な意図/自然言語出力を、レガシーの厳格なドメインモデル/APIへ翻訳（逆も）。レガシーのエラー/制約をエージェントが理解できる形に変換。エージェント側の変化（モデル差し替え等）をレガシーへ波及させない。
- **解決する課題**：レガシー連携時の概念不一致、エージェントの不安定さの伝播、双方の独立進化。
- **ユースケース**：既存の堅牢な基幹システムへエージェントを後付けする統合。
- **向き**：レガシー資産が大きい組織。
- **不向き**：両側を同時設計できる新規グリーンフィールド。
- **要素技術**：DDDのACL、アダプタ/変換器、ドメインモデルマッピング、明示的I/O契約。

## L-3. Agent Constitution（エージェント憲法）
- **概要**：行動原則・禁止事項・優先順位・組織ルールを「憲法」として一元管理し、プロンプト・ポリシー・eval・レビューに展開する。
- **設計**：「顧客データを外部送信しない」「根拠なしに断定しない」「削除前に承認を取る」等を体系化し、プロンプト埋め込みだけでなくpolicy-as-code（F-4）・eval suite（I-2）・tool guardrail（D-2）へ落とす。
- **解決する課題**：組織ルールがプロンプト断片に散らばり運用品質が属人化する。
- **ユースケース**：企業AI基盤、複数エージェント、社内ルールが重要な業務。
- **向き**：統治が必要な組織。
- **不向き**：個人用途の軽量AI。
- **要素技術**：policy/rule repository、constitution prompt、policy-as-code、eval suite。

# ステップ4：「程度」の選定基準

AIエージェント設計では「やるか/やらないか」より「**どの程度やるか**」が支配的に重要になる。極端はどちらも害をなす。本章では主要な連続量パラメータについて、**過小の害／過大の害／決め方の基準**を与える。値はあくまで出発点であり、本番トレース（I-1）と評価（I-2）で継続調整する前提とする。

## 4.1 タイムアウト

- **過小の害**：正常な長セッションを途中で殺し、すでに費やした高コストな処理（A-5の予算）を無駄にする。最初のトークンまで時間がかかるモデルで誤発火する。
- **過大の害**：ハングしたリクエストがコネクション・コスト・UXを蝕み、障害検知が遅れる。
- **基準**：
  - ストリーミングなら「**最初のトークンまで（TTFT）**」と「**全体（または無進捗時間）**」を別々に設定する。全体タイムアウトより「**N秒進捗がなければ打ち切り**」のアイドルタイムアウトが実用的。
  - 値は「観測P95レイテンシ × 1.5〜2」を起点に、ツール呼び出し回数・モデルサイズで調整。
  - ステップ単位（ツール1回、LLM1回）と、セッション全体（A-5の `deadline_at`）の二層で持つ。

## 4.2 リトライ回数とバックオフ

- **過小の害**：一時的なレート制限・5xxで容易に失敗し、可用性が落ちる。
- **過大の害**：1回が高コストなため、無制限リトライはコスト事故・輻輳の増幅・障害の長期化を招く。
- **基準**：
  - **指数バックオフ＋ジッタ**で、ネットワーク/レート系の一時障害は最大2〜3回。
  - リトライは**冪等なステップのみ**。副作用ツールはリトライ前に冪等キーで重複防止（A-6）。
  - 「再実行で直る失敗（429/5xz/形式違反）」に限定。ハルシネーション等「再実行で直る保証のない失敗」はリトライでなく**戦略変更**（プロンプト変更・モデル変更・人間エスカレーション）へ。
  - リトライ総量は**セッションのコスト/時間予算（A-5）に従属**させ、予算が尽きたらリトライしない。

## 4.3 自律ループ予算（max_steps / max_tool_calls / max_cost / deadline）

- **過小の害**：複雑タスクが途中で打ち切られ未完成になる。
- **過大の害**：暴走・過剰探索・請求事故（A-5の主眼）。
- **基準**：
  - タスク種別ごとに上限を分けて持つ（FAQ＝数ステップ、Deep Research＝数十ステップ）。
  - 上限到達は失敗でなく**分岐点**として扱う：部分結果返却／追加承認要求／簡易モデル切替／人間エスカレーション。
  - 「進捗していないループ」を別途検知（同一ツール同一引数の反復、スコア無改善のN回連続）してデッドマンスイッチを引く。

## 4.4 ログ／トレースの粒度（プロンプト本文をどこまで残すか）

- **過小の害**：障害の事後再構成・eval・リプレイ（I-1〜I-3）ができない。
- **過大の害**：全プロンプト/全レスポンスを検索インデックスに入れると、サイズ・コスト・PIIリスクで破綻する。
- **基準（程度の代表例）**：
  - **常時・トレース基盤へ**：メタデータ（モデル、版、トークン数、レイテンシ、コスト、相関ID、ツール名、成否、確信度、入出力ハッシュ）。
  - **本文はオブジェクトストレージへ退避**し、トレースから参照リンク。検索基盤にはハッシュ/要約/先頭数百文字のみ。
  - **サンプリング保存**：全件本文が重ければ「エラー時・低評価時・ランダムN%」のみフル保存。
  - **PII**：保存前にマスキング（G-2）。保持期間（TTL）を業務・規制で定める。

## 4.5 コンテキスト投入量（top-k・トークン予算）

- **過小の害**：必要な根拠が欠け、ハルシネーションや的外れが増える。
- **過大の害**：無関係情報がノイズになり精度低下（lost in the middle）。コストとレイテンシを直撃。
- **基準**：
  - 関連度でランキングし上位のみ注入。RAGの取得件数はリランカーで絞る。
  - タスクごとに「**コンテキスト予算（トークン上限）**」を設定し、Context Pack（E-2）で予算内に圧縮。
  - 長履歴は全文保持でなく要約に畳む（E-1のconsolidation）。

## 4.6 メモリの保持と忘却

- **過小の害**：継続性・パーソナライズが失われ、毎回ゼロから。
- **過大の害**：肥大・コスト増・古い情報の混入（E-4が解く問題）。
- **基準**：「重要度 × 鮮度 × 参照頻度」で残すものを選別。古い詳細は要約へ。TTLと再確認要求で失効管理。ユーザーが消去できる権利を設計に含める。

## 4.7 温度・決定性

- **過小の害**：低温度すぎると多様性・創造性が失われ、同型の失敗を繰り返す。
- **過大の害**：高温度すぎると不安定で再現性・検証性が落ちる。
- **基準**：抽出・分類・構造化出力（C-2）は低温度。創作は中〜高温度。重要判断は**低温度＋構造化出力＋スキーマ検証**で「緩い自然言語」を「硬い契約」に締める。アンサンブル（B-4）では多様性確保のため意図的に温度を上げる。

## 4.8 ガードレールの強度（誤検知 vs 見逃し）

- **過小の害**：機密漏洩・有害出力・インジェクションを見逃す。
- **過大の害**：正当な入力を弾く過剰ブロックで有用性が落ちる。
- **基準**：リスクの高い経路（外部公開・機密データ・副作用あり）ほど厳格に、低リスク経路は軽量チェック。**誤検知率（false positive）と見逃し率（false negative）を計測**し、業務の許容点で閾値を動かす。レイテンシ予算が厳しければ重い検査は非同期/サンプリングに。

## 4.9 検証の厳しさ（リフレクション反復・検証コスト）

- **過小の害**：誤り・要件未達が下流へ流出。
- **過大の害**：検証コストが生成コストを超え、収束しない揉め（特に検証もLLM任せの場合）。
- **基準**：**決定論的検証（コンパイル/テスト/スキーマ/事実照合）を最優先**し、LLM-as-judgeは補助に。反復は必ず上限つき（F-3）。検証コスト＞再生成コストになる領域では検証を簡略化する。

## 4.10 キャッシュの積極度（類似度しきい値・TTL）

- **過小の害**：しきい値が厳しすぎ/TTLが短すぎてヒット率が低く、節約効果が出ない。
- **過大の害**：しきい値が緩すぎる誤ヒットで「似て非なる答え」を返す。TTLが長すぎて陳腐化。
- **基準**：完全一致を一次、セマンティックを二次（H-2）。誤ヒットの害が大きい領域は類似度しきい値を高く、TTLを短く、または無効化。パーソナライズ・時系列依存・厳密性が必要な処理ではセマンティックキャッシュを使わない。

## 4.11 人間承認の範囲（自動可／要承認／禁止の境界）

- **過小の害**：何でも要承認にすると承認者がボトルネック化し、エージェントの価値が消える。
- **過大の害**：何でも自動にすると不可逆な誤実行リスク。
- **基準**：**影響の不可逆性 × 影響額/範囲 × 確信度**でリスク分類器を作り、「自動可／要承認／禁止」に振り分け（F-5）。導入初期は広めに要承認にし、evalとトレースで安全性を確認しながら自動化範囲を段階拡大（L-1）。

## 4.12 モデルの強さ（カスケードの昇格しきい値）

- **過小の害**：弱いモデルに留めすぎて品質不足、再試行で結局高コスト。
- **過大の害**：強いモデルに倒しすぎてコスト・レイテンシ過剰。
- **基準**：難易度推定と確信度/検証結果で昇格（H-1）。「弱モデルで解いて検証不合格なら強モデルへ」のカスケードを基本に、トラフィックの何%が昇格しているかを監視してしきい値調整。

## 4.13 カナリアの段階と統計的有意性

- **過小の害**：いきなり全量投入で回帰を全ユーザーに波及。
- **過大の害**：刻みが細かすぎ/期間が長すぎて改善の反映が遅い。
- **基準**：1%→5%→25%→100%の多段で、各段の品質/コスト/エラー率が基準割れなら自動ロールバック（I-4）。トラフィックが少なく統計的有意性が出ない場合はカナリアでなくオフラインeval（I-2）とリプレイ（I-3）に重心を置く。

# ステップ5：「相反する仕組み」の選定基準

設計では二者択一に見える対立が頻出する。実務の答えは多くの場合「どちらか一方」でなく「**どこで線を引くか**」である。各軸について判断基準を示す。

## 5.1 同期 vs 非同期

| 判断軸 | 同期を選ぶ | 非同期を選ぶ（A-1/A-2） |
|---|---|---|
| 処理時間 | 数秒以内で収まる | 数十秒〜数分以上 |
| UX | 対話的に即応が必要 | 完了後の通知で足りる |
| 失敗時コスト | やり直しが安い | 途中失敗の再実行が高い→耐久セッション併用 |
| 接続維持 | コネクション保持が現実的 | 保持コスト/タイムアウトが問題化 |

**基準**：チャットの単発応答は同期＋ストリーミング（A-3）で体感改善。バックグラウンドの複数ステップ自律タスクは非同期＋ジョブ＋耐久セッション。**ハイブリッド**（即時に部分結果をストリーム、重い処理を非同期化して後追い通知）が実用上の最適解になりやすい。

## 5.2 シングルエージェント vs マルチエージェント

| 判断軸 | シングル | マルチ（B-2/B-3/B-5） |
|---|---|---|
| タスク構造 | 単一責務で完結 | 明確に分解でき専門性が分かれる |
| プロンプト | 1つに収まる | 1プロンプトが肥大し精度低下 |
| コスト/レイテンシ | 制約が厳しい | 余裕があり品質優先 |
| デバッグ | 単純さ優先 | 複雑な協調を許容できる |

**基準**：**まずシングルで始める**。「責務過多で精度が落ちる」「並列化で速度/専門性の利得が明確」になって初めてマルチへ。マルチはコスト・レイテンシ・協調の複雑性・障害点を増やす。**安易なマルチ化は解決する問題より作る問題が多い**。

## 5.3 汎用（ジェネラリスト）vs 専門（スペシャリスト）エージェント

**基準**：入力種別が多様で各々別処理が要る → ルーター＋専門群（B-3）。入力が一様 → 単一汎用。専門化はコンテキストを絞れて精度・コストで有利だが、種類増で管理コストとルーティング誤りが増える。能力レジストリ（J-3）で選定根拠を管理する。

## 5.4 集中型オーケストレーション vs 分散型協調

**基準**：制御・監査・予測可能性が重要 → 集中型（オーケストレーター/ステートマシン、B-1/B-2）。動的・探索的でどの専門家がいつ要るか事前に決められない → 分散型（ブラックボード/イベント駆動、B-5）。本番業務は概ね集中型が安全。分散型は柔軟だがデバッグ・保証が難しい。

## 5.5 決定論的バックボーン vs 自律エージェント（どこまでLLMに委ねるか）

**基準**：状態遷移・権限・金額計算・DB更新は決定論コードに残し、曖昧処理・生成・探索だけをエージェントに委ねる（B-1）。委任度は「**誤りの不可逆性とコスト**」で決める。規制・監査が重い領域ほど決定論側に寄せ、探索的研究・創作ほどエージェント側に寄せる。

## 5.6 フレームワーク利用（buy）vs 自前実装（build）

**基準**：標準パターンを素早く立ち上げたい → フレームワーク（LangGraph/CrewAI/Agents SDK）。細かい制御・性能・依存最小化・長期保守が重要 → 薄い自前実装。エージェント領域は進化が速くロックインリスクがあるため、**コア制御フローは自前 or 薄い抽象（J-1）、周辺（観測・評価）はツール活用**の折衷が堅実。

## 5.7 プロバイダ抽象化 vs プロバイダ固有最適化

**基準**：マルチプロバイダ・フェイルオーバー・移行容易性が重要 → 抽象化（J-1/J-2、最大公約数のIF）。特定モデル固有機能（prompt caching、structured outputs、特殊ツール）で性能/コストを絞り出したい → 固有最適化。両立策：抽象化を基本としつつ、ホットパスだけ固有機能を「逃がし口」で使う。

## 5.8 RAG vs ロングコンテキスト vs ファインチューニング

| 手段 | 向く | 不向き |
|---|---|---|
| RAG（F-1） | 知識が大量・更新頻繁・出典が要る | 推論力が主役/検索精度が出ない |
| ロングコンテキスト | 文書が中規模で毎回全体が要る | 巨大/高頻度（コスト・精度劣化） |
| ファインチューニング | 様式・口調・形式の固定化、専門タスク反復 | 知識の鮮度が要る/データが少ない |

**基準**：鮮度・出典が要る→RAG。振る舞い/様式の固定化→FT。排他でなく**併用**が普通（FTで様式、RAGで知識）。

## 5.9 事前検証 vs 事後検証

**基準**：入力の不正・インジェクション・トピック逸脱 → 事前（入力ガード、F-2）。出力の形式・事実・安全性 → 事後（出力ガード＋再生成、F-3）。事前で弾ければLLM呼び出しを節約できるが、事後でしか分からない品質問題も多い。**両方を薄く併置**が基本。

## 5.10 ステートレス vs ステートフル

**基準**：1回完結・スケール容易性優先 → ステートレス。長期対話・継続パーソナライズ → 実質ステートフルだが、**状態は外部メモリ（E-1）/耐久セッション（A-2）に持たせ、エージェント本体はステートレスに保つ**（水平スケールしやすい）のが定石。

## 5.11 早期（eager）vs 遅延（lazy）ツール実行

**基準**：必要性が確実なら早期に並列実行してレイテンシ短縮。必要か不確実・高コスト・副作用ありなら遅延（必要と確定してから）。**副作用ツールは原則レイジー＋承認ゲート（D-3/F-5）**。

## 5.12 構造化出力 vs 自然言語出力

**基準**：下流が機械処理（基幹連携・ツール引数）→ 構造化必須（C-2、JSON Schema＋検証）。最終的に人間が読む対話 → 自然言語。境界（C-1/L-2）で「**内側は構造化、外側は自然言語**」と層を分けるのが実用的。

## 5.13 自前/オンプレモデル vs API/マネージド

**基準**：データ主権・コンプラ・大量定常トラフィックのコスト → 自前/オンプレ。最新性能・運用負荷最小・変動トラフィック → API。可用性は外因依存のため、APIは**フォールバック前提**（H-4）で。

## 5.14 自己検証（同一モデル）vs 独立検証器

**基準**：自己採点は甘くなりがち。誤りコストが高いほど**生成器と独立な検証器**（できれば決定論的、F-3）を置く。低リスクの内部メモなら自己検証で十分。

## 5.15 アンサンブル（N並列）vs 単発

**基準**：正誤が判定可能で誤りコストが高い高価値タスク → アンサンブル/best-of-N（B-4）。コスト・レイテンシ制約が厳しい大量処理 → 単発。難問のみNを増やす適応制御が現実的。

## 5.16 プロンプトで守る vs ポリシー/実行基盤で守る

**基準**：この軸の答えは明確で、**プロンプトはセキュリティ境界にならない**。権限・承認・検証・隔離・ポリシー（D-2/F-4/F-5/G系/L-3）で守る。プロンプトは品質・ふるまいの調整に使い、安全保証は実行基盤側に置く。

# ステップ6：アーキテクチャパターン全体の統合・選定・組み合わせ

## 6.1 パターン間の依存関係（積み上げ構造）

パターンは排他でなく、層として重ねて使う。代表的な依存は次のとおり。

- **A-2 Durable Session** は **A-4 中断 / F-5 人間承認 / I-3 リプレイ** の前提（状態の永続化がなければ中断も承認待ちもリプレイもできない）。
- **D-1 Tool Gateway** は **D-2 最小権限 / D-3 Dry-Run / I-1 監査** の実装点（ツール呼び出しが一点に集まるから統制できる）。
- **B-2 Planner** は **K-2 Editable Plan** の前提（計画が構造化アーティファクトだから編集できる）。
- **I-1 Trace** は **I-2 Eval / I-3 Replay** の素材（記録がなければ評価も再現もできない）。
- **F-4 Policy-as-Code** と **L-3 Constitution** は **D-2 / F-5 / G系** を貫く統治の背骨。
- **C-1/C-2 契約化** は **B-1 決定論的バックボーン** が成立する条件（出力が契約化されて初めて殻で包める）。
- **A-5 予算** は **4章すべての程度パラメータ**（タイムアウト・リトライ・カスケード・アンサンブル）の上位制約。

## 6.2 成熟度別 導入ロードマップ

- **段階0：PoC** … 同期呼び出し＋構造化出力（C-2）だけでよい。まず手動評価で品質感を掴む。
- **段階1：最小本番** … A-1 非同期 ＋ B-1 決定論的バックボーン ＋ C-2 構造化出力 ＋ H-4 縮退/フォールバック ＋ A-5 予算制御 ＋ I-1 トレース。これで「落ちない・暴走しない・壊れた出力を垂れ流さない・観測できる」。
- **段階2：副作用と統制** … A-2 耐久セッション ＋ A-6 Saga ＋ D-1 ツールGW ＋ D-2 最小権限 ＋ D-3 Dry-Run ＋ F-5 人間承認。金銭・物理が絡むほど効く。
- **段階3：信頼性と品質** … F-1 根拠優先 ＋ F-3 検証器 ＋ B-4 アンサンブル ＋ I-2 評価CI ＋ I-4 版固定/カナリア。確率を品質保証へ変える。
- **段階4：規模と統治** … E系 メモリ ＋ G系 セキュリティ/テナント分離 ＋ H-1〜3 コスト最適化 ＋ J系 抽象化 ＋ L-3 憲法。組織横断・マルチテナント・継続改善に耐える基盤へ。

## 6.6 設計原則（まとめ）

1. **同期リクエストでなく、長時間セッションとして扱う**（Session / Job / Workflow / Checkpoint を一級概念に）。
2. **LLMを業務ロジックの中心に置きすぎない**（中核の状態遷移・権限・金額計算・DB更新は決定論コードに残す）。
3. **自然言語をそのまま実行しない**（Intent → Plan → Action → Evidence → Approval へ構造化する）。
4. **ツール実行を最重要リスク境界として設計する**（Gateway・最小権限・Dry-Run・Sandbox・監査を必須に）。
5. **メモリはデータ管理機能として扱う**（保存・忘却・権限・鮮度・根拠・テナント分離）。
6. **プロンプトでなく、ポリシーと実行基盤で守る**（権限・承認・検証・隔離で守る）。
7. **評価をCI/CDに組み込む**（モデル・プロンプト・ツール・RAG変更はevalなしに本番反映しない）。
8. **観測性を通常システム以上に重視する**（なぜその判断か・何を見たか・どのツールを呼んだか・いくらかかったか）。
9. **コストを設計対象にする**（ルーティング・キャッシュ・予算・早期停止・フォールバック）。
10. **「やるか」でなく「どの程度か」を設計する**（極端を避け、トレースとevalで継続調整する）。
11. **完全自律でなく、制御された自律性を目指す**。

> 最も凝縮した一文を選ぶなら——**AIエージェントの本番化とは、賢いが信用しきれない・遅く・高コストで・騙されうる実行主体を、決定論的な殻・契約・権限・検証・予算・観測・統治の中に安全に閉じ込め、その周囲に従来のソフトウェア工学（状態管理・トランザクション・監査・テスト・デプロイ・可観測性）を配置し、各設計判断を「程度」と「相反軸」の明示的なトレードオフとして決める工学である。**
