# F12 · 集中 vs 連合メモリ

!!! abstract "一言で"
    エージェントが参照・蓄積する記憶（ユーザープロファイル、対話履歴、学習データ）を
    単一の中央ストアに統合するか、各データソースに分散して保持するかという設計分岐。
    横断的な個別化と、データ主権・越境規制のバランスが核心となる。

## 集中メモリを選ぶ条件

- **複数チャネル・システムを横断**してユーザーの文脈を一貫させたい（例: Slack での会話履歴を Salesforce のエージェントでも活用する）
- ユーザーの嗜好・過去の意思決定を統合し、**パーソナライズされた応答**を提供することが価値の中核
- データの所在が単一リージョンに収まり、越境転送の規制制約が少ない
- 全社統一の顧客プロファイルストア（Salesforce CDP 等）を中心に、全エージェントが同一の顧客理解を共有する構成
- 記憶の整合性を強整合で担保する必要がある（例: 顧客の解約意思を全エージェントが即座に反映）

## 連合メモリを選ぶ条件

- **データ主権・GDPR・業界規制**により、データを元のシステムから移動できない
- 各部門・システムが独自のデータ管理ポリシーを持ち、中央集約に組織的抵抗がある
- データソースごとに**アクセス権限が異なり**、統合すると最小権限の原則に違反する
- EU の顧客データと日本の従業員データを別リージョンに保持しつつ、エージェントが必要なときだけ参照する構成
- ServiceNow のインシデント履歴は IT 部門の管轄、Workday の人事データは HR 部門の管轄として、それぞれのソースが権限を制御する

## ハイブリッド（併用）

- **メタデータは集中、本体は連合**: ユーザーの識別子・最終アクセス日・嗜好サマリなど軽量なメタデータは中央に持ち、詳細データ（対話ログ本文、添付ファイル）は各ソースに残す
- **キャッシュ層による仮想集中**: 各ソースからリアルタイムにフェッチした結果を短期キャッシュに統合し、リクエスト単位で疑似的な集中ビューを構築する。データの永続的な複製を避けられる
- **コンテキストブローカー**: エージェントのリクエスト時に必要なデータを各ソースから収集・結合するブローカーを置き、集中ストアを持たずに横断的な文脈を提供する

## 判断フロー

1. 対象データに**越境転送規制**（GDPR、中国データセキュリティ法等）が適用されるか → Yes なら**連合**が前提
2. 複数システムを横断した**一貫したパーソナライズ**が必要か → Yes なら**集中**を検討
3. 各データソースの**アクセス権限モデル**が統一可能か → No なら**連合**で各ソースの権限を尊重する
4. データ量と更新頻度から、中央への**同期コスト**が許容範囲か → No なら**連合 + キャッシュ層**
5. ハイブリッドの場合、メタデータと本体の分離基準を定義し、データ分類ポリシーに基づいて仕分ける

## よくある失敗

- **集中ストアに全データを複製**: ストレージコストが膨らむだけでなく、元データとの同期遅延でエージェントが古い情報を参照する。Salesforce で顧客情報を更新したのに、集中メモリには反映されていない状態
- **連合のまま横断クエリを強行**: 各ソースへのリアルタイムクエリが多段になり、レイテンシが目安として 5 秒を超えてユーザー体験が劣化する
- **データ分類の欠如**: どのデータが集中可能でどのデータが越境不可かを事前に分類せず、コンプライアンス違反が本番で発覚する
- **削除権（忘れられる権利）への対応漏れ**: 集中メモリにコピーしたデータの削除が元ソースの削除と連動しておらず、GDPR 上のリスクとなる
- 連合メモリのソース障害時にエージェントがエラーを返す。フォールバック（キャッシュからの応答 + 鮮度警告）を設計しておく必要がある
