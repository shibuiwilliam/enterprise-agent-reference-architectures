# F18 · LLM-judge vs 人手評価

!!! abstract "一言で"
    エージェントの出力品質を LLM に自動評価させるか、人間の専門家が判定するかの選択です。
    評価の速度・スケーラビリティと、判定の信頼性・安全性保証がトレードオフになります。
    実運用では両者を組み合わせ、自動評価で広くカバーし人手評価で深く検証する二段構成が一般的です。

## LLM-judge を選ぶ条件

- CI/CD パイプラインに組み込み、プロンプト変更やモデル更新のたびに回帰テストを自動実行したい場合
- 評価対象が大量（目安として日次 100 件以上）で、人手では物理的にカバーできない場合
- 評価基準が形式化しやすい軸（事実整合性、フォーマット準拠、トーン一貫性など）に限定される場合
- Salesforce のケース要約や Jira チケットの自動分類など、出力の正解パターンがある程度定型化されている場合
- 開発サイクルが速く、週次〜日次でプロンプトを改善するイテレーションを回している場合

## 人手評価を選ぶ条件

- 安全性・倫理性・コンプライアンスに関わる判定で、誤判定のリスクが許容できない場合（金融商品の説明、医療情報、法務回答など）
- 評価基準が主観的・文脈依存であり、「正解」を形式化しにくい場合（顧客への共感表現、ブランドトーンの適切さなど）
- 新規ユースケースの立ち上げ時で、LLM-judge の評価基準自体を設計するためのゴールドデータを作成する必要がある場合
- 規制当局への報告に「人間が検証した」というエビデンスが求められる場合
- LLM-judge のバイアス（自己評価の甘さ、特定パターンへの偏り）が疑われ、キャリブレーションが必要な場合

## ハイブリッド（併用）

- **LLM-judge でフィルタリング → 人手で精査**: 全出力を LLM-judge で自動スコアリングし、低スコア（目安として下位 10〜20%）のみ人手レビューに回します。人手の工数を最も効率的に配分できます。
- **人手評価でゴールドセット作成 → LLM-judge のキャリブレーション**: 専門家が 200〜500 件のゴールドラベルを付与し、LLM-judge の評価プロンプトと採点基準をチューニングします。定期的に再キャリブレーションを実施しましょう。
- **段階ゲート**: 開発環境では LLM-judge のみ、ステージング環境で LLM-judge ＋サンプル人手評価、本番リリース前に人手評価によるゲート承認とします。

## 判断フロー

1. **評価対象の出力は安全性・規制に直結するか？** — YES → 人手評価を必須としましょう（LLM-judge は補助です）。
2. **評価基準を明文化・数値化できるか？** — YES → LLM-judge が有効です。NO → 人手評価から始めましょう。
3. **日次の評価対象件数は？** — 目安として 100 件超 → LLM-judge で自動化しましょう。50 件以下 → 人手評価も現実的です。
4. **ゴールドデータは十分にあるか？** — 200 件以上 → LLM-judge のキャリブレーションが可能です。不足 → まず人手でゴールドセットを構築してください。
5. **評価結果を外部監査に提出するか？** — YES → 人手評価のエビデンスを含めましょう。

## よくある失敗

- **LLM-judge の自己評価バイアス**: 同一モデルファミリーで生成と評価を行い、自分の出力を過大評価する傾向が出ます。評価モデルは生成モデルと異なるプロバイダを使うか、評価プロンプトにバイアス対策を組み込みましょう。
- **人手評価のスケール不足**: 「全件人手レビュー」を方針にして運用が破綻します。Zendesk で日次 500 件の要約を全件レビューすることは現実的ではありません。
- **評価基準の暗黙知化**: 人手評価者の判断基準が明文化されず、評価者間のばらつき（Inter-Annotator Agreement）が低いまま運用されます。
- **LLM-judge の精度を検証しない**: LLM-judge を導入したまま人手との一致率を測定せず、スコアの信頼性が不明なまま意思決定に使い続けてしまいます。
- **評価コストの過小見積り**: LLM-judge の API コストを軽視し、全出力×複数評価軸で評価した結果、生成コストと同程度の評価コストが発生します。サンプリングや評価軸の絞り込みで制御しましょう。

## 評価設計のベストプラクティス

- **評価軸を分離する**: 事実整合性、フォーマット準拠、トーン、安全性など、軸ごとに LLM-judge と人手の適性を判断しましょう。全軸を同一手法で評価する必要はありません。
- **定期的な再キャリブレーション**: LLM-judge のゴールドセットは四半期ごとに更新しましょう。ユースケースの変化やモデルバージョンアップに伴い、評価基準のドリフトが起きます。
- **評価結果の可視化**: Langfuse や Braintrust などの eval プラットフォームで、LLM-judge スコアと人手スコアの推移・乖離をダッシュボード化し、劣化を早期に検知しましょう。
- **評価者間一致率の追跡**: 人手評価を行う場合、Cohen's Kappa 等で評価者間の一致率を計測し、目安として 0.7 以上を維持してください。低い場合は評価基準の再教育を実施しましょう。
