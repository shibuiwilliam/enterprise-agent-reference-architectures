# F17 · 自前構築 vs プラットフォーム購入

!!! abstract "一言で"
    AIエージェント基盤を自社で構築するか、既存プラットフォームを購入・採用するかの選択です。
    差別化要素と技術的自由度を取るか、市場投入速度と運用負荷の低減を取るかが判断の核心になります。
    多くの企業では「全部自前」でも「全部購入」でもなく、レイヤーごとに判断を分けます。

## 自前構築を選ぶ条件

- エージェントの振る舞い自体が事業の競争優位であり、差別化の中核を成す場合（例: 独自の与信判断ロジック、業界特化の推論パイプライン）
- 既存プラットフォームがサポートしない特殊要件がある場合（独自プロトコル、オンプレミス LLM、特殊なデータ分類要件）
- 規制要件によりデータが自社環境から出せず、SaaS 型プラットフォームが利用不可の場合
- 社内に十分なMLエンジニアリングチーム（目安として 5 名以上の専任）が存在し、構築・運用を継続できる場合
- ベンダーロックインを回避し、LLM プロバイダやインフラの切り替え自由度を確保したい場合
- エージェントの評価・改善サイクルを完全にコントロールし、独自の eval パイプラインを構築したい場合

## プラットフォーム購入を選ぶ条件

- 市場投入速度が最優先であり、3 か月以内に本番稼働が求められる場合
- エージェントの用途が標準的（FAQ 応答、ドキュメント検索、定型業務の自動化）で、差別化要素がインフラ層にない場合
- 運用チームが小規模（目安として 3 名以下）で、インフラの監視・スケーリング・障害対応を自前で担えない場合
- Salesforce Einstein, ServiceNow Now Assist, Microsoft Copilot Studio など、既存 SaaS にエージェント機能が組み込まれている場合
- コスト構造として、初期投資を抑えて従量課金で始めたい場合

## ハイブリッド（併用）

- **コア自前＋周辺購入**: オーケストレーション層とビジネスロジックは自前で構築し、LLM 推論（Azure OpenAI, Amazon Bedrock）、ベクトルDB（Pinecone, Weaviate）、監視（Datadog, Langfuse）は SaaS を採用します。最も一般的な戦略です。
- **購入＋カスタム拡張**: Microsoft Copilot Studio や Amazon Bedrock Agents をベースに採用し、自社固有のツール・コネクタ・ガードレールをプラグインとして自前開発します。
- **段階的内製化**: 初期はプラットフォームで素早く立ち上げ、ユースケースが定まった段階で差別化部分を順次内製に切り替えます。

## 判断フロー

1. **この機能は競争優位の源泉か？** — YES → 自前構築を検討しましょう。NO → 購入を優先してください。
2. **既存プラットフォームで要件の 80% 以上をカバーできるか？** — YES → 購入＋カスタム拡張がおすすめです。NO → 自前構築を検討しましょう。
3. **構築・運用を担える専任チームがいるか？** — YES → 自前構築が現実的です。NO → 購入またはハイブリッドを選びましょう。
4. **規制・データ主権の制約でSaaSが使えないか？** — YES → 自前構築（オンプレミス/プライベートクラウド）を選びましょう。
5. **市場投入までの期限は？** — 3 か月以内 → 購入を選びましょう。6 か月以上 → 自前構築も選択肢に入ります。

## よくある失敗

- **NIH 症候群（Not Invented Here）**: 差別化に寄与しない基盤部分まで自前構築し、開発リソースを浪費します。ログ基盤やベクトル DB を独自実装して、保守に追われる例が典型的です。
- **プラットフォームの限界に気づくのが遅い**: 購入したプラットフォームの制約（カスタムツールの上限、プロンプトの文字数制限、特定 LLM への依存）に本番稼働後に直面し、移行コストが膨らみます。
- **ベンダーロックインの過小評価**: 特定プラットフォームの独自 API・独自フォーマットに深く依存し、切り替え時にエージェント定義・プロンプト・評価データの移行が困難になります。
- **運用コストの見積もり漏れ**: 自前構築の初期開発コストだけで判断し、継続的な LLM API コスト、モデル更新対応、セキュリティパッチ適用などの運用コストを軽視してしまいます。
- **購入後のカスタマイズ地獄**: プラットフォームの標準機能で 80% はカバーできたものの、残り 20% のカスタマイズに開発工数の 80% を費やします。事前に「カスタマイズ不可の境界」を確認しましょう。

## 判断の更新タイミング

- **半年ごとの再評価**: プラットフォーム市場は急速に進化しているため、「当時自前が正解だった」判断も半年後には変わりえます。定期的にプラットフォームの機能アップデートを確認しましょう。
- **チーム構成の変化時**: キーパーソンの退職やチーム縮小が発生した場合、自前運用の持続可能性を再評価してください。
- **利用規模の変化時**: エージェントの利用が当初の想定を大幅に超えた場合、プラットフォームのスケーラビリティの恩恵が自前構築のコストを上回ることがあります。
