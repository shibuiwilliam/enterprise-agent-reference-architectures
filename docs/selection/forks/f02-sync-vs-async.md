# F02 · 同期 vs 非同期

!!! abstract "一言で"
    エージェントの応答を「ユーザーが待つ（同期）」か「完了通知を受け取る（非同期）」かの
    選択は、体験設計・アーキテクチャ・スケーラビリティに直結する。数秒で返せる会話的な
    やり取りと、数十秒〜数分かかるバッチ処理やマルチステップ調査では、最適な実行モデルが
    根本的に異なる。

## 同期（A）を選ぶ条件

- ユーザーとの往復対話が価値の源泉であり、応答までの待ち時間が体験を左右する。
- 処理時間が数秒以内（目安 5 秒未満）で完了する見込みがある。
- Slack のチャットボット、Zendesk のライブチャット対応など、リアルタイム性が求められる。
- ツール呼び出しが 1〜2 回で済み、外部 API のレイテンシも低い。
- ストリーミング出力で体感速度を補える（LLM の生成をトークン単位で逐次表示）。

## 非同期（B）を選ぶ条件

- 処理時間が数十秒〜数分以上かかる（複数 SaaS の横断調査、大量データの集計・分析）。
- Salesforce の商談データ分析、Jira の全スプリント横断レポート生成など、重い処理。
- 同時に大量のリクエストが発生し、バックプレッシャー制御が必要。
- 結果をメールや Slack 通知で受け取れば十分で、ユーザーが画面の前で待つ必要がない。
- イベント駆動（Webhook / CDC）でエージェントが起動するケース。
- 長時間実行中の障害からの再試行・チェックポイントが必要。

## ハイブリッド（併用）

多くの本番システムでは同期と非同期を組み合わせる。典型的なアプローチは以下の通り。

- **同期開始 → 非同期エスカレーション**: 最初は同期で応答し、処理が長引く場合（目安 10 秒超）
  に「バックグラウンドで処理中です」とユーザーに伝え、ジョブキューに移行する。
  完了後に Slack / メールで通知する。
- **ストリーミング＋進捗表示**: 同期的にストリーミング出力しつつ、裏でツール呼び出しを
  並列実行する。ユーザーには中間結果を逐次表示し、体感待ち時間を短縮する。
- **ServiceNow のインシデント対応**: 一次回答は同期チャットで即座に返し、根本原因分析や
  類似インシデントの横断調査は非同期ジョブで実行する。

## 判断フロー

1. **処理時間の見込みは？** — 5 秒未満なら同期。10 秒超なら非同期を検討。5〜10 秒は
   ストリーミングで同期を維持できるか評価する。
2. **ユーザーは画面の前で待つか？** — 待たない（バッチ処理、定期レポート）なら非同期一択。
3. **往復対話が価値を生むか？** — 追加質問・確認・修正のラリーが必要なら同期。
4. **同時リクエスト数のピークは？** — 数千以上のスパイクがあるなら、ジョブキューで
   バックプレッシャーを制御する非同期が安全。
5. **障害時のリカバリ要件は？** — 途中で失敗した場合にチェックポイントから再開したいなら
   非同期＋永続キューが必須。

## よくある失敗

- **すべてを同期で実装する** — 重い処理でタイムアウトが頻発し、API Gateway の 30 秒制限に
  引っかかる。ユーザーは空白画面を見続ける。
- **すべてを非同期にする** — 簡単な質問への回答にもキュー経由の遅延が入り、
  チャット体験が著しく劣化する。
- **タイムアウト設計の欠如** — 同期エンドポイントに長時間処理が流れ込み、コネクションプールが
  枯渇してシステム全体が停止する。
- **進捗通知の不在** — 非同期ジョブの完了を通知しないため、ユーザーが結果を取りに来ない。
  Slack や Teams への Webhook 通知を必ず組み込む。
