# F01 · 従業員向け vs 顧客向け

!!! abstract "一言で"
    AIエージェントの利用者が「社内従業員」か「外部顧客」かによって、信頼モデル・データ境界・
    ガードレール強度・障害時の影響範囲がまったく異なります。この分岐を初期に誤ると、
    顧客向けに甘い設計を適用して情報漏洩を招くか、従業員向けに過剰な制約を課して
    生産性を潰すかのどちらかに陥ります。

## 従業員向け（A）を選ぶ条件

- 利用者が雇用契約・NDA で拘束されており、入力を信頼できる前提が成り立つ。
- エージェントが参照するデータが社内ナレッジベース（Notion / Confluence / Box）や
  業務システム（Salesforce / ServiceNow / Workday）に閉じている。
- 操作の失敗コストが「業務非効率」「手戻り」で済み、多くは可逆です。
- 認証が社内 IdP（Okta / Entra ID）の SSO に統一されている。
- 同時利用者数が数千〜数万規模で、業務時間帯に集中します。

## 顧客向け（B）を選ぶ条件

- 利用者が外部ユーザーであり、敵対的入力（ジェイルブレイク・間接インジェクション）を
  前提にした設計が必要。
- エージェントの出力が顧客の意思決定やブランドイメージに直結し、誤回答・情報漏洩が
  法的責任やブランド毀損に発展します。
- 顧客ごとにデータを厳密に隔離する必要がある（自テナント・自顧客のデータのみ参照可能）。
- 認証が顧客 IdP（Auth0 / CIAM）で、匿名アクセスも想定される。
- 同時利用者数が数万〜数百万に達し、24/365 のスパイクに耐える必要がある。

## ハイブリッド（併用）

実務では従業員向けと顧客向けの両方を同一プラットフォーム上で運用するケースが大半を占めます。
共有できるのはオーケストレーション基盤やモデルゲートウェイ（AI Gateway）までであり、
**データ到達経路・ガードレール・監査ログは分離する**のが基本方針となります。

- Shopify 連携 EC で、顧客向け問い合わせエージェントと社内受発注オペレーション支援
  エージェントを同一基盤上の別プレーンとして構築します。
- Zendesk 上の顧客向け一次対応エージェントには公開可能ナレッジの投影（read model）
  のみを参照させ、社内 Notion / Box の機密データへの直接到達を遮断します。

## 判断フロー

1. **利用者は誰か？** — 雇用関係のある従業員のみなら A を選びましょう。外部顧客が含まれるなら B を検討してください。
2. **両方が含まれるか？** — 含まれる場合、A と B を別プレーンとして設計し、データ経路を分離してください。
3. **顧客向けの出力に社内データが混入するリスクはあるか？** — ある場合、DLP（データ漏洩防止）
   を出力パイプラインに組み込みましょう。
4. **規模とスパイク特性は？** — 顧客向けは非同期ジョブ＋負荷制御（P22）の併用を検討してください。
5. **ガードレールの強度差を明文化したか？** — 顧客向けはジェイルブレイク検知・トピック制限・
   トーン制御・拒否方針を従業員向けより格段に厳格に設定しましょう。

## よくある失敗

- **従業員向けの設計をそのまま顧客向けに流用する** — 社内で許容されるデータアクセス範囲を
  顧客向けに適用してしまい、他テナントの情報が漏洩します。
- **顧客向けの厳格なガードレールを全社展開する** — 従業員の業務効率が著しく低下し、
  エージェント導入の ROI が出なくなります。
- **プレーン分離を「後で対応」にする** — 初期は単一スタックで構築し、顧客向けリリース時に
  分離コストが膨れ上がります。分離は初期設計時に決定しましょう。
- **監査ログを混在させる** — 顧客 PII と社内データが同一ログストアに混在し、
  データ保持期間やアクセス制御の管理が破綻します。
