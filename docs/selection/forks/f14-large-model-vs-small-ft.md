# F14 · 汎用大モデル vs 小型FT

!!! abstract "一言で"
    エージェントの推論エンジンとして汎用大規模モデル（GPT-4 クラス）をプロンプトで制御するか、
    タスク特化の小型ファインチューニング（FT）モデルを使うかという設計分岐です。タスクの多様性・
    呼び出し頻度・コスト制約・変化の速度によって最適解が変わります。

## 汎用大モデル + プロンプトを選ぶ条件

- タスクが**多様で事前に列挙しきれない**（例: 社内ヘルプデスクで、IT・総務・人事・経理の幅広い問い合わせに対応）
- ビジネス要件の変化が速く、**モデルの再学習なしに即座にプロンプトで対応**を変更したい
- 呼び出し頻度が低〜中程度（目安として月間数万リクエスト以下）で、1件あたりのコストが許容範囲
- 新しいツールや API の追加に対して、プロンプトに説明を加えるだけで対応可能
- Slack Bot が全社横断で質問を受け付け、Salesforce・Jira・Confluence・Workday を横断して回答する汎用エージェント

## 小型FT モデルを選ぶ条件

- タスクが**狭く定義**され、入出力パターンが安定している
- 呼び出し頻度が**高い**（目安として月間数十万〜数百万リクエスト）で、1件あたりのコスト削減が重要
- **レイテンシ要件が厳しい**（目安として P95 で 500ms 以内）で、大規模モデルの推論時間が許容できない
- 十分な量の教師データ（目安として数千〜数万件）を用意・維持できる体制がある
- Zendesk のチケット分類（カテゴリ・優先度・担当チームの自動振り分け）を毎日数万件処理するエージェント
- Shopify の商品レビューから感情分析とタグ付けを行い、ダッシュボードに反映するバッチエージェント

## ハイブリッド（併用）

- **ルーター + 専門モデル**: 汎用大モデルでリクエストの意図を分類し、定型タスクは小型 FT モデルにルーティングします。意図分類のコストは小さく、専門タスクの推論コストを削減できます
- **FT で下書き、大モデルで仕上げ**: 小型 FT モデルで高速に下書きを生成し、品質が重要なケースだけ汎用大モデルで検証・修正します
- **段階的移行**: 初期はすべて汎用大モデルで運用し、呼び出しログから教師データを蓄積して、高頻度タスクから順に小型 FT に置き換えます

## 判断フロー

1. タスクの種類が**10 以上**あり、頻繁に新しいタスクが追加されるか → Yes なら**汎用大モデル**を選びましょう
2. 月間リクエスト数が**数十万件以上**か → Yes なら小型 FT のコストメリットを試算してください
3. レイテンシ要件が P95 で**500ms 以内**か → Yes なら**小型 FT** を優先的に検討しましょう
4. 教師データを**数千件以上**用意・継続的に更新できるか → No なら汎用大モデルに留めましょう
5. ビジネス要件の変更頻度が**月次以上**か → Yes なら再学習コストを考慮し、汎用大モデルを維持する方が有利な場合が多いです
6. 判断が微妙な場合 → まず汎用大モデルで運用開始し、ログから教師データを構築して FT の効果を検証しましょう

## よくある失敗

- **早すぎる FT 投資**: 十分なログが溜まる前に FT モデルを構築し、精度が汎用大モデルを下回ります。最低でも目安として 3 か月分のログを蓄積してから FT を検討してください
- **FT モデルの陳腐化**: ビジネスルールや製品ラインナップが変わったのに教師データが更新されず、古い知識で回答し続けます。再学習パイプラインの自動化が必須です
- **コスト比較の誤り**: API 呼び出しコストだけで比較し、FT モデルの学習コスト・推論インフラ（GPU）運用コスト・教師データ管理コストを見落とします
- **汎用大モデルへの過依存**: 全タスクを大モデルで処理し、月額コストが目安として数万ドルに達してスケールが止まります
- **品質評価の欠如**: FT モデルの精度を定期的に評価せず、モデルドリフトに気づきません。汎用大モデルとの A/B テストを継続的に行う仕組みが必要です
