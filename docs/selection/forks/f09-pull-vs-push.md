# F09 · Pull vs Push

!!! abstract "一言で"
    エージェントの起動トリガがユーザーの明示的な要求（Pull）か、システムイベントによる
    自動起動（Push）かという設計分岐。ユーザー体験・負荷特性・権限モデルが大きく異なるため、
    ワークロードごとに適切な起動モデルを選定する必要がある。

## Pull（ユーザー起点）を選ぶ条件

- ユーザーが**意図を持って質問・指示を出す**対話型タスク（例: Slack Bot への問い合わせ）
- 実行タイミングがユーザーの判断に依存し、事前にスケジュールできない
- 実行のたびにユーザーのコンテキスト（選択中の画面、直前の操作）が必要
- Salesforce の商談画面から「この顧客の過去のサポート履歴をまとめて」と指示するエージェント
- 権限はリクエスト発行者のセッションに紐づくため、認可モデルがシンプル

## Push（イベント起点）を選ぶ条件

- **異常検知・予防的アクション**が価値の中核（例: Jira チケットの SLA 超過を検知して自動エスカレーション）
- トリガが明確なシステムイベント（Webhook、CDC、スケジュール）で定義できる
- ユーザーが気づく前に対処することで損失を防ぐ
- ServiceNow のインシデントが P1 に昇格した瞬間にオンコール担当へ通知と初期分析を送るエージェント
- 夜間バッチで Workday の勤怠異常を検出し、翌朝マネージャーに是正案を提示するエージェント

## ハイブリッド（併用）

- **Push で気づかせ、Pull で深掘り**: イベント駆動でサマリ通知を送り、ユーザーが詳細を知りたければ対話に移行する。Zendesk で顧客満足度が急落した際にアラートを飛ばし、担当者が「原因を分析して」と指示する流れ
- **Pull の裏で Push を仕込む**: ユーザーの問い合わせ対応中に、関連するイベント（同じ顧客の未解決チケット）をバックグラウンドで監視し、追加情報をプロアクティブに提示する
- 多くの本番環境では Pull と Push の比率は目安として 7:3〜5:5 に落ち着く

## 判断フロー

1. トリガは明確なシステムイベント（Webhook / CDC / cron）で定義できるか → Yes なら **Push** を検討
2. ユーザーの意図・コンテキストが実行に不可欠か → Yes なら **Pull** を主軸にする
3. 「ユーザーが気づく前に対処する」ことに価値があるか → Yes なら **Push** を優先
4. 両方に該当する場合 → Push で通知、Pull で対話のハイブリッド構成にする
5. Push の場合、イベント量の見積もりとスロットリングの設計を必ず行う

## よくある失敗

- **Push 過多で通知疲れ**: イベントの閾値を適切に設定せず、大量のアラートがユーザーに押し寄せて無視される。Slack チャンネルが未読通知で埋まる状態
- **Pull のみで予防機会を逃す**: 異常を検知できるイベントソースがあるのに Pull だけに頼り、ユーザーが気づいた時には SLA 違反が確定している
- **Push エージェントの権限設計漏れ**: ユーザーセッションに紐づかないため、どの権限で API を叩くかが曖昧になり、過剰権限のサービスアカウントを使ってしまう
- **イベントストームへの備え不足**: デプロイ直後に大量の CDC イベントが発生し、Push エージェントがバックログを処理しきれずタイムアウトが連鎖する
