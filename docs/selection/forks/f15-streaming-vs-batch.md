# F15 · ストリーミング vs バッチ出力

!!! abstract "一言で"
    エージェントの応答をトークン単位で逐次送出するか、全文生成後に一括返却するかの選択です。
    ユーザー体感速度と後続処理の整合性がトレードオフになります。対話型チャットと
    バックエンド処理では最適解が異なるため、用途ごとに使い分ける設計が必要です。

## ストリーミングを選ぶ条件

- ユーザーが画面上でリアルタイムに応答を読む対話型UIを持つ場合（Slack Bot、社内チャット、カスタマーサポート窓口など）
- 応答が長文（目安として 500 トークン超）になり、全文生成まで待つと体感待ち時間が数十秒に達する場合
- First Token Latency（最初のトークンが届くまでの時間）が UX の主要 KPI である場合
- Zendesk や Intercom のようなカスタマーサポートツールで、顧客が「応答中」の表示を見ながら待つシナリオ
- 途中経過を見てユーザーが早期にキャンセル・方向修正できることに価値がある場合
- 社内ナレッジ検索エージェントで、回答を段階的に表示しながらソースを提示する UX を実現したい場合

## バッチ出力を選ぶ条件

- 後続処理が JSON・CSV・SQL など構造化フォーマットの完全な出力を必要とする場合（部分的な JSON はパースできない）
- Salesforce のレコード更新や Workday の人事データ処理など、API 呼び出しの入力として全文が必要な場合
- 出力に対してガードレール検証・PII マスキング・フォーマットバリデーションを全文に対して実行する必要がある場合
- 非同期ジョブとしてバックグラウンドで実行し、完了通知のみをユーザーに送るワークフローの場合
- 複数エージェントのオーケストレーションで、下流エージェントが上流の完全な出力を入力として受け取る場合
- レスポンス全体のトークン数を事前に把握してコスト計算・ログ記録に使う必要がある場合

## ハイブリッド（併用）

- **フロントはストリーミング、バックはバッチ**: ユーザー向け UI にはストリーミングで表示しつつ、全文完成後にバッファした結果を後続 API に渡します。最も一般的な併用パターンです。
- **構造化ヘッダ＋ストリーミング本文**: メタデータ（ステータスコード、分類ラベル等）はバッチで先行送出し、本文部分をストリーミングします。
- **段階的バッチ**: 長いレポート生成では、セクション単位でバッチ出力し、各セクション完成時に部分表示します。Jira のチケット要約を複数件処理する場合などに有効です。

## 判断フロー

1. **出力の消費者は誰か？** — 人間が直接読む → ストリーミング優先。システムが消費する → バッチ優先です。
2. **出力フォーマットに構造制約があるか？** — JSON/XML 等の構造化出力 → バッチを選びましょう。自然言語テキスト → ストリーミングが可能です。
3. **全文に対する検証が必須か？** — コンプライアンス検証やガードレールが全文を要求する → バッチまたはハイブリッドを選びましょう。
4. **応答時間の期待値は？** — 数秒以内の体感を求める → ストリーミングを選びましょう。数十秒〜分単位で許容される → バッチで対応できます。
5. **インフラがストリーミングを支援するか？** — WebSocket / SSE 非対応の環境 → バッチに倒しましょう。

## よくある失敗

- **構造化出力をストリーミングしてパースエラー**: JSON 応答をストリーミングで流し、クライアント側で不完全な JSON をパースしようとしてエラーが頻発します。
- **バッチ出力で体感速度を無視**: 全文生成に 30 秒かかるレポートをバッチで返し、ユーザーが「フリーズした」と誤認して離脱します。
- **ストリーミング中のエラーハンドリング不備**: 途中でツール呼び出しが失敗した場合、既に送出済みのトークンを取り消せず、不完全な応答が残ります。
- **ガードレールのバイパス**: ストリーミングで逐次送出した結果、全文検証が実行される前にユーザーに不適切な内容が表示されてしまいます。
- **コスト追跡の困難**: ストリーミングモードでトークン消費量の集計が遅延し、コストガバナンスの閾値超過を検知できません。バッファリング層でのトークンカウントを併用しましょう。
- **再送・リトライの複雑化**: ストリーミング途中でネットワーク切断が発生した場合、どこから再送するかの管理が煩雑になります。チェックポイント機構がないと重複送出や欠損が起きます。

## 実装上の考慮点

- ストリーミングには SSE（Server-Sent Events）または WebSocket を使用します。SSE は HTTP/2 環境で導入が容易ですが、双方向通信が不要な場合に限定されます。
- バッチ出力で応答時間が長い場合は、プログレスバーや「処理中」インジケータを表示し、ユーザーの離脱を防ぎましょう。
- ハイブリッド構成では、ストリーミングバッファの最大サイズとタイムアウトを設定し、無限にバッファが膨らむことを防止してください。
