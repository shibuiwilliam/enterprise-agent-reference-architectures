# F06 · コンパイル時 vs ランタイムLLM

!!! abstract "一言で"
    エージェントの振る舞いを「事前に確定させる（コンパイル時）」か「実行時に LLM が
    動的に決定する（ランタイム）」かの選択は、タスクの頻度・形式の安定性・コスト・
    レイテンシ要件によって決まります。高頻度で形式が安定したタスクはコンパイル時に最適化し、
    一回性で変動が大きいタスクはランタイム LLM に委ねましょう。

## コンパイル時（A）を選ぶ条件

- タスクの入出力形式が安定しており、パターンが固定的（毎日同じ形式の日次レポート生成）。
- 実行頻度が高く（1日数百〜数千回）、LLM 呼び出しコストの削減が重要。
- レイテンシ要件が厳しく、LLM の推論時間（数百ミリ秒〜数秒）を排除したい。
- Salesforce の商談ステージ変更通知を Slack に定型フォーマットで投稿するなど、
  テンプレート化が可能な処理。
- コード生成・テンプレートエンジン・ルールエンジンで実装できます。
- 処理結果の再現性・テスト可能性が重要（同一入力 → 同一出力を保証したい）。

## ランタイムLLM（B）を選ぶ条件

- タスクの内容が毎回異なり、事前にパターンを網羅できない（自由記述の問い合わせ対応）。
- 自然言語の理解・生成が本質的に必要（要約、分類、対話、推論）。
- 実行頻度が低く（1日数十回以下）、LLM 呼び出しのコストが許容範囲。
- Zendesk の顧客問い合わせへの回答生成、Jira チケットの自動分類と優先度判定など。
- 入力の多様性が高く、ルールベースでの分岐が爆発します。
- 新しいパターンへの適応が必要で、コード変更なしに対応したい。

## ハイブリッド（併用）

多くの本番システムでは、コンパイル時とランタイムを組み合わせたパイプラインを構築します。

- **LLM で分類 → コンパイル済みフローで実行**: ランタイム LLM がユーザーの意図を
  分類し、分類結果に応じてコンパイル済みのワークフロー（テンプレート・ルールエンジン）
  を実行します。意図分類の柔軟性と実行の効率性を両立できます。
- **コンパイル済みパイプライン＋LLM フォールバック**: 定型パターンはコンパイル済みの
  ロジックで高速処理し、パターンに該当しない入力だけ LLM にフォールバックします。
- **段階的コンパイル化**: 初期はランタイム LLM で運用し、頻出パターンをログから抽出して
  順次コンパイル済みフローに移行します。P40（フィードバックループ）と組み合わせましょう。

## 判断フロー

1. **入出力の形式は安定しているか？** — 安定していればコンパイル時を選びましょう。変動が大きければ
   ランタイムを選びます。
2. **実行頻度は？** — 1日数百回以上ならコンパイル時でのコスト最適化を検討してください。
   数十回以下ならランタイム LLM のコストは許容範囲内の場合が多いです。
3. **自然言語の理解・生成が本質的に必要か？** — 必要ならランタイム LLM を選びましょう。
   構造化データの変換だけならコンパイル時で対応できます。
4. **再現性は重要か？** — 同一入力に対して同一出力を保証したいならコンパイル時を選びましょう。
   LLM は temperature > 0 では非決定的である点に注意してください。
5. **新パターンへの適応速度は？** — コード変更なしに対応したいならランタイム LLM が適しています。
   デプロイサイクルに数日かかる環境ではランタイムの柔軟性が有利です。

## よくある失敗

- **すべてをランタイム LLM に頼る** — 定型レポート生成を毎回 LLM に投げ、月間コストが
  数万ドルに膨らみます。テンプレートエンジンで 1/100 のコストで済むタスクを識別しましょう。
- **すべてをコンパイル時に固定する** — 分岐条件が爆発し、メンテナンス不能なルール
  エンジンが出来上がります。ルール数が 100 を超えたら LLM への移行を検討してください。
- **コンパイル化のタイミングが早すぎる** — パターンが安定する前にコンパイル化し、
  頻繁なコード変更が必要になります。まずランタイムで運用し、パターンの安定を確認してから
  コンパイル化しましょう。
- **ハイブリッドの分岐条件が不明確** — どのリクエストがコンパイル済みフローに行き、
  どれが LLM に行くかの判定ロジックが属人化します。分類基準をコードで明文化してください。
