# F20 · 強整合 vs 結果整合（メモリ）

!!! abstract "一言で"
    エージェントが参照・更新するメモリ（会話履歴、ユーザープロファイル、共有知識）の
    整合性モデルを、強整合（即時反映）と結果整合（遅延許容）のどちらにするかの選択。
    同一セッション内の一貫性を取るか、複数エージェント・複数リージョンにまたがる
    可用性とスループットを取るかがトレードオフになる。

## 強整合を選ぶ条件

- 同一ユーザーの同一セッション内で、直前の操作結果が即座に次のステップに反映される必要がある場合（Salesforce の商談更新直後にその値を参照する、Workday で申請した内容を即座に確認するなど）
- 複数エージェントが同一レコードを同時に更新する可能性があり、競合による不整合が業務に直接影響する場合
- 金銭・在庫・契約ステータスなど、不整合が即座にビジネス損失につながるデータを扱う場合
- 規制要件により、データの「最新の正しい状態」を常に保証する必要がある場合（金融取引の残高、医療記録の最新バイタルなど）
- エージェントが単一リージョンで稼働し、レイテンシの制約が緩い場合

## 結果整合を選ぶ条件

- 複数リージョンにまたがるグローバル展開で、リージョン間のレイテンシ（数十〜数百ms）が強整合のボトルネックになる場合
- エージェントが参照するメモリが「過去の要約」「ユーザーの嗜好」「知識ベースの更新」など、数秒〜数分の遅延が許容されるデータである場合
- 高可用性が最優先で、一時的なネットワーク分断時にもエージェントが応答を継続する必要がある場合
- 大量のエージェントが並行稼働し、全メモリ更新を同期的に処理するとスループットが著しく低下する場合
- Slack の会話履歴集約や Box のドキュメントインデックスなど、バッチ的に収集・統合する性質のデータの場合

## ハイブリッド（併用）

- **セッション内は強整合、セッション間は結果整合**: 同一セッション内の会話履歴とツール呼び出し結果は強整合で管理し、長期メモリ（ユーザープロファイル、過去セッションの要約）は結果整合で非同期に更新する。最も一般的な設計。
- **データ分類による使い分け**: 金銭・契約・在庫などのトランザクションデータは強整合、ナレッジベース・嗜好データ・分析用集約データは結果整合。データの重要度で整合性レベルを分ける。
- **Read-Your-Own-Writes**: 書き込みを行ったエージェント自身は即座に最新値を読めるが、他のエージェントには結果整合で伝播する。単一ユーザーの体験を損なわずにスケーラビリティを確保する。

## 判断フロー

1. **対象データの不整合による影響は？** — 金銭損失・規制違反 → 強整合。体験の軽微な劣化 → 結果整合を検討。
2. **更新と参照は同一セッション内で発生するか？** — YES → 強整合（少なくとも Read-Your-Own-Writes）。NO → 結果整合で十分な場合が多い。
3. **地理的分散の要件は？** — マルチリージョン → 結果整合が現実的（強整合はリージョン間レイテンシが大きい）。単一リージョン → 強整合のコストが低い。
4. **並行エージェント数は？** — 同一データへの同時アクセスが多い → 強整合はロック競合のリスク。少ない → 強整合でも問題なし。
5. **データの鮮度要件は？** — 秒単位 → 強整合。分〜時間単位 → 結果整合。

## よくある失敗

- **全データを強整合にしてスケールしない**: 長期メモリや知識ベースまで強整合で管理し、データベースのロック競合とレイテンシ増大でエージェントの応答速度が劣化する。
- **結果整合の遅延を考慮しない設計**: エージェント A が Salesforce の商談を更新した直後に、エージェント B がその商談を参照するフローで結果整合を採用し、古い値で処理を進めてしまう。
- **Stale Read の検知漏れ**: 結果整合環境でメモリの鮮度をチェックせず、数時間前の古い情報に基づいて顧客に回答してしまう。タイムスタンプや世代番号によるチェックが必要。
- **障害時のフォールバック未設計**: 強整合を前提とした設計で、DB 障害時にエージェントが完全停止する。キャッシュからの読み取り（結果整合へのフォールバック）を用意しておく。
- **整合性モデルの混在による混乱**: 同一データに対して強整合と結果整合が混在し、開発者がどのパスで読んでいるか意識せずにバグを埋め込む。データ分類ごとの整合性レベルをドキュメント化する。

## 実装上の考慮点

- 強整合が必要なメモリ層にはトランザクション対応のデータストア（PostgreSQL, DynamoDB with Strong Consistency 等）を使用する。結果整合層には Redis のレプリケーションや DynamoDB の Eventually Consistent Read が適する。
- 結果整合の遅延許容時間を SLA として定義する。「最大 5 秒以内に全レプリカに反映」のように具体的な数値を設定し、モニタリングでレプリケーション遅延を追跡する。
- マルチエージェント環境では、共有メモリへの書き込みに楽観的ロック（バージョン番号）を採用し、競合発生時のリトライ戦略を定めておく。
