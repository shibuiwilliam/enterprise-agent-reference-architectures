# F05 · 自律 vs 決定論ワークフロー

!!! abstract "一言で"
    エージェントに「自由にステップを計画・実行させる（自律）」か「事前定義されたフローに
    従わせる（決定論ワークフロー）」かは、副作用の可逆性・影響度・規制要件によって決まる。
    自律は柔軟だが予測不能、決定論は硬直だが監査可能である。多くの本番システムでは
    両者をハイブリッドで組み合わせる。

## 自律（A）を選ぶ条件

- エージェントの操作による副作用が軽微であり、誤りがあっても容易に可逆（取り消し・修正可能）。
- タスクのバリエーションが多く、事前にすべてのフローを定義するのが非現実的。
- 情報検索・要約・分析など、読み取り専用の操作が中心。
- Notion でのナレッジ検索、Slack での情報要約、Jira チケットの分類提案など。
- タスクの頻度が低く、個別対応のコストが許容される。
- ユーザーが結果を確認してから次のステップを判断する対話的なワークフロー。

## 決定論ワークフロー（B）を選ぶ条件

- 操作の副作用が不可逆または高影響（決済処理、契約変更、本番デプロイ、人事異動）。
- 規制・コンプライアンス要件により、実行ステップの事前定義と監査証跡が必須。
- Workday での給与変更、Salesforce での契約ステージ変更、Shopify での一括価格変更など。
- 同一パターンのタスクが高頻度で繰り返され、フロー最適化の ROI が高い。
- 障害時の再試行・補償トランザクションを確実に実行する必要がある。
- SOX / GDPR / 金融規制の対象となる業務プロセス。

## ハイブリッド（併用）

本番環境では純粋な自律も純粋な決定論も少なく、ハイブリッドが主流である。

- **自律で計画、決定論で実行**: エージェントが自律的にタスクを分解・計画し、
  各ステップの実行は事前定義されたワークフロー（API 呼び出しの順序・バリデーション・
  承認ゲート）に従う。計画の柔軟性と実行の安全性を両立する。
- **リスクレベルによる切り替え**: 読み取り操作は自律、書き込み操作は決定論ワークフローに
  自動的にルーティングする。P20（二重トラック検証）と組み合わせる。
- **ServiceNow のインシデント対応**: 原因調査は自律（ログ検索・仮説生成・検証を柔軟に
  実行）、復旧操作は決定論（定義済みのランブック手順に従う）。

## 判断フロー

1. **副作用はあるか？** — 読み取り専用なら自律で問題ない。
2. **副作用は可逆か？** — 可逆なら自律＋事後検証。不可逆なら決定論＋事前承認。
3. **規制要件はあるか？** — SOX / GDPR / 金融規制の対象なら、決定論ワークフローで
   監査証跡を確保する。人間のレビューが必須。
4. **タスクの多様性は？** — パターンが固定的なら決定論で最適化。多様で予測不能なら
   自律で柔軟に対応。
5. **HITL（Human-in-the-Loop）で補完できるか？** — 自律でも承認ゲート（P18）を
   高リスク操作の前に挿入することで安全性を確保できる。

## よくある失敗

- **不可逆な操作を自律に任せる** — Shopify で数千商品の価格を一括変更する操作を
  エージェントの自律判断に委ね、誤った価格設定で損害が発生する。
- **すべてを決定論にする** — 情報検索や要約まで固定フローにし、エージェントの柔軟性を
  完全に殺す。「AIエージェント」である意味がなくなる。
- **ハイブリッドの境界が曖昧** — 自律と決定論の切り替え条件が明文化されておらず、
  開発者によって実装がバラつく。操作のリスク分類を先に定義する。
- **監査証跡の欠如** — 自律モードでの操作ログが不十分で、問題発生時に「なぜその操作を
  したか」を追跡できない。全モードで操作ログを記録する。
