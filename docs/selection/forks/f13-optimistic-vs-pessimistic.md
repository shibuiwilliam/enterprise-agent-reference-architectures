# F13 · 楽観 vs 悲観ロック

!!! abstract "一言で"
    複数のエージェント（またはエージェントと人間）が同一リソースを同時に操作する場合の
    排他制御戦略です。競合の発生頻度とリトライコストによって、楽観ロック（衝突時に検知・リトライ）と
    悲観ロック（操作前にロック取得）を選び分けます。

## 楽観ロック（Optimistic）を選ぶ条件

- 同一リソースへの**競合が稀**（目安として衝突率 5% 未満）
- **スループットとレイテンシ**を優先し、ロック待ちによる遅延を避けたい
- リトライのコストが低い（べき等な操作、または計算量が軽い）
- Salesforce の商談メモを複数エージェントが更新するが、同時に同じ商談を扱うことは稀なケース
- Jira チケットのステータス更新で、ETag やバージョン番号による衝突検知が可能

## 悲観ロック（Pessimistic）を選ぶ条件

- 同一リソースに**複数エージェントが集中**し、競合が頻発する
- 操作の**リトライコストが高い**（長時間の推論チェーン、外部 API の副作用が伴う）
- **一貫性が最優先**で、中途半端な状態を許容できない
- Shopify の在庫数を複数のエージェントが同時に減算する場合（売り越し防止）
- ServiceNow の変更管理レコードを1つのエージェントが長時間にわたって編集する場合

## ハイブリッド（併用）

- **リソース種別で使い分け**: 参照系の多いリソース（ナレッジ記事、FAQ）は楽観ロック、書き込みが集中するリソース（在庫、残高）は悲観ロックを適用します
- **楽観から悲観へのエスカレーション**: 通常は楽観ロックで運用し、特定リソースの衝突率が閾値（目安として 10%）を超えた場合に自動的に悲観ロックに切り替えます
- **短期悲観ロック + タイムアウト**: ロック保持時間に上限（目安として 30 秒）を設け、デッドロックを防止します。タイムアウト後は自動解放し、後続のエージェントが処理を引き継ぎます

## 判断フロー

1. 対象リソースへの**同時アクセス頻度**を見積もりましょう → 競合率が目安として 5% 未満なら**楽観ロック**を選びます
2. 操作は**べき等**か → Yes なら楽観ロックが適しています（リトライが安全です）
3. 操作に**不可逆な副作用**があるか → Yes なら悲観ロック（または HITL 承認と併用）を選びましょう
4. リトライ時の**再計算コスト**は許容範囲か → No なら悲観ロックを採用してください
5. 悲観ロックを採用する場合、**デッドロック防止策**（タイムアウト、ロック順序の統一）を設計しましょう

## よくある失敗

- **楽観ロックでリトライ爆発**: 競合率を過小評価し、繰り返しリトライが発生してシステム負荷が急増します。特に、複数エージェントが同じ Salesforce レコードを更新するバッチ処理で頻発します
- **悲観ロックのデッドロック**: エージェント A がリソース X をロック→リソース Y を要求、エージェント B がリソース Y をロック→リソース X を要求、という循環待ちが発生します
- **ロックの粒度ミス**: テーブル全体をロックしてしまい、無関係なレコードの処理までブロックされます。ロック粒度はレコード単位を基本としましょう
- **ロック解放漏れ**: エージェントの推論が異常終了した際にロックが解放されず、後続の処理が永久にブロックされます。TTL 付きロック（Redis の `SET ... EX`）やリース方式で対策してください
- **エージェント固有の問題**: LLM の推論時間が不定のため、ロック保持時間の見積もりが難しくなります。推論時間の P99 を計測し、TTL の起点としましょう
