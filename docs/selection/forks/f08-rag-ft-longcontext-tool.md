# F08 · RAG / FT / 長文脈 / ツール

!!! abstract "一言で"
    エージェントに知識を与える手段は RAG・ファインチューニング（FT）・長文脈ウィンドウ・ツール呼び出しの
    4つがあり、**排他ではなく組み合わせて使う**のが前提です。データの鮮度・量・形式安定性・
    真実性要件によって主軸を選び、残りを補助に回しましょう。

## RAG を主軸にする条件

- ナレッジが日次〜週次で更新され、**鮮度が価値の中核**（例: Zendesk のチケット履歴、Confluence の社内 Wiki）
- 対象ドキュメントが数千〜数十万件規模で、トークンウィンドウに収まらない
- 検索精度をチャンク設計・リランカーで制御できるチームがいる
- Salesforce のナレッジベースから最新の製品 FAQ を引いて回答するカスタマーサポートエージェント

## FT を主軸にする条件

- 出力の**形式・トーン・ドメイン語彙**を安定させたい（例: 法務レビューの定型コメント生成）
- 入力パターンが限定的で、大量の教師データを用意できる
- 推論時にコンテキストを付与するコストを削減したい
- Workday の人事評価コメントを社内ガイドラインに沿った文体で生成するエージェント

## 長文脈を主軸にする条件

- 対象が**1〜数件の長文ドキュメント**で、全文を一括して渡すほうが検索ミスを回避できる
- 契約書レビュー・議事録要約など、文書内の相互参照が多いタスク
- チャンク分割すると意味が壊れるリスクが高い
- Box に保存された契約書 PDF（目安として 50 ページ以下）を丸ごと読み込んで条項を比較するエージェント

## ツール呼び出しを主軸にする条件

- **真実性が最優先**で、最新の正確な値を返す必要がある（例: 在庫数、為替レート、注文ステータス）
- データソースが API を持ち、構造化された応答を返す
- Shopify の注文 API を叩いて配送状況を正確に回答するエージェント

## 組み合わせの典型パターン

- **RAG + ツール**: 社内ナレッジで回答を下書きし、最終的な数値は API で検証します
- **FT + RAG**: ドメイン特化の語彙・形式を FT で学習し、最新情報は RAG で補完します
- **長文脈 + ツール**: 契約書全文を長文脈で処理し、関連判例は検索ツールで取得します
- 単一手法で全要件を満たそうとせず、**知識供給チェーンとして設計する**ことが重要です

## 判断フロー

1. データが API で取得でき、正確性が最優先か → **ツール**を主軸に据えましょう
2. 対象が少数の長文ドキュメントか → **長文脈**で丸ごと投入を検討してください
3. 出力形式・トーンの安定が最重要か → **FT** を検討しましょう（教師データの確保が前提です）
4. 大量かつ頻繁に更新されるドキュメント群か → **RAG** を主軸にしましょう
5. 上記の複数に該当する場合 → 主軸を1つ決め、残りを補助として組み合わせてください

## よくある失敗

- **RAG 一本足**: API で正確に取れるデータまで RAG 経由にし、古いチャンクを返してしまいます
- **FT 過信**: 学習データに含まれない新製品情報をハルシネーションで回答してしまいます
- **長文脈の乱用**: 100 件のドキュメントを無理にプロンプトに詰め込み、コストが爆発します
- **ツール偏重**: すべての質問を API 呼び出しに変換しようとし、自然言語の柔軟性を失います
- 手段の選定理由を記録せず、後からなぜ RAG を選んだのか分からなくなります
