# F19 · HITL vs HOTL vs 全自動

!!! abstract "一言で"
    エージェントの意思決定・実行に対する人間の関与度を、HITL（Human-in-the-Loop: 都度承認）、
    HOTL（Human-on-the-Loop: 監視＋例外介入）、全自動（人間介入なし）の三段階から選択します。
    リスクの大きさとエージェントの実績に応じて段階的に自動化レベルを引き上げるのが基本戦略です。

## HITL（都度承認）を選ぶ条件

- 操作が不可逆で影響が大きい場合（Workday での給与変更、Salesforce での大型商談のステージ変更、本番 DB への書き込みなど）
- エージェントの導入初期で、出力品質と判断精度の実績データがまだ蓄積されていない場合
- 規制要件により「人間が承認した」というエビデンスが必須の場合（金融取引、医療判断、法的文書の発行など）
- 副作用が外部顧客に直接影響する場合（顧客向けメール送信、契約書の自動生成、請求処理など）
- 1 日あたりの対象件数が少量（目安として 50 件以下）で、人手承認が運用上ボトルネックにならない場合

## HOTL（監視＋例外介入）を選ぶ条件

- エージェントに一定の実績があり、過去の承認率が高い（目安として 95% 以上が承認されている）場合
- 操作のリスクが中程度で、一定の条件下では自動実行しても許容できる場合（Jira チケットの優先度変更、Slack への社内通知、ServiceNow のインシデント分類など）
- 処理件数が多く（目安として日次 100 件超）、全件の都度承認が運用負荷として持続不可能な場合
- 異常検知の仕組み（信頼度スコア、異常値アラート）で逸脱をリアルタイムに検知できる場合
- ロールバック手段が確保されており、誤操作を事後的に修正できる場合

## 全自動を選ぶ条件

- 操作が可逆かつ影響範囲が限定的な場合（社内ナレッジの検索と要約、ログの自動分類、ダッシュボード更新など）
- エージェントの精度が十分に検証され、エラー率が目安として 1% 未満で安定している場合
- 即時性が求められ、人間の介入が SLA 違反につながる場合（リアルタイムのカスタマーサポート初期応答、アラートのトリアージなど）
- 出力が「情報提供のみ」で、最終的な意思決定や実行は別の人間またはシステムが担う場合

## ハイブリッド（併用）

- **リスクベースの動的切り替え**: 操作の種類・金額・影響範囲に応じて自動的にモードを切り替えます。例: 10 万円未満の経費承認は全自動、10〜100 万円は HOTL、100 万円超は HITL とします。
- **段階的昇格**: 導入初期は全操作 HITL → 2 週間の実績で承認率 95% 超の操作タイプを HOTL に昇格 → さらに 1 か月の実績で HOTL から全自動に昇格します。昇格基準は事前に定義しましょう。
- **信頼度スコア連動**: エージェントの出力に信頼度スコアを付与し、高信頼度（目安として 0.9 超）は全自動、中信頼度（0.7〜0.9）は HOTL、低信頼度（0.7 未満）は HITL にルーティングします。

## 判断フロー

1. **操作は可逆か？** — 不可逆 → HITL を基本としましょう。可逆 → 次のステップへ進みましょう。
2. **影響範囲は？** — 外部顧客・金銭・法的効力に直結 → HITL を選びましょう。社内のみ → HOTL 以上を検討できます。
3. **エージェントの実績は十分か？** — 導入初期・実績不足 → HITL から開始しましょう。承認率 95% 超の実績 → HOTL への昇格を検討してください。
4. **処理件数は人手で対応可能か？** — 日次 50 件以下 → HITL が現実的です。100 件超 → HOTL または条件付き全自動を検討しましょう。
5. **異常検知とロールバックの仕組みがあるか？** — YES → HOTL/全自動が可能です。NO → HITL に留めましょう。

## よくある失敗

- **承認疲れ（Approval Fatigue）**: 全操作を HITL にした結果、承認者が形骸的に全件承認し、ゲートとして機能しなくなります。HITL の対象は本当に人間の判断が必要な操作に絞りましょう。
- **昇格基準の未定義**: 「いつ HITL から HOTL に移行するか」を決めずに運用を開始し、いつまでも HITL のままで自動化の恩恵を得られません。
- **全自動への早すぎる移行**: 十分な実績データなしに全自動化し、エッジケースでの誤操作が顧客影響を引き起こします。特に顧客向けの Zendesk 応答や Shopify の注文処理で問題になりやすいです。
- **HOTL の監視体制不備**: HOTL モードにしたものの、異常アラートの閾値設定が不適切で、検知が遅れて対応が間に合いません。
- **降格ルールの未定義**: 全自動に昇格した後に精度が劣化した場合の降格（全自動 → HOTL → HITL への巻き戻し）ルールがなく、問題が長期化します。

## 段階的昇格の運用ガイドライン

- **昇格の前提条件を数値で定義する**: 「承認率 95% 以上を 2 週間継続」「重大エラー 0 件」など、曖昧さのない基準を設定してください。
- **降格のトリガーも事前に決める**: 「重大エラーが 1 件でも発生したら即座に HITL に戻す」「承認率が 90% を下回ったら HOTL に降格」など、明確に定めましょう。
- **昇格履歴を記録する**: どの操作タイプがいつ昇格し、その根拠となったデータを監査証跡として残してください。規制対応や社内説明に必要になります。
