# F10 · ツール呼び出し vs コード生成

!!! abstract "一言で"
    エージェントが外部システムと相互作用する手段として、事前定義されたツール（関数）を
    呼び出すか、LLM がコードを動的に生成して実行するかという設計分岐です。安全性と柔軟性の
    トレードオフであり、ワークロードのリスクプロファイルに応じて使い分ける必要があります。

## ツール呼び出しを選ぶ条件

- 操作が**事前に定義・列挙**でき、入出力スキーマが安定している
- **副作用が不可逆**または影響範囲が大きい操作（例: Salesforce の商談ステージ更新、Shopify の注文キャンセル）
- セキュリティポリシーとして、実行可能な操作のホワイトリスト管理が求められる
- 監査証跡として「どのツールをどのパラメータで呼んだか」を構造化ログに残す必要がある
- ServiceNow のインシデント更新 API を定義済みツールとして公開し、エージェントにパラメータだけを決定させる

## コード生成を選ぶ条件

- タスクが**事前に列挙しきれない計算・データ加工**を含む（例: CSV の集計、正規表現による抽出、グラフ生成）
- ツール定義の組み合わせでは表現しきれない柔軟なロジックが必要
- **サンドボックス環境**で安全に実行でき、ホストシステムへの影響を遮断できる
- Workday から取得した人事データを pandas で集計し、部門別離職率のレポートを生成するエージェント
- Jira のチケットデータを集計して、スプリントごとのバーンダウンチャートを動的に描画する

## ハイブリッド（併用）

- **ツールで取得、コードで加工**: API 呼び出しはツール経由で安全に実行し、取得データの変換・集計はサンドボックス内のコード生成で処理します
- **コード生成の出力をツールで適用**: 計算結果を算出するコードを生成・実行し、最終的な書き込みは定義済みツール経由で行うことで副作用を制御します
- 実務では「読み取り・計算はコード生成、書き込みはツール」という分担が多いです

## 判断フロー

1. 操作に**不可逆な副作用**があるか → Yes なら**ツール呼び出し**を必須としましょう（人間のレビューも検討してください）
2. タスクが事前定義可能な操作の組み合わせで完結するか → Yes なら**ツール呼び出し**のみで構成しましょう
3. 動的な計算・データ加工が必要か → Yes なら**コード生成**を検討してください
4. コード生成を採用する場合、**サンドボックス**（コンテナ、gVisor、Firecracker 等）を確保できるか → No なら採用しないでください
5. サンドボックスの実行時間・メモリ・ネットワークの制限値を設定しましょう

## よくある失敗

- **コード生成で直接 API を叩かせる**: サンドボックスの外部通信を許可してしまい、認証情報の漏洩やレート制限の迂回が発生します
- **ツール定義の肥大化**: あらゆる計算パターンをツールとして定義しようとし、ツール数が目安として 100 を超えて LLM の選択精度が劣化します
- **サンドボックスの制約不足**: CPU・メモリ・実行時間の上限を設けず、無限ループや大量メモリ確保でホストに影響を与えてしまいます
- **コード生成の検証省略**: 生成されたコードを実行前にレビューせず、意図しないファイル操作やデータ破壊が発生します
- ツールとコード生成の境界が曖昧で、同じ操作が両方の経路から実行可能になり、監査が困難になります
