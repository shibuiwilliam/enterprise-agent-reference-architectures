# F07 · 事前 vs 事後検証

!!! abstract "一言で"
    エージェントの出力や操作を「実行前に検証する（事前）」か「実行後に検証する（事後）」かは、
    副作用の可逆性と影響度によって決まる。不可逆な操作は事前検証が必須であり、
    情報提供のみの操作は事後検証で十分な場合が多い。この分岐を誤ると、安全性か速度の
    どちらかを不必要に犠牲にする。

## 事前検証（A）を選ぶ条件

- エージェントの操作に不可逆な副作用がある（決済実行、契約締結、データ削除、本番デプロイ）。
- 誤操作の影響が金銭的損害・法的責任・ブランド毀損に直結する。
- Workday での給与変更、Salesforce での契約金額変更、Shopify での一括価格改定など。
- 規制・コンプライアンス要件により、実行前の承認プロセスが義務付けられている。
- エージェントの出力が顧客に直接提示され、誤情報が許容されない（医療・金融・法律助言）。
- 操作の頻度が低く、事前検証のレイテンシ（人間の承認待ち含む）が許容できる。

## 事後検証（B）を選ぶ条件

- エージェントの出力が情報提供のみであり、副作用がない（検索結果の表示、要約の提示）。
- 操作の副作用が可逆であり、問題があれば取り消しや修正が容易。
- レイテンシ要件が厳しく、事前検証による遅延が体験を著しく損なう。
- Slack での情報要約、Notion でのドキュメント下書き生成、Jira チケットの分類提案など。
- 高頻度の操作で、すべてに事前検証を挟むとスループットが大幅に低下する。
- 事後のサンプリング検査や品質モニタリングで十分な品質を維持できる。

## ハイブリッド（併用）

実務では事前と事後を組み合わせた二重トラック構成が標準的である。

- **リスクレベルによる振り分け**: 操作のリスクを事前に分類し、高リスク操作（不可逆・
  高影響）は事前検証トラック、低リスク操作（情報提供・可逆）は事後検証トラックに
  自動ルーティングする。P20（二重トラック検証）の直接的な実装となる。
- **段階的緩和**: 初期はすべての操作を事前検証し、信頼度が蓄積されたカテゴリから
  段階的に事後検証に移行する。P26（評価ゲート/カナリア）で品質を計測しながら
  閾値を調整する。
- **Zendesk の顧客対応**: 回答ドラフトの生成は事後検証（サンプリングレビュー）、
  返金処理の実行は事前検証（人間の承認必須）とする。

## 判断フロー

1. **操作に副作用はあるか？** — 副作用なし（情報提供のみ）なら事後検証で十分。
2. **副作用は可逆か？** — 可逆なら事後検証＋自動ロールバック。不可逆なら事前検証。
3. **影響範囲はどの程度か？** — 影響が1ユーザーに限定されるか、全顧客に波及するか。
   広範囲に影響する場合は事前検証を強化する。
4. **規制要件で事前承認が必須か？** — 金融取引・人事変更・医療関連は規制により
   事前検証が義務付けられる場合がある。人間のレビューが必須。
5. **信頼度スコアを活用できるか？** — エージェントの出力に信頼度スコアを付与し、
   閾値未満は事前検証、閾値以上は事後検証にルーティングする（P21）。

## よくある失敗

- **不可逆な操作を事後検証にする** — Shopify で商品を一括削除した後に検証し、
  復旧不能な事態に陥る。不可逆操作は例外なく事前検証。
- **すべてを事前検証にする** — 情報検索の結果表示にまで人間の承認を求め、
  エージェントの応答速度が数分に低下する。ユーザーが使わなくなる。
- **事後検証の仕組みがない** — 事後検証トラックに振り分けたものの、実際にはレビューが
  行われず、品質劣化に気づかない。サンプリング率と品質指標の閾値を定義する。
- **リスク分類が更新されない** — 初期のリスク分類のまま運用し、新たに追加された操作の
  リスクレベルが未定義のままデフォルト（事後検証）で動作する。
  新規ツール追加時にリスク分類を必須プロセスとする。
- **信頼度スコアの閾値が不適切** — 閾値が低すぎて危険な操作が事後検証に流れるか、
  高すぎてほぼすべてが事前検証になる。初期値を保守的に設定し、実測で調整する。
