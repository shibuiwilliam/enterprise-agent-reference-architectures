# F11 · オンライン vs オフライン知性

!!! abstract "一言で"
    エージェントの推論処理をリクエスト時にリアルタイムで行う（オンライン）か、
    事前にバッチで計算しておき結果を参照する（オフライン）かという設計分岐です。
    レイテンシ・コスト・精度のバランスに直結し、多くの本番システムでは両者を併用します。

## オンライン（リアルタイム推論）を選ぶ条件

- ユーザーの入力が**事前に予測できず**、都度 LLM の推論が必要（例: Slack Bot への自由文質問）
- **即時性**が価値の中核で、数秒以内の応答が求められる対話型ユースケース
- コンテキストがリクエストごとに異なり、キャッシュが効きにくい
- Zendesk のライブチャットで顧客の問い合わせにリアルタイムで回答するエージェント
- Salesforce の商談画面で営業担当が次のアクションを相談するエージェント

## オフライン（事前計算・バッチ処理）を選ぶ条件

- 処理対象が**大量かつ定期的**で、1件ずつリアルタイム処理するとコストが膨大になる
- 推論に**重い処理**（複数 LLM の連鎖、大規模 RAG 検索、外部 API の多段呼び出し）が含まれる
- 結果の鮮度が数時間〜1日程度の遅延を許容できる
- Workday の全従業員スキルデータを夜間バッチで分析し、翌朝にスキルギャップレポートを生成する
- Jira の全チケットを週次でクラスタリングし、重複チケットの候補リストを作成する
- Shopify の全商品説明を一括でSEO 最適化する月次バッチ

## ハイブリッド（併用）

- **オフラインで事前計算、オンラインで仕上げ**: 夜間バッチで商品レコメンデーションの候補を生成しておき、ユーザーのリアルタイムの行動に応じてオンラインで最終的なランキングを調整します
- **オンラインの結果をオフラインで集約**: リアルタイムの顧客対応ログをオフラインバッチで分析し、FAQ の自動更新や応答テンプレートの改善に活用します
- **キャッシュ層の活用**: 頻出クエリの応答をセマンティックキャッシュに蓄積し、類似クエリにはキャッシュから返すことでオンラインのコストを抑制します

## 判断フロー

1. 応答が**数秒以内**に必要か → Yes ならオンラインが必須です
2. 処理対象が**数百件以上のバッチ**か → Yes ならオフラインを検討しましょう
3. 1件あたりの推論コストが**高い**か（目安として $0.05 以上/件） → Yes ならオフラインでまとめて処理するほうがコスト効率が良いです
4. 結果の鮮度に**数時間の遅延**が許容されるか → Yes ならオフラインが候補になります
5. 両方の特性が必要な場合 → オフラインで事前計算し、オンラインで差分補正するハイブリッド構成を取りましょう

## よくある失敗

- **全処理をオンライン化**: 大量データの分析まで同期で処理し、レイテンシが目安として 30 秒を超えてユーザーが離脱します。API のタイムアウトにも抵触します
- **オフラインの鮮度管理不足**: バッチの実行頻度とデータの更新頻度が合っておらず、古い結果を返し続けます。商品の在庫状況が1日遅れで表示されるなどの問題が起きます
- **コスト見積もりの欠如**: オンラインで全リクエストを LLM に通した場合の月次コストを事前に試算せず、本番投入後に予算超過が判明します
- **オフライン結果の無効化戦略がない**: マスタデータが更新されたときにオフラインの計算結果を無効化する仕組みがなく、整合性が崩れます
- オフラインバッチのエラーハンドリングが甘く、途中で失敗した場合に部分的な結果だけが反映されてしまいます
