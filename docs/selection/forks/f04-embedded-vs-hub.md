# F04 · 埋め込み vs ハブ

!!! abstract "一言で"
    エージェントを「既存の業務システムに埋め込む」か「独立した統合ハブとして構築する」かは、
    ユーザーの業務動線と横断性によって決まります。単一システム内で完結する業務には埋め込みが
    自然ですが、複数システムをまたぐ業務にはハブが有効です。実務では両方を併用し、
    入口を使い分けるのが現実的な解となります。

## 埋め込み（A）を選ぶ条件

- ユーザーが特定の SaaS（Salesforce / ServiceNow / Jira など）に長時間滞在し、
  その画面を離れずに AI 支援を受けたい。
- エージェントが操作するデータとツールがそのシステム内に閉じている。
- 既存の UI / UX フローにエージェントを自然に統合できる（サイドパネル、コマンドパレット）。
- Salesforce の商談画面で「この案件の勝率分析をして」と依頼するケース。
- Zendesk のチケット画面で顧客対応の下書きを生成するケース。

## ハブ（B）を選ぶ条件

- 業務が複数のシステムを横断する（Salesforce の商談 → Jira のチケット → Slack への通知）。
- ユーザーの入口を一本化したい（「何でも聞けるポータル」「社内 AI アシスタント」）。
- エージェントが 3 つ以上の SaaS に対してツール呼び出しを行う必要がある。
- IT ヘルプデスクのように、問い合わせの種類に応じて裏側のシステムが変わるが、
  ユーザーは意識せず統一インタフェースで操作したい。
- Slack / Teams をフロントエンドとし、バックエンドで複数 SaaS を統合するケース。

## ハイブリッド（併用）

実務では埋め込みとハブを併用するのが最も一般的なパターンです。

- **Slack をハブ、各 SaaS を埋め込みとして構成**: Slack 上の統合ボットから横断的な
  タスクを実行しつつ、Salesforce 画面内のサイドパネルでは商談固有の分析を行います。
- **ハブから埋め込みへのディープリンク**: ハブでの調査結果が「Jira でチケットを作成」
  という操作に至った場合、Jira 画面に遷移してそこの埋め込みエージェントに引き継ぎます。
- **共通バックエンドの共有**: 埋め込みもハブも同一のオーケストレーション基盤・
  モデルゲートウェイ（P03）を共有し、フロントエンドの差分だけを実装します。

## 判断フロー

1. **ユーザーの主な業務動線は？** — 1つのシステムに 1 日の大半を費やすなら埋め込みが優先です。
   複数システムを行き来するなら、ハブで入口を統一する価値が高いです。
2. **操作対象のデータは単一システムに閉じるか？** — 閉じるなら埋め込み、横断するならハブを選びましょう。
3. **ユーザーはシステム切り替えのコストを許容できるか？** — コンテキストスイッチが
   頻繁に発生するなら、ハブで吸収しましょう。
4. **既存 SaaS のエージェント統合 API はあるか？** — Salesforce Einstein、ServiceNow
   Virtual Agent など、既存の埋め込み基盤があるなら活用を検討してください。
5. **両方必要か？** — 多くの場合 Yes です。その場合、共通バックエンドを設計し、
   フロントエンドの差分だけを実装する方針としましょう。

## よくある失敗

- **ハブだけで構築し、埋め込みを無視する** — ユーザーが業務システムとハブを行き来し、
  コンテキストスイッチのコストが生産性を相殺します。特に Salesforce や ServiceNow で
  長時間作業するユーザーにはフラストレーションが大きくなります。
- **埋め込みだけで構築し、横断タスクに対応できない** — 各システム内のエージェントが
  孤立し、「商談状況を Slack に要約して通知」のような横断操作ができません。
- **バックエンドを二重に構築する** — 埋め込みとハブで別々のオーケストレーション基盤を
  構築してしまい、保守コストが倍増します。共通基盤を設計してフロントを分けましょう。
- **ハブのルーティング精度が低い** — 「何でも聞ける」ハブで意図分類の精度が低く、
  誤った専門エージェントに振り分けてしまいます。初期は対応範囲を絞りましょう。
