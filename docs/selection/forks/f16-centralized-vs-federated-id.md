# F16 · 集中IDブローカー vs 連合認証

!!! abstract "一言で"
    エージェントが複数 SaaS にアクセスする際の認証・認可を、単一の ID ブローカーに
    集約するか、各システムの既存認証基盤を連合的に活用するかの選択です。
    権限の一貫性と統制力を取るか、各システムの自律性と既存投資の活用を取るかが分かれ目になります。

## 集中IDブローカーを選ぶ条件

- エージェントが横断的に 5 システム以上（Salesforce, ServiceNow, Workday, Box, Slack 等）にアクセスし、統一的な権限ポリシーが求められる場合
- 「誰が何をエージェントに委任したか」の監査証跡を一元的に管理する必要がある場合
- Okta や Azure AD を既に全社 IdP として導入しており、エージェント用の OAuth 2.0 Token Exchange を集約できる基盤がある場合
- ゼロトラスト方針により、全アクセスを単一ポリシーエンジン（OPA, Cedar 等）で検証する要件がある場合
- 新規にエージェント基盤を構築し、認証設計を白紙から始められる場合
- エージェントの権限を「最小権限」で一括管理し、定期的な権限棚卸を集中的に行いたい場合

## 連合認証を選ぶ条件

- 各事業部門が独自の IdP やセッション管理を運用しており、統合コストが高い場合
- データ主権・地域規制（GDPR, 中国データ安全法等）により、認証トークンを特定リージョンから出せない場合
- 既存の SSO 基盤（各 SaaS ネイティブの SAML/OIDC 連携）が成熟しており、エージェント用に再構築する合理性がない場合
- M&A 直後など、異なる認証基盤の統合が中長期課題として残っている過渡期の場合
- 各システムが独自のきめ細かい権限モデル（Salesforce のプロファイル/権限セット等）を持ち、集中化すると粒度が失われる場合

## ハイブリッド（併用）

- **集中ブローカー＋ローカルスコープダウン**: Okta で統一認証した後、各 SaaS 側でエージェント専用のスコープ制限を適用します。集中管理の一貫性と各システムのきめ細かい制御を両立する最も一般的なパターンです。
- **フェデレーション＋集中監査**: 認証は各システムに委ねつつ、全認証イベントを SIEM（Splunk, Sentinel 等）に集約して監査証跡の一元化だけを実現します。
- **段階的集約**: 初期は連合認証で稼働させ、エージェントの利用が拡大するにつれて高頻度アクセス先から順に集中ブローカーへ移行します。

## 判断フロー

1. **全社 IdP は統一されているか？** — Okta/Azure AD で統一済み → 集中ブローカーが自然です。IdP が分散 → 連合認証から開始しましょう。
2. **エージェントがアクセスするシステム数は？** — 目安として 5 以上 → 集中の管理効率が勝ります。3 以下 → 個別連携でも運用可能です。
3. **データ主権・地域制約があるか？** — リージョン制約あり → 連合認証またはリージョン別ブローカーを選びましょう。
4. **監査・コンプライアンス要件の厳しさは？** — SOX/ISMAP 等で統一監査証跡が必須 → 集中ブローカーまたは集中監査の併用を検討してください。
5. **移行に使える期間は？** — 短期（3 か月以内） → 連合認証で早期稼働しましょう。中長期 → 集中ブローカーへの段階的移行を計画してください。

## よくある失敗

- **集中ブローカーの単一障害点化**: ID ブローカーの障害で全エージェントが停止します。冗長構成とサーキットブレーカーの設計を怠ると、全社業務に影響します。
- **連合認証でのトークン管理の複雑化**: 各 SaaS ごとに異なるトークンライフサイクルを管理し、リフレッシュ失敗やトークン失効の検知漏れが多発します。
- **権限の不整合**: 集中ポリシーと各 SaaS のローカル権限が矛盾し、エージェントが「集中側では許可されているがSaaS側で拒否される」状態になります。
- **過度な集中化による硬直**: 全権限変更が集中ブローカーのチケット承認を経由するため、事業部門のアジリティが低下します。
- **連合認証でのセキュリティ基準のばらつき**: 各 SaaS の認証強度（MFA 要否、セッション有効期限等）がシステムごとに異なり、最弱のリンクがセキュリティホールになります。
- **移行期の二重管理**: 連合から集中への移行途中に、両方の仕組みが並存し、どちらが正（source of truth）か混乱します。移行計画にはカットオーバー基準を明確にしましょう。

## 実装上の考慮点

- 集中ブローカーを採用する場合、OAuth 2.0 Token Exchange（RFC 8693）を活用してユーザートークンからエージェント用のスコープ制限トークンを発行する設計が一般的です。
- 連合認証の場合、各 SaaS のトークンリフレッシュ間隔とエージェントのセッション長を整合させてください。Salesforce は 2 時間、ServiceNow は 30 分などデフォルト値が異なるため、エージェント側で統一的なリフレッシュ戦略を持つことが重要です。
- いずれの方式でも、エージェントが保持するトークンの暗号化保存とメモリ上の保持時間最小化は必須要件となります。
