# D09 · モデルサイズ

!!! abstract "一言で"
    AIエージェントが使用する LLM のモデルサイズ（パラメータ数・能力クラス）を制御する
    チューニングパラメータです。大きいモデルほど推論品質が高いですがコストとレイテンシが増大し、
    小さいモデルは高速・低コストですが複雑なタスクでの精度が低下します。エージェントではタスクの
    難易度に応じて動的にモデルを切り替える「難易度ルーティング」が鍵となります。

## トレードオフ

| 緩める方向（大きいモデル） | 締める方向（小さいモデル） |
|---|---|
| 複雑な推論、多段の論理展開、微妙なニュアンスの理解が正確になる | API コスト（トークン単価）を大幅に削減できる |
| ツール選択の精度が向上し、エージェントの信頼性が上がる | レスポンスレイテンシが短縮され、ユーザー体感が向上する |
| 少ないプロンプトエンジニアリングで高い性能を引き出せる | スループットが高く、大量リクエストの同時処理に適する |

## 判断軸と決め方

すべてのリクエストに同一モデルを使うのではなく、**難易度ルーティング**によりタスクに応じて最適なモデルを選択します。

**難易度分類の基準**

- **高難易度**: 複数ツールの連鎖、複雑な条件分岐を含む判断、自由記述の生成、不明確な指示の解釈 → 大モデル（GPT-4 クラス、Claude Opus 等）
- **中難易度**: 定型的だが文脈理解が必要なタスク、1〜2 ツールの呼び出し、要約・翻訳 → 中モデル（GPT-4o-mini クラス、Claude Sonnet 等）
- **低難易度**: 分類、抽出、テンプレート埋め込み、意図分類 → 小モデル（Claude Haiku 等）または fine-tuned 小モデル

**ルーティングの実装方法**

1. **ルールベース**: タスク種別（意図分類の結果）に基づいて静的にモデルを割り当てます。Zendesk の問い合わせ分類は小モデル、解決策の提案は大モデル、という振り分けです
2. **LLM ルーター**: 小モデルでまずリクエストの難易度を判定し、必要に応じて大モデルにエスカレーションします。初期コストは増えますがリクエスト全体のコスト最適化が可能です
3. **カスケード**: まず小モデルで処理し、confidence が閾値未満なら大モデルで再処理します

**コスト意識の重要性**

大モデルと小モデルではトークン単価が 10〜50 倍異なることがあります。月間 100 万リクエストのエージェントで全量を大モデルに流すと、月額コストが数百万円〜数千万円に達する可能性があります。

## 目安（出発点）

- 意図分類・エンティティ抽出: 小モデル（Haiku クラス）
- FAQ 応答・定型回答生成: 小〜中モデル
- 複数ツール連携・判断を伴う業務処理: 大モデル（Opus / GPT-4 クラス）
- コスト目標: 目安として全リクエストの 60〜70% を小〜中モデルで処理し、大モデルの使用は 30〜40% に抑えます

## 計測指標

- **モデル別の処理件数割合**: 大/中/小モデルそれぞれの処理件数比率
- **モデル別の品質スコア**: 各モデルでの応答品質（LLM-as-judge, ユーザー評価）を比較
- **モデル別コスト**: トークン単価 x 使用量でモデルごとの月間コストを追跡
- **ルーティング精度**: 難易度判定が適切だったかの事後評価（小モデルで処理して品質不足だったケースの割合）

## 調整方針

- 小モデルでの品質不足率が目安として 10% を超える場合: ルーティングルールを見直し、該当タスクを中〜大モデルに切り替えてください
- 大モデルの使用比率が 50% を超えコスト目標を超過する場合: 意図分類の精度を改善し、小モデルで処理可能なタスクの範囲を広げてください。プロンプトの改善で小モデルの性能を引き上げることも検討してください
- 新モデルのリリース時: 既存のベンチマーク（自社 eval セット）で性能を検証し、コスト対効果が改善されるならモデルを切り替えてください
