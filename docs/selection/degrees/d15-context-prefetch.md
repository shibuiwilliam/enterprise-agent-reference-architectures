# D15 · コンテキスト読込

!!! abstract "一言で"
    エージェントがタスク実行前にどの程度の情報を先読み（prefetch）するかの程度です。
    積極的に先読みすれば応答が速くなりますが、不要な情報の取得コストとコンテキスト
    汚染のリスクが増します。利用確率の高い情報のみを先読みするのが基本戦略です。

## トレードオフ

| 緩める方向（積極的に先読み） | 締める方向（必要時のみ取得） |
|---|---|
| 初回応答が速くなり、ユーザー体験が向上する | API 呼び出し回数とトークン消費を最小化できる |
| 不要な情報がコンテキストに混入し、回答精度を下げる可能性がある | 会話の途中で追加取得が発生し、応答が遅くなることがある |
| API レート制限に早く到達するリスクがある | 必要な情報を見落とし、再取得のラウンドトリップが発生する |
| 個人情報や機密情報を不必要に読み込むリスクが増える | ユーザーが「待たされている」と感じる場面が増える |

## 判断軸と決め方

### 利用確率による分類

先読みの対象を「利用確率」で分類し、高確率の情報のみを prefetch します。

| 利用確率 | 例 | 先読み方針 |
|---|---|---|
| **ほぼ確実（90%超）** | ユーザーの所属部署、権限ロール、直近のセッション履歴 | 常に prefetch |
| **高い（50〜90%）** | ユーザーが関わる進行中の Jira チケット、直近の Salesforce 商談 | セッション開始時に prefetch |
| **中程度（20〜50%）** | 関連する Confluence ドキュメント、過去のやりとり要約 | 意図が確定した時点で取得 |
| **低い（20%未満）** | 他部署のデータ、過去 6 ヶ月以上前の履歴 | 明示的な要求があった場合のみ取得 |

### prefetch 戦略のパターン

1. **ユーザープロファイル prefetch**: セッション開始時にユーザーの基本情報（Okta プロファイル、Workday の所属情報、権限）を取得します。ほぼすべてのタスクで必要です。
2. **意図ベース prefetch**: ユーザーの最初の発話を分類した後、そのカテゴリに関連する情報を一括取得します。「経費精算について」→ Workday の未承認経費一覧を取得します。
3. **履歴ベース prefetch**: 過去のセッションで頻繁に参照されたデータソースを学習し、先読み対象を動的に調整します。
4. **なし（lazy fetch）**: 一切先読みせず、必要になった時点で都度取得します。コスト最小ですが応答遅延が大きくなります。

### エンタープライズでの典型シナリオ

- **Salesforce 営業支援エージェント**: ユーザーの担当商談リスト（上位 10 件）をセッション開始時に prefetch します。商談の詳細は指定された時点で取得します。
- **ServiceNow IT サポート**: ユーザーのデバイス情報と未解決チケットを prefetch します。ナレッジ記事は意図分類後に検索します。
- **Slack 社内ボット**: ユーザーの所属チャンネル・ロールのみ prefetch します。具体的なデータは要求時に取得します。

## 目安（出発点）

| prefetch 対象 | タイミング | 取得量の目安 |
|---|---|---|
| ユーザープロファイル・権限 | セッション開始時 | 1 API 呼び出し、1KB 以内 |
| 直近のコンテキスト（チケット・商談等） | セッション開始時 | 上位 5〜10 件、要約のみ |
| ドメイン固有の参照データ | 意図分類後 | RAG 検索上位 3〜5 件 |
| 詳細データ（レコード全体・添付ファイル） | 明示的な要求時 | 必要最小限 |

目安として、prefetch で消費するトークンはコンテキストウィンドウの **10〜15% 以内** に抑えてください。

## 計測指標

- **prefetch ヒット率**: 先読みした情報が実際にタスクで使われた割合です。60% 未満なら先読み対象を見直してください。
- **初回応答時間（TTFR）**: prefetch の効果を測る主要指標です。prefetch なしと比較して改善幅を確認します。
- **追加取得のラウンドトリップ回数**: prefetch でカバーできず追加 API 呼び出しが発生した回数です。
- **prefetch による API コスト**: 先読みだけで発生する API 呼び出し回数とトークン消費量です。
- **コンテキスト利用率**: 投入したコンテキストのうち、実際に回答生成に寄与したトークンの割合です。

## 調整方針

1. **prefetch ヒット率を週次で計測** し、60% を下回る情報ソースは prefetch 対象から外してください。
2. **初回応答時間が目標（目安として 3 秒以内）を超える場合**、prefetch の並列化または対象の絞り込みを行ってください。
3. **API レート制限に近づいている場合**、低利用確率の prefetch を削減し、キャッシュ（P24）の活用を検討してください。
4. **新しいデータソースを追加した際**、最初は lazy fetch で開始し、利用頻度データが溜まってから prefetch に昇格してください。
5. **個人情報・機密情報を含むデータソース** は、prefetch の対象とする前にデータ分類とアクセス権限の確認を必須としてください。
