# D01 · ログ詳細度

!!! abstract "一言で"
    AIエージェントが生成・処理するログの粒度と保持範囲を制御するチューニングパラメータ。
    詳細にしすぎるとストレージコストが膨らみ、絞りすぎるとインシデント時の原因追跡や
    監査対応が困難になる。エージェント固有の課題として、プロンプト全文・中間推論・ツール呼び出しの
    どこまでを記録するかという判断が従来のアプリケーションログよりも複雑になる。

## トレードオフ

| 緩める方向（詳細に記録） | 締める方向（簡素に記録） |
|---|---|
| デバッグが容易になり、不具合の再現・原因特定が速い | ストレージコストを抑制でき、大量リクエスト環境でも持続可能 |
| 監査証跡が充実し、コンプライアンス要件を満たしやすい | ログ検索のノイズが減り、運用チームの認知負荷が下がる |
| ハルシネーション発生時の根本原因分析が可能 | PII・機密情報の意図しない記録リスクを低減できる |

## 判断軸と決め方

ログ詳細度の初期設定は、**記録対象の分類**と**サンプリング戦略**の2軸で決める。

**記録対象の分類**

- **エラー・異常系**: LLM のタイムアウト、ツール呼び出し失敗、ガードレール発動などは**全量記録**が原則。Zendesk 連携エージェントで顧客対応が失敗した場合、プロンプト・レスポンス・ツール入出力の全文がないと原因を特定できない。
- **低評価フィードバック**: ユーザーが「役に立たなかった」と評価したセッションは、プロンプト全文と中間推論を含めて全量記録する。改善のための最重要データソースとなる。
- **正常系**: 成功リクエストは**メタデータ（トークン数、レイテンシ、モデル名、ツール呼び出し回数）のみ全量**、プロンプト本文は**サンプリング（目安として 1〜5%）**で記録する。

**保存先の分離**

- メタデータ（構造化データ）: OpenTelemetry 経由でログ基盤（Datadog, Splunk 等）に投入。検索・集計・アラートに使う。
- プロンプト本文・レスポンス全文（非構造化データ）: オブジェクトストレージ（S3, GCS 等）に保存し、必要時にのみ参照する。ログ基盤に全文を入れるとコストが桁違いに膨らむ。

## 目安（出発点）

- エラー・低評価セッション: 全量記録（本文含む）
- 正常セッションのメタデータ: 全量記録
- 正常セッションの本文: 目安として 1〜5% サンプリング。Salesforce 連携のような高頻度エージェントでは 1%、社内ナレッジ検索のような低頻度エージェントでは 5〜10% を起点とする
- 保持期間: メタデータは 90 日、本文は 30 日を起点に、業界規制に応じて延長する

## 計測指標

- **月間ログストレージコスト**: 急増していないか（目安としてエージェント1リクエストあたりのログコストを追跡）
- **インシデント解決時間（MTTR）**: ログ不足で原因特定に時間がかかっていないか
- **監査クエリ対応時間**: コンプライアンスチームからの照会に必要な情報を何分で抽出できるか
- **PII 検出率**: ログ内に意図しない PII が混入していないか（DLP スキャンで定期チェック）

## 調整方針

- MTTR が悪化（目安として平均 30 分超）している場合: 正常系のサンプリング率を引き上げるか、特定のエージェント/ツール経路を全量記録に切り替える
- ストレージコストが予算の 120% を超過した場合: 正常系サンプリング率を下げる、保持期間を短縮する、本文の保存を圧縮形式に切り替える
- 新規エージェント導入後 2 週間は全量記録とし、安定を確認してからサンプリングに移行する
