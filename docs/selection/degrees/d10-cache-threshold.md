# D10 · キャッシュ類似閾値

!!! abstract "一言で"
    セマンティックキャッシュにおいて、新しいクエリと過去のキャッシュエントリが「十分に類似している」
    と判定する閾値を制御するチューニングパラメータです。閾値が低い（緩い）とキャッシュヒット率が
    上がりコストが削減されますが、文脈が微妙に異なるクエリに誤った回答を返す「誤ヒット」のリスクが
    増大します。

## トレードオフ

| 緩める方向（低い閾値＝類似判定が甘い） | 締める方向（高い閾値＝類似判定が厳しい） |
|---|---|
| キャッシュヒット率が上がり、LLM API コストを大幅に削減できる | 誤ヒット（異なる意図のクエリに過去の回答を返す）を防止できる |
| 応答レイテンシが短縮される（キャッシュ応答は即座に返せる） | 個別化・文脈依存の応答品質を維持できる |
| LLM プロバイダのレート制限への到達を遅らせられる | 機密情報のテナント間漏洩リスクを回避できる |

## 判断軸と決め方

キャッシュ類似閾値は**データの性質**と**クエリの個別性**で大きく変わります。

**データの機密度による分類**

- **高機密・テナント固有**: 顧客ごとの契約情報、個人の勤怠・給与（Workday）、テナント固有の設定値 → キャッシュ自体を無効化するか、テナント ID をキャッシュキーに含めたうえで類似閾値を最高に設定します（目安として cosine similarity 0.98 以上）
- **中機密・組織共通**: 社内ポリシー、業務手順書、ServiceNow のナレッジベース → 組織内でのキャッシュ共有は可能ですが、閾値は厳格寄りに設定します（目安として 0.92〜0.95）
- **低機密・公開情報**: 製品 FAQ、公開ドキュメント、一般的なヘルプ → キャッシュを積極的に活用し、閾値を緩めに設定できます（目安として 0.85〜0.90）

**クエリの個別性による分類**

- **高個別性**: ユーザー固有の文脈に依存する質問（「私の先月の売上は？」）→ キャッシュ不適合、LLM に直接問い合わせます
- **低個別性**: 定型的な質問（「パスワードのリセット方法は？」「返品ポリシーは？」）→ キャッシュ効果が最大です。Zendesk の顧客サポートで頻出する定型質問は、低い閾値でもリスクが低くなります

**キャッシュキーの設計**

類似閾値だけでなく、キャッシュキーに含める要素（テナント ID、ユーザーロール、言語、時点など）で制御範囲を調整します。テナント ID をキーに含めれば、テナント間の誤ヒットは閾値に関係なく防止できます。

## 目安（出発点）

- 公開 FAQ / ヘルプデスク定型質問: cosine similarity 目安として 0.85〜0.90
- 社内ナレッジベース: cosine similarity 目安として 0.92〜0.95
- テナント固有データを含むクエリ: cosine similarity 目安として 0.98 以上（またはキャッシュ無効化）
- キャッシュ TTL: 目安として 1〜24 時間（データの更新頻度に応じて調整）
- キャッシュサイズ上限: 目安として 10,000〜100,000 エントリ（LRU で古いエントリを破棄）

## 計測指標

- **キャッシュヒット率**: 全リクエストに対するキャッシュ応答の割合。目安として 20〜40% を目標とします（ドメインにより大きく異なります）
- **誤ヒット率**: キャッシュから返された回答がユーザーの意図と合致しなかった割合。低評価フィードバックとキャッシュヒットの相関で推定します
- **キャッシュによるコスト削減額**: キャッシュヒットにより回避できた LLM API コストの月間総額
- **キャッシュ鮮度**: キャッシュエントリの平均経過時間と、古いキャッシュに基づく回答の品質劣化

## 調整方針

- 誤ヒット率が目安として 3% を超える場合: 閾値を引き上げてください（例: 0.88 → 0.92）。誤ヒットが特定のクエリパターンに集中しているなら、そのパターンのみキャッシュ除外してください
- ヒット率が 10% 未満でコスト削減効果が薄い場合: 閾値を引き下げるか、エンベディングモデルの精度を改善してください。キャッシュキーの設計（含める要素）を見直すことも有効です
- データ更新後に古い回答が返される場合: TTL を短縮するか、データ更新イベントをトリガーにキャッシュを明示的に無効化する仕組みを導入してください
