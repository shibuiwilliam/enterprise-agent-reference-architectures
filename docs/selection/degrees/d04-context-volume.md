# D04 · コンテキスト量

!!! abstract "一言で"
    AIエージェントが LLM に送信するコンテキスト（検索結果、ドキュメント、会話履歴など）の
    量を制御するチューニングパラメータ。コンテキストが多いほど根拠に基づいた応答が可能になるが、
    トークンコスト・推論レイテンシ・注意の分散（"lost in the middle" 問題）が悪化する。

## トレードオフ

| 緩める方向（多くのコンテキスト） | 締める方向（少ないコンテキスト） |
|---|---|
| グラウンディングが向上し、ハルシネーションが減る | トークンコストを大幅に削減できる |
| 複雑な質問にも必要な情報が揃いやすい | 推論レイテンシが短くなり、ユーザー体感が向上する |
| 網羅的な回答が可能になる | 関連度の低い情報による注意分散を防ぎ、精度が安定する |

## 判断軸と決め方

コンテキスト量の最適化は**静的な上限設定**ではなく、**動的な最適化パイプライン**として設計する。

**リランキングによる品質制御**

RAG で取得した候補ドキュメント（目安として 20〜50 件）から、リランカー（Cross-Encoder 等）で関連度上位を選別する。Zendesk のナレッジベースから FAQ を検索する場合、ベクトル検索で 30 件取得し、リランキングで上位 5〜8 件に絞るのが典型的な流れ。

**トークン予算による量制御**

LLM のコンテキストウィンドウに対して「トークン予算」を設定する。目安としてウィンドウの 60〜70% をコンテキストに、残りをシステムプロンプト・会話履歴・出力用に確保する。128K トークンのモデルであれば、コンテキストに目安として 70K〜90K トークンを上限とする。

**情報の種類別の優先度**

- **最重要**: ユーザーの直近の質問に直接関連するドキュメント
- **重要**: 会話履歴（直近 3〜5 ターン）
- **補助**: 背景情報、関連する過去のチケット（ServiceNow, Jira の類似ケース等）
- **低優先**: 一般的なガイドライン、ポリシー文書（必要時のみ）

予算を超過する場合は、低優先から順に除外するか要約に置き換える。

## 目安（出発点）

- RAG 取得件数: 目安として 20〜50 件（リランキング前）
- リランキング後の投入件数: 目安として 3〜8 件
- 会話履歴: 目安として直近 3〜5 ターン（古いターンは要約）
- コンテキストのトークン予算: 目安としてウィンドウの 60〜70%

## 計測指標

- **グラウンディング率**: 応答が提供されたコンテキストに基づいている割合（LLM-as-judge で評価）
- **ハルシネーション率**: コンテキストにない事実を生成した割合
- **コンテキストあたりのコスト**: 1 リクエストあたりの入力トークン数とそのコスト
- **応答レイテンシ vs コンテキスト量の相関**: コンテキスト量の増減がレイテンシに与える影響

## 調整方針

- ハルシネーション率が高い場合: リランキング後の投入件数を増やし、検索クエリの改善（HyDE、クエリ分解等）を検討する
- レイテンシが目標を超過する場合: 投入件数を減らすか、コンテキストの事前要約を導入する
- コスト予算を超過する場合: モデルサイズの見直し（D09）と合わせて、コンテキスト量を削減する。セマンティックキャッシュ（P24）の導入も検討する
