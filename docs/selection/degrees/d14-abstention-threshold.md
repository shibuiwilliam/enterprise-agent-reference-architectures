# D14 · 棄権閾値

!!! abstract "一言で"
    エージェントが「自信がない」と判断して回答を棄権（人間にエスカレーション）する
    信頼度スコアの閾値。低くすれば自動解決率が上がるが誤答リスクも上がる。
    高くすれば安全だがエスカレーションが増え、人間の負荷と応答遅延が増大する。

## トレードオフ

| 緩める方向（低い閾値＝棄権しにくい） | 締める方向（高い閾値＝棄権しやすい） |
|---|---|
| 自動解決率が上がり、人間の負荷が減る | 誤答・誤操作が減り、信頼性・安全性が向上する |
| 誤答がユーザーに到達し、信頼毀損や実害が発生しうる | エスカレーションが増えすぎ、人間がボトルネックになる |
| 「間違った回答を自信満々にする」ケースが増える | ユーザーの待ち時間が増加し、エージェント導入の意味が薄れる |

## 判断軸と決め方

### 誤答コストによる分岐

棄権閾値の設定は「誤答した場合のコスト」を基準に決める。

| 誤答コスト | 例 | 閾値の方針 |
|---|---|---|
| **致命的**（不可逆・法的リスク・金銭損害） | 返金処理の金額誤り、契約条件の誤案内、医療・法律相談 | 高め（0.85〜0.95）。少しでも不確実なら棄権 |
| **中程度**（修正可能だが手間がかかる） | Jira チケットの優先度誤判定、Salesforce の商談ステージ誤更新 | 中程度（0.70〜0.85） |
| **軽微**（すぐ修正でき影響が限定的） | FAQ の回答候補表示、Slack での情報検索結果 | 低め（0.50〜0.70）。多少の誤りは許容 |

### 信頼度スコアの算出方法

棄権閾値を機能させるには、信頼度スコアの設計が重要になる。

- **モデルの logprob**: トークンレベルの確率を集約する。分類タスクでは有効だが、自由形式の回答では使いにくい。
- **自己評価プロンプト**: 「回答の確信度を 0〜1 で評価せよ」と追加プロンプトで問う。キャリブレーションが必要。
- **複数回生成の一致度**: 同じ入力を N 回（目安として 3〜5 回）生成し、回答の一致率を信頼度とする。コストはかかるがロバスト。
- **検索ヒットの関連度スコア**: RAG ベースの回答では、検索結果の類似度スコアを信頼度の代理指標にする。

### エンタープライズでの典型シナリオ

- **Zendesk 顧客対応**: 返金・解約に関する回答は閾値 0.90 で厳格に棄権。商品情報の案内は 0.65 で自動回答を優先。
- **ServiceNow IT サポート**: パスワードリセット手順（定型）は閾値 0.50、権限変更の承認判断は閾値 0.90。
- **Salesforce 営業支援**: 商談の受注確度予測は閾値 0.75。確度が低い場合は「データが不十分なため判断を保留します」と棄権。

## 目安（出発点）

| リスクレベル | 棄権閾値の目安 | 期待される自動解決率 |
|---|---|---|
| 致命的（金銭・法的・不可逆） | 0.85〜0.95 | 40〜60% |
| 中程度（修正可能） | 0.70〜0.85 | 60〜80% |
| 軽微（情報提供のみ） | 0.50〜0.70 | 80〜95% |

これらは起点であり、本番データで校正（キャリブレーション）して調整する前提。

## 計測指標

- **自動解決率**: エスカレーションせずにエージェントが完了した割合。
- **誤答率（false positive）**: 自動回答のうち誤っていた割合。人間レビューまたは後続フィードバックで計測。
- **不要棄権率（false negative）**: 棄権したが実際には正しく回答できていたケースの割合。棄権ケースのサンプルを人間が事後評価する。
- **エスカレーション後の解決時間**: 棄権→人間対応の所要時間。長すぎればキュー設計の問題。
- **信頼度スコアのキャリブレーション**: 「信頼度 0.8 と出力した回答の実際の正答率が 80% 前後か」を検証する。

## 調整方針

1. **初期は高めの閾値（0.85）で開始** し、誤答率を十分に低く保った状態でデータを蓄積する。
2. **2〜4 週間のデータ蓄積後**、不要棄権率を分析し、閾値を 0.05 刻みで段階的に下げる。
3. **誤答率が許容範囲（目安として 2〜5%）を超えたら** 即座に閾値を戻す。
4. **タスクカテゴリごとに異なる閾値** を設定する。一律の閾値は避け、リスクレベルに応じた多段構成にする。
5. **信頼度スコアのキャリブレーションを月次で実施** し、モデル更新やデータ変化によるドリフトを補正する。
6. **棄権時のユーザー体験** も設計する。「確認中です、担当者におつなぎします」のように、棄権を透明に伝える。
