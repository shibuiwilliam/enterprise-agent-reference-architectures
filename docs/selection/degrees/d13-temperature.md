# D13 · temperature

!!! abstract "一言で"
    LLM の出力の確率分布をどの程度「なます」かを制御するパラメータです。
    高いほど多様・創造的な出力になり、低いほど再現性・決定論的な出力になります。
    タスクの性質に応じて使い分けることで、品質と安定性を両立させます。

## トレードオフ

| 緩める方向（高 temperature） | 締める方向（低 temperature） |
|---|---|
| 多様な表現・発想が得られ、創造的タスクに適する | 同一入力に対して同一出力が得られやすく、テスト・監査が容易 |
| 出力のばらつきが大きく、品質の安定性が下がる | 表現が画一的になり、ユーザーに「機械的」と感じさせる |
| 幻覚（hallucination）の発生確率が上がる傾向がある | 最適解以外の選択肢を探索できず、局所最適に陥りやすい |

## 判断軸と決め方

### タスク分類による設定

temperature の最適値はタスクの性質で大きく異なります。以下の分類を出発点としてください。

**低 temperature（0〜0.2）が適するタスク**

- 事実抽出・データ変換: Salesforce のレコードから特定フィールドを抽出する、JSON を整形する
- 分類・ラベリング: Zendesk チケットの優先度判定、Jira イシューのカテゴリ分類
- コード生成・SQL 生成: 構文の正確性が最優先
- Yes/No 判定: 承認条件の充足判定、ポリシー違反チェック

**中 temperature（0.3〜0.7）が適するタスク**

- 要約・報告書生成: Confluence ページの要約、週次レポートの文章化
- 顧客対応の返信案: Zendesk の返信ドラフト（定型とカスタマイズのバランス）
- 会話的な応答: Slack ボットのカジュアルな受け答え

**高 temperature（0.7〜1.0）が適するタスク**

- ブレインストーミング支援: 新機能のアイデア出し、マーケティングコピーの案出し
- 多様な言い回し生成: A/B テスト用の文面バリエーション
- 創作的コンテンツ: ブログ記事の下書き、プレゼン資料のストーリー構成

### エージェント内での使い分け

1つのエージェント内でも、ステップごとに temperature を切り替えることが有効です。

- **意図分類ステップ**: temperature = 0（決定論的に分岐）
- **情報検索・ツール選択ステップ**: temperature = 0〜0.1（正確なツール呼び出し）
- **回答生成ステップ**: temperature = 0.3〜0.5（自然な文章だが安定）
- **提案・アイデア出しステップ**: temperature = 0.7〜1.0（多様性重視）

### エンタープライズでの典型シナリオ

- **ServiceNow IT ヘルプデスク**: チケット分類（temperature=0）→ ナレッジ検索（0）→ 回答生成（0.3）の 3 段構成です。
- **Shopify 商品説明生成**: 商品属性の構造化抽出（0）→ 説明文の複数バリエーション生成（0.8）→ 品質チェック（0）です。
- **Slack ブレスト支援ボット**: 全ステップで temperature=0.9 とし、多様なアイデアを優先します。

## 目安（出発点）

| タスク種別 | temperature 目安 | top_p 目安 |
|---|---|---|
| 事実抽出・分類・判定 | 0〜0.1 | 0.9〜1.0 |
| 構造化データ変換 | 0 | 1.0 |
| 要約・報告書 | 0.3〜0.5 | 0.9 |
| 顧客対応ドラフト | 0.3〜0.5 | 0.9 |
| 会話応答 | 0.5〜0.7 | 0.9 |
| 創作・アイデア出し | 0.7〜1.0 | 0.95 |

temperature と top_p を同時に大きく動かすと出力が不安定になります。片方を固定し片方で調整するのが基本です。

## 計測指標

- **出力一貫性（同一入力 N 回の一致率）**: 分類タスクなら 95% 以上の一致を目標とします。
- **幻覚率**: 事実と異なる記述の発生頻度です。temperature を上げると上がる傾向を監視してください。
- **ユーザー満足度（CSAT）**: 特に顧客対応の文面品質に対する評価です。
- **eval 成功率の分散**: temperature が高いほど eval 結果がぶれます。分散が大きすぎれば下げてください。

## 調整方針

1. **まず temperature=0 で eval を作成し**、ベースラインの品質を確認してください。そこから必要に応じて上げます。
2. **分類精度が不安定な場合**、temperature を 0 に固定した上でプロンプトを改善してください。temperature で精度問題を解決しようとしないでください。
3. **顧客対応の文面が画一的と指摘されたら**、0.3 → 0.5 のように 0.1〜0.2 刻みで段階的に上げてください。
4. **幻覚率が許容範囲を超えたら**、temperature を下げるか、事実検証ステップ（P20 二重トラック検証）を挟んでください。
5. **本番とテストで同一の temperature 設定** を使ってください。テスト時だけ 0 にすると、本番で初めて出力のぶれに気づくことになります。
