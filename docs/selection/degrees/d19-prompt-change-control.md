# D19 · プロンプト変更の統制

!!! abstract "一言で"
    エージェントのシステムプロンプトやツール定義の変更に対して、どの程度の承認・
    テスト・バージョン管理を求めるかの程度。厳格にすれば再現性と安全性が上がるが、
    開発速度が落ちる。本番影響度に応じて統制レベルを分けるのが現実的。

## トレードオフ

| 緩める方向（開発速度優先） | 締める方向（再現性・安全優先） |
|---|---|
| プロンプトの迅速な改善・実験が可能。A/B テストのサイクルが速い | 全変更が追跡可能で、障害時にどの変更が原因かを特定できる |
| 意図しない変更が本番に入り、品質低下やセキュリティ問題を引き起こす | 変更の承認プロセスがボトルネックになり、改善のリードタイムが長くなる |
| 「誰がいつ何を変えたか」が追跡できず、障害の原因究明が困難 | 小さな改善でも重い手続きが必要になり、チームのモチベーションが下がる |

## 判断軸と決め方

### 本番影響度による統制レベルの分岐

| 影響度 | 例 | 統制レベル |
|---|---|---|
| **高（顧客向け・金銭・法的）** | Zendesk 顧客対応、Shopify 注文処理、Salesforce 契約管理 | 厳格：コードレビュー＋eval 通過＋承認者 2 名 |
| **中（社内業務・広範囲）** | ServiceNow IT サポート、Workday 人事照会、Slack 全社ボット | 標準：コードレビュー＋eval 通過＋承認者 1 名 |
| **低（内部実験・限定公開）** | 開発チーム内のプロトタイプ、PoC、限定ユーザーへのカナリア | 軽量：セルフレビュー＋基本 eval＋ログ記録 |

### 統制の構成要素

1. **バージョン管理**: プロンプトを Git でコードと同様に管理する。差分の可視化と履歴の追跡が可能になる。
2. **eval ゲート**: 変更後のプロンプトが既存の eval セットを通過することをデプロイの前提条件とする。
3. **承認プロセス**: 影響度に応じた承認者（同僚エンジニア・テックリード・プロダクトオーナー）のレビュー。
4. **カナリアデプロイ**: 一部のトラフィックにのみ新プロンプトを適用し、品質指標を比較した上で全体展開する。
5. **ロールバック手順**: 問題が発生した場合に即座に前バージョンに戻せる仕組み。

### プロンプトの構成要素ごとの統制

すべてのプロンプト変更を同一レベルで統制する必要はない。

| 構成要素 | 変更頻度 | リスク | 統制方針 |
|---|---|---|---|
| **システムプロンプト（ロール・制約）** | 低い | 高い | 厳格。変更は設計判断として扱う |
| **ツール定義（名前・説明・スキーマ）** | 中程度 | 高い | 厳格。ツール選択精度に直結する |
| **Few-shot 例示** | 中程度 | 中程度 | 標準。eval で品質を確認 |
| **コンテキスト注入テンプレート** | 高い | 低〜中 | 軽量〜標準。フォーマット変更は eval 確認 |
| **出力フォーマット指示** | 低い | 低い | 軽量。下流システムとの整合を確認 |

### エンタープライズでの典型シナリオ

- **Zendesk 顧客対応エージェント**: システムプロンプトの変更は PM＋エンジニアリードの承認必須。返金上限額の記述変更は法務レビューも追加。eval 通過＋カナリア（10% トラフィック × 24 時間）を経て全体展開。
- **Slack 社内実験ボット**: 開発者がセルフレビューで変更可能。基本 eval の通過は必須だが、承認プロセスは省略。変更ログは自動記録。
- **Salesforce 営業支援**: ツール定義の追加・変更はコードレビュー必須。商談ステージの判定ロジックに影響するプロンプト変更はテックリードの承認。

## 目安（出発点）

| 統制レベル | 対象 | プロセスの目安 |
|---|---|---|
| 厳格 | 顧客向け・金銭影響・規制対象 | Git 管理＋eval 100% 通過＋承認者 2 名＋カナリア |
| 標準 | 社内業務・広範囲 | Git 管理＋eval 95% 通過＋承認者 1 名 |
| 軽量 | 内部実験・限定公開 | Git 管理＋基本 eval 通過＋変更ログ記録 |

目安として、厳格レベルの変更リードタイムは 1〜3 営業日、軽量レベルは数時間以内を目標とする。

## 計測指標

- **プロンプト変更の頻度**: 週あたりの変更回数。環境（本番・ステージング・実験）ごとに分けて計測。
- **変更→デプロイのリードタイム**: 変更がコミットされてから本番に反映されるまでの時間。
- **プロンプト変更起因の障害件数**: 変更後 24〜48 時間以内に発生した品質低下やインシデントの回数。
- **eval の回帰検出率**: プロンプト変更による品質低下を eval が事前に検出できた割合。
- **ロールバック発生率**: デプロイ後にロールバックが必要になった割合。

## 調整方針

1. **初期は厳格寄りで運用を開始** し、チームの習熟と eval の充実に応じて段階的に緩和する。
2. **変更リードタイムが長すぎる場合**、eval の自動化・承認プロセスの並列化で対応する。統制レベル自体を安易に下げない。
3. **プロンプト変更起因の障害が発生したら**、そのケースを eval に追加し、再発防止をプロセスに組み込む。
4. **実験環境と本番環境の統制レベルを明確に分ける**。実験の速度を本番の安全性と引き換えにしない。
5. **プロンプトの変更理由を必ず記録する**。「なぜ変えたか」が追跡できないと、将来の判断が困難になる。
6. **大規模なプロンプト変更（ロール定義の書き換え等）は段階的に実施** する。一度に大きく変えると影響範囲の特定が困難になる。
