# D02 · タイムアウト

!!! abstract "一言で"
    AIエージェントの各処理ステップに設ける時間上限を制御するチューニングパラメータです。
    LLM 推論はレスポンスタイムのばらつきが大きく、従来の API とは異なるタイムアウト設計が必要になります。
    短すぎると複雑な推論が途中で打ち切られ、長すぎるとリソースを占有しユーザー離脱を招きます。

## トレードオフ

| 緩める方向（長いタイムアウト） | 締める方向（短いタイムアウト） |
|---|---|
| 複雑な推論・多段ツール呼び出しが完了しやすい | リソース（スレッド・コネクション）の占有を抑制できる |
| LLM プロバイダの一時的な遅延を吸収できる | ユーザーの待ち時間を短く保ち、離脱率を下げられる |
| 完了率が向上し、中途半端な結果の再試行コストが減る | 障害時のカスケード（タイムアウト待ちの連鎖）を防止できる |

## 判断軸と決め方

タイムアウト設計は**処理段階ごと**に分けて考えましょう。エージェント全体に単一のタイムアウトを設定するのは避けてください。

**段階別の考え方**

- **LLM 推論**: 使用モデルの P99 レイテンシの 2〜3 倍を起点とします。GPT-4 クラスで P99 が 15 秒なら、目安として 30〜45 秒です。ストリーミングを使う場合は first-token タイムアウトと全体タイムアウトを分離してください。
- **ツール呼び出し**: 外部 SaaS の SLA に依存します。Salesforce API なら目安として 10〜15 秒、ServiceNow API なら 15〜20 秒を起点に、各 API の P99 を実測して決めましょう。
- **エージェント全体（エンドツーエンド）**: 上記の合計に余裕を加算します。対話型なら目安として 60〜120 秒、バッチ型なら数分〜数十分です。

**副作用の有無による区別**

- **副作用なし（読み取り系）**: タイムアウト後に即座にフォールバック応答を返せます。やや短めでも許容できます。
- **副作用あり（書き込み・送信系）**: タイムアウトで打ち切った時点で副作用が実行済みかどうか不明になります。Jira チケットの作成や Slack メッセージの送信など不可逆な操作では、タイムアウトを長めに取り、打ち切り時のロールバック/確認手順を設計しておきましょう。

## 目安（出発点）

- LLM 推論（first-token）: 目安として 10〜15 秒
- LLM 推論（全体）: 目安として 30〜60 秒（モデルサイズに依存）
- 外部ツール呼び出し: 目安として各 API の P99 の 2 倍
- エージェント全体（対話型）: 目安として 60〜120 秒
- エージェント全体（バッチ型）: 目安として 5〜30 分（タスク複雑度に依存）

## 計測指標

- **タイムアウト発生率**: 全リクエストに対するタイムアウト打ち切りの割合です。目安として 1% 未満を目標にしましょう
- **P50 / P95 / P99 レイテンシ**: 各処理段階の分布を継続的に追跡します
- **タイムアウト後の再試行率**: タイムアウトで失敗した後にユーザーが同じリクエストを再送している割合です
- **副作用不整合件数**: タイムアウト打ち切り後に副作用の状態確認が必要になったケース数です

## 調整方針

- タイムアウト発生率が 1% を超える場合: P99 を再計測し、タイムアウト値を引き上げてください。ただし根本原因（モデル遅延、API 劣化）の調査を優先しましょう
- ユーザー離脱率が高い場合: ストリーミング応答を導入し体感待ち時間を短縮します。タイムアウト自体は変えずに UX で対処することも検討してください
- 副作用不整合が発生した場合: 副作用ありの操作のタイムアウトを延長し、冪等キーの導入を検討しましょう
