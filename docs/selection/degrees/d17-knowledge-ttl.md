# D17 · 知識鮮度(TTL)

!!! abstract "一言で"
    エージェントが参照する知識（ベクトルストア・キャッシュ・ナレッジベース）を
    どの頻度で更新・無効化するかの程度です。短い TTL は鮮度を保ちますが計算・コストが
    増え、長い TTL はコスト効率が良いですが古い情報で回答するリスクがあります。

## トレードオフ

| 緩める方向（短い TTL＝頻繁に更新） | 締める方向（長い TTL＝更新を抑える） |
|---|---|
| 最新情報で回答でき、誤情報リスクが低い | 再インデックス・再計算のコストと負荷を抑えられる |
| 再インデックスの計算コスト・API コストが増大する | 古い情報・削除済み情報で回答してしまうリスクがある |
| 更新中の一時的な不整合（部分更新状態）が発生しうる | ユーザーが「情報が古い」と不信感を持つ |

## 判断軸と決め方

### 情報の変動速度とリスクで分岐

TTL は一律に決めるのではなく、情報の種類ごとに異なる TTL を設定します。

| 情報の種類 | 変動速度 | 誤情報のコスト | TTL の方針 |
|---|---|---|---|
| **権限・アクセス制御** | 即時反映必須 | 致命的（情報漏洩） | TTL=0（リアルタイム or イベント駆動） |
| **削除されたデータ** | 即時反映必須 | 致命的（プライバシー違反） | TTL=0（削除イベントで即時無効化） |
| **価格・在庫・為替** | 秒〜分単位 | 高（金銭的損失） | TTL=1〜5 分 |
| **チケット・商談のステータス** | 時間〜日単位 | 中（業務判断の誤り） | TTL=5〜30 分 |
| **社内ポリシー・マニュアル** | 週〜月単位 | 中（誤った手順の案内） | TTL=1〜24 時間 |
| **製品ドキュメント・FAQ** | 月〜四半期単位 | 低〜中 | TTL=24〜72 時間 |
| **法令・規制文書** | 年単位 | 高（コンプライアンス違反） | TTL=24 時間（変更は稀だが影響大） |

### 更新トリガーの種類

- **時間ベース（TTL）**: 一定時間経過後に無効化し再取得します。最もシンプルですが、変更がなくても再取得が走ります。
- **イベント駆動（CDC）**: データソース側の変更イベント（Webhook・CDC）をトリガーに更新します。効率的ですが、イベント基盤の構築が必要です。
- **ハイブリッド**: イベント駆動を基本とし、イベントの取りこぼしに備えて長めの TTL でフォールバックします。

### エンタープライズでの典型シナリオ

- **Salesforce 商談データ**: ステージ変更は Salesforce の Platform Event で CDC を受信し、即時反映します。商談メモは TTL=30 分です。
- **Confluence ナレッジベース**: Webhook で更新イベントを受け取り再インデックスします。Webhook が届かない場合に備えて TTL=6 時間のフォールバックを設定します。
- **Shopify 商品情報**: 価格・在庫は TTL=5 分です。商品説明は TTL=24 時間です。在庫切れは Webhook で即時反映します。
- **Box/Google Drive のドキュメント**: ファイル更新イベントで再インデックスします。削除イベントでベクトルストアからも即時削除します。

## 目安（出発点）

| データ分類 | TTL の目安 | 更新トリガー |
|---|---|---|
| 権限・削除・セキュリティ | 0（即時） | イベント駆動必須 |
| トランザクション（価格・在庫） | 1〜5 分 | イベント駆動＋短 TTL |
| 業務データ（チケット・商談） | 5〜30 分 | イベント駆動＋中 TTL |
| ナレッジ（マニュアル・FAQ） | 1〜24 時間 | Webhook＋長 TTL |
| アーカイブ（過去レポート） | 24〜72 時間 | 定期バッチ |

## 計測指標

- **知識の鮮度（staleness）**: 回答に使われた情報と、データソースの最新版との差分時間です。
- **古い情報による誤回答率**: 回答時点で既に更新・削除されていた情報を使った割合です。
- **再インデックスのコスト**: TTL による再取得・再計算にかかる API 呼び出し数とコスト（月額）です。
- **CDC イベントの取りこぼし率**: イベント駆動を使う場合、イベントが欠落した頻度です。
- **ユーザーからの「情報が古い」フィードバック件数**: 鮮度問題の直接的なシグナルです。

## 調整方針

1. **「情報が古い」フィードバックが増えたら** 該当データソースの TTL を短縮してください。
2. **再インデックスコストが予算を圧迫したら** 変動の少ないデータソースの TTL を延長し、イベント駆動への移行を検討してください。
3. **権限変更・データ削除は TTL に頼らないでください**。必ずイベント駆動で即時反映し、TTL はフォールバックとして設定します。これはセキュリティ要件であり交渉の余地がありません。
4. **新しいデータソースを追加する際**、最初は短めの TTL で運用し、変動頻度のデータが溜まってから TTL を延長してください。
5. **部分更新の整合性** に注意してください。関連する複数のデータ（例: 商談とその活動履歴）の TTL が大きく異なると、不整合が生じます。関連データは同じタイミングで更新する設計にしてください。
