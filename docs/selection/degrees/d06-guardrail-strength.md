# D06 · ガードレール強度

!!! abstract "一言で"
    AIエージェントの入出力に適用するガードレール（プロンプトインジェクション検知、トピック制限、
    PII マスキング、トーン制御など）の検知感度とブロック閾値を制御するチューニングパラメータです。
    強すぎると正当なリクエストが誤検知でブロックされ業務が止まり、弱すぎるとセキュリティや
    コンプライアンスのリスクが顕在化します。

## トレードオフ

| 緩める方向（弱いガードレール） | 締める方向（強いガードレール） |
|---|---|
| 正当なリクエストがブロックされにくく、業務のスループットが維持される | 敵対的入力（ジェイルブレイク、間接インジェクション）を高い確率で遮断できる |
| ユーザーの自由度が高く、創造的な用途に対応しやすい | PII 漏洩や不適切な出力のリスクを最小化できる |
| 誤検知によるユーザー不満やエスカレーションが減る | コンプライアンス要件（金融規制、医療情報保護等）を確実に満たせる |

## 判断軸と決め方

ガードレール強度は**利用者の種別**と**データの機密度**で大きく異なる設定が必要になります。

**利用者種別による分類**

- **顧客向け（外部）**: 敵対的入力を前提とした設計が必要です。プロンプトインジェクション検知、トピック制限、出力のDLP検査を最高強度で適用します。Shopify 連携の EC チャットボットなら、競合製品への誘導やシステムプロンプトの抽出を試みるユーザーを想定してください。
- **従業員向け（内部）**: 一定の信頼を前提に、業務を阻害しない範囲で設定します。ただし内部情報の社外持ち出し防止は必要です。Slack 上の社内エージェントなら、PII マスキングと機密情報の分類フィルタは有効にしつつ、トピック制限は緩めに設定します。

**データ機密度による分類**

- **高機密（個人情報、財務データ、医療情報）**: PII マスキング、出力分類器、DLP を最高強度で適用します。人間のレビュー必須とする設計判断が推奨されます。
- **中機密（社内業務データ）**: PII マスキングは有効にしつつ、トピック制限はビジネス文脈に合わせて調整します
- **低機密（公開情報、一般ナレッジ）**: 基本的なインジェクション検知のみです。トピック制限やトーン制御は最低限にします

**多層防御の考え方**

単一のガードレールに依存せず、入力側（プロンプト検査）と出力側（レスポンス検査）の両方に設置します。一方を緩めてもう一方で補完する設計が可能です。

## 目安（出発点）

- 顧客向けエージェント: インジェクション検知・トピック制限・PII マスキング・出力 DLP を全て有効（高強度）
- 従業員向けエージェント: インジェクション検知（中強度）・PII マスキング（高強度）・トピック制限（低強度）
- 検知閾値（confidence score）: 目安として 0.7〜0.85 を起点に、誤検知率を見ながら調整

## 計測指標

- **誤検知率（False Positive Rate）**: 正当なリクエストがブロックされた割合。目安として 1% 未満を目標
- **見逃し率（False Negative Rate）**: ガードレールを通過した不適切な入出力の割合。定期的な Red Team テストで計測
- **ブロック後のユーザー行動**: ブロック後にユーザーが離脱するか、言い換えて再試行するかの割合
- **ガードレール処理レイテンシ**: ガードレール追加による応答遅延の増分

## 調整方針

- 誤検知率が 1% を超える場合: 検知閾値を引き下げる（例: 0.85 → 0.75）か、誤検知パターンをホワイトリストに追加してください
- Red Team テストで見逃しが見つかった場合: 該当パターンのルールを追加し、閾値を引き上げてください。ただし誤検知率とのバランスを再確認してください
- ユーザーからの「正当なリクエストがブロックされた」報告が増加した場合: ブロックされたリクエストのログをサンプリングレビューし、ルールの過剰適用を修正してください
