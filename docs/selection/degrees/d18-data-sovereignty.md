# D18 · データ主権の厳格度

!!! abstract "一言で"
    エージェントが扱うデータの保存・処理・転送をどの程度厳密に地理的・法的制約に
    従わせるかの程度です。厳格にするほど規制順守は確実になりますが、利用できるモデル・
    リージョン・サービスの選択肢が狭まり、コストと運用複雑度が上がります。

## トレードオフ

| 緩める方向（利便性優先） | 締める方向（規制順守優先） |
|---|---|
| 最適なモデル・リージョンを自由に選べ、コスト最適化が容易 | GDPR・個人情報保護法・業界規制への準拠を確実にできる |
| 越境データ転送のリスクが発生し、規制違反時の罰則が厳しい | 利用可能なモデル・クラウドリージョンが限定され、選択肢が減る |
| 監査時に「データがどこにあるか」を説明できない | マルチリージョン構成の運用コストとレイテンシが増加する |

## 判断軸と決め方

### データ分類と居住地域による分岐

データ主権の厳格度は、**データの分類（機密度）** と **関連する法規制** の組み合わせで決まります。

| データ分類 | 規制が厳しい地域（EU・日本・中国等） | 規制が緩い地域 |
|---|---|---|
| **個人情報（PII）** | 域内処理必須。越境には十分性認定 or SCC | 域内処理が望ましい。リスク評価の上で判断 |
| **機密業務データ** | 域内処理を推奨。暗号化＋アクセス制御必須 | 暗号化＋アクセス制御で対応可能 |
| **公開情報・匿名化済みデータ** | 越境可能。ただし匿名化の妥当性を検証 | 制約なし |
| **規制対象データ（金融・医療）** | 業界固有規制に完全準拠。域内処理が事実上必須 | 業界規制に準拠。所在地の確認は必須 |

### 構成要素ごとの検討

データ主権は「モデル推論」「データ保存」「ログ・監査」の 3 層で個別に検討します。

1. **モデル推論（LLM API 呼び出し）**: プロンプトに含まれるデータがモデルプロバイダのリージョンに送信されます。Azure OpenAI Service や AWS Bedrock のリージョン指定が有効です。
2. **データ保存（ベクトルストア・DB）**: 永続化されるデータの物理的な保存場所です。クラウドのリージョン設定で制御します。
3. **ログ・監査証跡**: 会話ログ・トレースデータにも PII が含まれうる点に注意してください。ログ基盤のリージョンも対象です。

### エンタープライズでの典型シナリオ

- **EU 顧客のデータを扱う Zendesk エージェント**: GDPR 準拠のため、EU リージョン内でモデル推論・データ保存・ログすべてを完結します。Azure OpenAI の EU リージョンを利用します。
- **日本の金融機関の Salesforce 連携**: 金融庁のガイドラインに基づき、顧客データは日本国内のリージョンで処理します。クラウド利用自体の承認プロセスも必要です。
- **グローバル企業の Workday 人事エージェント**: 従業員の所在国に応じて処理リージョンを動的にルーティングします。米国従業員は US リージョン、EU 従業員は EU リージョンです。
- **Shopify EC の顧客データ**: 購買履歴は顧客の居住国に応じたリージョンで保持します。商品マスタ（非個人情報）はグローバル共有です。

## 目安（出発点）

| データ × 地域 | 厳格度の目安 | 主な対策 |
|---|---|---|
| PII × EU | 最厳格 | EU 域内完結。十分性認定のない国への転送禁止 |
| PII × 日本 | 厳格 | 国内リージョン推奨。越境は本人同意 or 基準適合 |
| 機密業務データ × 任意 | 中〜厳格 | 暗号化＋リージョン指定。監査ログ必須 |
| 公開/匿名化データ × 任意 | 緩め | リージョン制約なし。コスト最適化優先 |
| 規制対象（金融・医療） × 任意 | 最厳格 | 業界規制に完全準拠。所管官庁の承認を確認 |

## 計測指標

- **データ所在地の可視化率**: 全データストアのうち、リージョンが明確に管理・記録されている割合です。100% を目標とします。
- **越境転送の発生件数**: 意図しない越境データ転送の検出回数です。ゼロが目標です。
- **規制準拠の監査結果**: 外部監査・内部監査でのデータ主権関連の指摘事項数です。
- **リージョン制約によるレイテンシ増加**: 最適リージョンと実際のリージョンの差分によるレイテンシ影響です。
- **データ分類の正確性**: 自動分類ツールによる PII 検出の精度（適合率・再現率）です。

## 調整方針

1. **データ分類を先に確定してください**。分類が曖昧なまま厳格度を決めると、過剰な制約か不十分な保護のどちらかに傾きます。
2. **規制要件は最低ラインであり緩和の対象にしません**。利便性のために規制要件を下回る判断は許容しません。この判断は人間のレビュー必須です。
3. **新しいリージョンやモデルプロバイダが利用可能になった際** に、制約の見直しを行ってください。選択肢が増えれば厳格度を維持しつつコストを下げられる場合があります。
4. **マルチリージョン構成のコストが大きい場合**、データの匿名化・仮名化によってデータ分類を下げ、リージョン制約を緩和する方向を検討してください。
5. **モデルプロバイダのデータ処理契約（DPA）を定期的に確認** してください。契約条件の変更がデータ主権に影響する場合があります。
6. **データ主権の設計判断は法務・コンプライアンス部門と連携** し、技術チーム単独で決定しないでください。
