# D03 · リトライ回数

!!! abstract "一言で"
    AIエージェントが外部 API 呼び出しや LLM 推論に失敗した際に再試行する回数の上限を制御する
    チューニングパラメータです。リトライは一時的な障害を吸収する基本手段ですが、回数を増やしすぎると
    コスト増大や副作用の重複実行といったエージェント特有のリスクを招きます。

## トレードオフ

| 緩める方向（多くリトライ） | 締める方向（少なくリトライ） |
|---|---|
| 一時的なネットワーク障害・レート制限を吸収し、成功率が向上する | LLM API コスト（トークン課金）の無駄遣いを防止できる |
| 外部 SaaS のスパイク的な遅延に耐えられる | 副作用の重複実行リスクを最小化できる |
| ユーザーへの失敗応答を減らし体験が向上する | 障害時に早期フォールバックへ移行でき、回復が速い |

## 判断軸と決め方

リトライ回数は**操作の性質（冪等性と副作用の有無）**で大きく変わります。

**冪等な読み取り操作**

Salesforce からの顧客情報取得、ServiceNow のチケット検索など副作用がない操作は、リトライを多めに設定しても安全です。目安として 3〜5 回を推奨します。指数バックオフ（initial: 1 秒、factor: 2）と組み合わせてください。

**副作用のある書き込み操作**

Jira チケットの作成、Slack メッセージの送信、Workday の申請承認など不可逆な操作では、冪等キー（Idempotency Key）を前提としたうえで控えめに設定しましょう。冪等キーがない場合はリトライしない（0 回）か、1 回のみとしてください。

**LLM 推論のリトライ**

LLM の推論失敗（レート制限 429、サーバーエラー 5xx）は別のリトライポリシーで管理します。レート制限にはプロバイダの `Retry-After` ヘッダを尊重し、5xx にはバックオフ付きで 2〜3 回リトライしてください。ただしプロンプト全体を再送するため、トークンコストが倍増する点に注意が必要です。

**リトライ不適切なケース**

- 認証エラー（401/403）: リトライしても解決しません。即座にエラー返却してください
- バリデーションエラー（400）: 入力を修正しない限り結果は変わりません
- コンテキスト長超過: プロンプトの縮小が必要でリトライでは解決しません

## 目安（出発点）

- 冪等な読み取り API: 目安として 3〜5 回（指数バックオフ、最大待機 30 秒）
- 副作用あり（冪等キーあり）: 目安として 1〜2 回
- 副作用あり（冪等キーなし）: 目安として 0〜1 回
- LLM 推論（レート制限）: Retry-After に従い最大 3 回
- LLM 推論（5xx）: 目安として 2 回（バックオフ付き）

## 計測指標

- **リトライ成功率**: リトライによって最終的に成功した割合です。低い場合はリトライが無駄なコストになっています
- **リトライ後の総レイテンシ**: リトライを含むエンドツーエンドの応答時間への影響を確認します
- **副作用重複件数**: 冪等キーなしでリトライした結果、重複処理が発生したケース数です
- **リトライ起因のコスト**: リトライによる追加トークン消費量・API コール数を追跡します

## 調整方針

- リトライ成功率が 30% 未満の場合: 障害が一時的でない可能性が高いです。サーキットブレーカー（P23）の導入を検討してください
- 副作用重複が月 1 件以上発生: 該当操作のリトライを無効化し、冪等キーの実装を優先しましょう
- リトライ起因コストが全体の 10% を超える: リトライ上限を引き下げ、フォールバック応答への移行を早めましょう
