# D11 · ツール露出数

!!! abstract "一言で"
    エージェントに一度に見せるツール（関数・API）の数を制御する程度。
    多すぎるとモデルの選択精度が落ち、少なすぎるとユーザーの要求に応えられない。
    ツール数が増えるほど「正しいツールを選ぶ」認知負荷がモデルにかかる点を意識する。

## トレードオフ

| 緩める方向（効かなすぎ） | 締める方向（効きすぎ） |
|---|---|
| 全ツールを常時公開し機能網羅を優先。ユーザーがどんな要求をしても対応できる | ツールを極端に絞り選択精度を最優先。対応できるタスクの幅が狭くなる |
| ツール間の名前・説明の類似度が上がり、モデルが誤選択する頻度が増加 | ユーザーが「できるはずのこと」を断られるため、体験品質が低下する |
| ツール説明のトークン消費が増え、実質的なコンテキスト予算を圧迫する | ツール追加のたびに分割設計の見直しが必要になり、開発速度が落ちる |

## 判断軸と決め方

### ツール数の閾値

目安として、1エージェントに同時公開するツール数は **10〜20 が安定ゾーン** とされる。
30 を超えるとモデルの選択精度が有意に低下する傾向がある。

ツール数が 30 を超える場合は、以下のいずれかで対処する。

1. **Tool RAG（動的選択）**: ユーザーの発話をもとに関連ツールをベクトル検索し、上位 N 件だけをプロンプトに注入する。Salesforce や ServiceNow のように API が数百に及ぶ SaaS 連携では必須に近い。
2. **サブエージェント分割**: 業務ドメインごとにサブエージェントを分け、ルーターが適切なサブエージェントに振り分ける。例えば「人事系（Workday）」「IT ヘルプデスク（ServiceNow）」「営業支援（Salesforce）」のように分割する。
3. **ハイブリッド**: ルーターでドメイン分割した上で、各サブエージェント内でも Tool RAG を使う二段構成。

### ツール説明の品質

ツール数を減らす前に、まず **ツール説明（description）の品質** を見直す。名前や説明が曖昧・重複しているとモデルが混乱する。以下を確認する。

- ツール名が動詞＋名詞で一意か（`create_ticket` vs `make_ticket` の重複がないか）
- 説明に「いつ使うか」「いつ使わないか」が明記されているか
- 引数の型・制約・デフォルト値が明示されているか

### エンタープライズでの典型シナリオ

- **Slack ボットで社内業務を統合**: 初期は IT ヘルプデスク（5 ツール）から始め、HR・経費・施設と段階的に追加。15 ツールを超えた時点で Tool RAG を導入。
- **顧客向け Zendesk 連携エージェント**: FAQ 検索・チケット作成・ステータス確認・エスカレーションの 4 ツールに絞り、選択精度 95% 以上を維持。
- **Jira + Confluence + GitHub 開発支援**: 各サービス 5〜8 ツール、合計 20 前後。サブエージェント分割で対応。

## 目安（出発点）

| シナリオ | 目安のツール数 | 手法 |
|---|---|---|
| 単一ドメイン・顧客向け | 5〜10 | 静的定義で十分 |
| 社内マルチドメイン | 10〜20 | ドメイン別グルーピング |
| 大規模統合（SaaS 10 種超） | 20〜30 | Tool RAG またはサブエージェント分割 |
| 超大規模（API 数百） | 30 超 | ルーター＋サブエージェント＋Tool RAG |

## 計測指標

- **ツール選択正答率**: 正しいツールが呼ばれた割合。eval セットで定期計測する。
- **ツール未選択率（fallback 率）**: 適切なツールがあるのに「できません」と回答した割合。
- **ツール誤選択によるエラー率**: 間違ったツールが呼ばれ、エラーまたは不正な結果になった割合。
- **ツール説明のトークン消費量**: コンテキストウィンドウに占めるツール定義の割合。

## 調整方針

1. **ツール選択正答率が 90% を下回ったら** ツール数の削減または Tool RAG の導入を検討する。
2. **fallback 率が 5% を超えたら** ツールが足りない可能性がある。ユーザー要求のログを分析し、不足ツールを追加する。
3. **新ツールを追加するたびに** 既存ツールとの名前・説明の重複をチェックし、必要に応じて統合または改名する。
4. **ツール説明のトークン消費がコンテキストの 20% を超えたら** Tool RAG への移行を本格検討する。
5. 調整は **週次の eval 結果** をもとに行い、一度に大幅な変更を避ける。
