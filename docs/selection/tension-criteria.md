# 「相反する仕組み」の選定基準 ― ルックアップ表

設計プロトコル STEP4 で参照する。候補に競合がある場合、判断軸を引いて選定理由を記録し、
提案にそのまま転記する。

| 対立軸 | A を選ぶ | B を選ぶ | 関連 |
|---|---|---|---|
| 従業員向け vs 顧客向け | 社内・高信頼・広い権限 | 外部・敵対前提・隔離・厳格ガードレール | P01 |
| 同期 vs 非同期 | 会話的・数秒・往復が価値 | 数十秒超・バッチ・イベント・大量並列 | P22 |
| シングル vs マルチ | 単一目的・低レイテンシ・コスト敏感・ツール≤30 | 専門分離・並列調査・文脈/ツール過多・役割別権限 | P15 |
| 埋め込み vs ハブ | 単一システム完結・滞在長い | 横断業務・入口一本化（実務は併用） | P05,P06 |
| 自律 vs 決定論WF | 副作用が軽微・可逆・低頻度 | 不可逆・高影響・規制対象 | P17 |
| コンパイル時 vs ランタイムLLM | 高頻度・形式安定 | 一回性・高変動 | P19 |
| 事前 vs 事後検証 | 不可逆な副作用 | 情報提供のみ | P20 |
| RAG/FT/長文脈/ツール | 鮮度高い大量=RAG/ツール、形式習得=FT、小さく完結=長文脈、真実=ツール | （組み合わせ） | P13 |
| Pull vs Push | ユーザー起点・要求駆動 | 予防的・イベント起点 | P07 |
| ツール呼び出し vs コード生成 | 定義済み操作・安全性重視 | 柔軟な計算/加工（要サンドボックス） | P11 |
| オンライン vs オフライン知性 | 即時性・予測不能 | 大量・重い・事前計算可・コスト効率 | P30 |
| 集中 vs 連合メモリ | 横断個別化・一貫性重視 | データ越境制約・ソース権限尊重 | P12 |
| 楽観 vs 悲観ロック | 競合稀・スループット重視 | 同一リソースに複数エージェント集中 | P15 |
| 汎用大モデル+プロンプト vs 小型FT | 多様・低頻度・変化が速い | 狭く反復・大量・コスト/レイテンシ厳しい | P19 |
| ストリーミング vs バッチ出力 | 対話的・体感速度・長文 | 構造化結果・後続処理が全文を要する | P28 |
| 集中IDブローカー vs 連合認証 | 一貫した権限統制 | 各システムの自律・既存セッション流用 | P08 |
| 自前構築 vs プラットフォーム購入 | 差別化の中核・特殊要件 | 速度・標準機能・運用負荷低減 | P03 |
| LLM-judge vs 人手評価 | 高速・大量・回帰検知 | 最終品質・安全性判定 | P26 |
| HITL vs HOTL vs 全自動 | 高リスク・初期 | 中リスク・実績後 / 低リスク・可逆 | P18 |
| 強整合 vs 結果整合(メモリ) | 同一レコードへの即時反映 | 横断集約・可用性優先 | P12 |
