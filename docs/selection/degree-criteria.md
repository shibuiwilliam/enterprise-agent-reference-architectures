# 「程度（チューニング）」の選定基準 ― ルックアップ表

設計プロトコル STEP5 で参照する。各「程度」は **初期値 ＋ 計測指標 ＋ 調整方針** の3点で
提案に書き、実測で調整する前提を残す。

| 程度の対象 | 緩める方向 | 締める方向 | 主な判断軸 | 関連 |
|---|---|---|---|---|
| ログ詳細度 | デバッグ/監査容易 | コスト/容量節約 | エラー/低評価は全量、正常系はサンプリング。本文はストレージ、メタデータはログ基盤 | P25 |
| タイムアウト | 完了率向上 | リソース/離脱抑制 | P99の2〜3倍を起点に実測。副作用ありは慎重 | P23 |
| リトライ回数 | 不安定性吸収 | コスト/重複副作用抑制 | 冪等な読み取りは多め、副作用は冪等キー前提で控えめ | P23 |
| コンテキスト量 | グラウンディング向上 | コスト/精度/遅延 | リランキング＋トークン予算で動的最適化 | P13 |
| メモリ保持 | 継続性/個別化 | プライバシー/汚染/コスト | 重要度スコア＋要約＋TTL/減衰 | P12 |
| ガードレール強度 | 安全 | 誤検知による業務阻害 | リスク領域は強、低リスクは緩。誤検知率を計測 | P01,P10 |
| HITL対象範囲 | 安全/説明責任 | スループット/形骸化回避 | 影響度×可逆性×実績精度で対象を絞り段階自動化 | P18 |
| 自律レベル | 効率/省人化 | 事故抑制 | 可逆性と影響度。下書き→自動の段階昇格 | P07,P17 |
| モデルサイズ | 品質 | コスト/速度 | 難易度ルーティング | P16 |
| キャッシュ類似閾値 | ヒット率/コスト減 | 誤ヒット防止 | 機密/個別化は厳格、公開FAQは緩め | P24 |
| ツール露出数 | 機能網羅 | 選択精度 | 30超ならtool RAG or サブエージェント分割 | P04 |
| eval網羅度 | 品質保証 | 開発速度/コスト | リスクの高い経路を厚く、定型は薄く | P26 |
| temperature | 多様/創造的 | 再現性/決定論 | 事実/抽出/分類は低め(0付近)、発想/文章は高め | P19,P20 |
| 棄権閾値 | 自動解決率↑ | 誤答↓ | 誤答コストが高いほど棄権寄り | P21 |
| 投機実行の積極度 | 体感速度↑ | コスト↓ | 命中率×レイテンシ価値 vs 無駄コスト | P28 |
| コンテキスト読込 | 先読みで速い | 無駄取得回避 | 利用確率が高い情報のみ先読み | P13,P28 |
| フィードバック収集量 | 改善材料↑ | レビュー負荷↓ | 低評価/高コスト/新パターンを優先抽出 | P40 |
| 知識鮮度(TTL) | 鮮度↑ | 計算/コスト↓ | 変動速度と誤情報コスト。削除/権限は即時 | P31 |
| データ主権の厳格度 | 利便/選択肢↑ | 規制順守 | データ分類と居住地域で分岐 | P38 |
| プロンプト変更の統制 | 開発速度↑ | 再現性/安全↑ | 本番影響度。顧客向けは厳格、内部実験は緩い | P37 |
