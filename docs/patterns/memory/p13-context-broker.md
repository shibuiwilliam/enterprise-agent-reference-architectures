# P13 · CONTEXT-BROKER — コンテキスト・ブローカー（動的組み立て）

!!! abstract "メタ"
    **カテゴリ**: メモリ ／ **対象**: 両方 ／ **成熟度**: 高度 ／ **関連**: [P12](p12-layered-memory.md), [P14](p14-semantic-layer.md), [P24](../reliability/p24-semantic-cache.md)

**トリガ（採用検討の条件）**: 情報源が多様で文脈窓のコスト・精度が課題になっている。

## 概要

エージェント起動時に「いま必要なコンテキストだけ」を複数ソースから動的に組み立てる専用サービス。意図に応じて関連メモリ・文書・属性・組織情報を取得し、トークン予算内で優先度付け・要約・圧縮を行うことで、文脈窓の効率を最大化する。

## 設計

- ユーザーの意図（分類結果）に基づき、関連するメモリ層（[P12](p12-layered-memory.md)）・ドキュメント・ユーザー属性・組織情報（[P14](p14-semantic-layer.md)）を取得する。
- トークン予算を設定し、優先度に応じて情報を選択・要約・圧縮。lost in the middle（中間に配置された情報が無視される問題）を避ける配置を行う。
- リランキングで関連度を再評価し、低関連情報を除外する。
- キャッシュ可能な共通コンテキスト（[P24](../reliability/p24-semantic-cache.md)）は再利用する。

## 解決する課題

- **文脈窓の無駄遣い**: 関連の低い情報で文脈窓を消費し、コストが膨張する。
- **低関連情報による精度劣化**: 大量のコンテキストを投入すると、かえって回答精度が下がる。
- **コスト膨張**: トークン量に比例するコストが、情報量の増加とともに制御不能になる。

## ユースケース

- Jira のプロジェクト管理エージェントが、ユーザーの質問に応じて関連チケット・Confluenceドキュメント・Slackの過去議論・チームメンバー情報を動的に収集し、トークン予算8,000トークンを目安に最も関連の高い情報だけを組み立てる。
- Salesforce の営業エージェントが商談コンテキストを構築する際、顧客情報・過去の対応履歴・競合情報・製品ドキュメントから、いま必要な情報のみをリランキングで選択する。

## 向き / 不向き

- **向き**: 複雑なエージェント、情報源が多い業務、コンテキスト最適化が品質に直結する場面。
- **不向き**: 固定の少量コンテキストで足りる用途（ブローカーのオーバーヘッドが見合わない）。

## 要素技術

リランキング（Cohere Rerank / cross-encoder）、トークン予算管理、コンテキスト圧縮、構造化テンプレート。

!!! note "相反する仕組みの選定（知識供給方式）"
    頻繁に変化する大量データ → RAG＋ツール呼び出し。形式や専門知識の習得 → ファインチューニング。小さく完結する情報 → 長文脈への直接投入。リアルタイムの真実が必要 → ツール呼び出し（API経由）。実務ではこれらを組み合わせ、コンテキストブローカーが最適な供給方式を選択する。
