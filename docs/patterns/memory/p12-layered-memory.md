# P12 · LAYERED-MEMORY — 階層化メモリ＋コンテキスト・ブローカー

!!! abstract "一言で"
    エージェントのメモリを寿命と粒度の異なる4層に分離し、さらに起動時に「いま必要なコンテキストだけ」を動的に組み立てるパターンです。ワーキング（現セッション・揮発）/エピソード（過去要約・ユーザー単位）/セマンティック（RAG対象の知識）/組織知識（人・部署・プロジェクトの知識グラフ）の4層に適切な保存先・TTL・ACLを割り当て、コンテキスト・ブローカーが意図に応じて各層から情報を取得・トークン予算内で優先度付け・要約・圧縮を行い、文脈窓の効率を最大化します。

## 解決する問題

- **跨ぎのデータ受け渡し**: セッション間・エージェント間で文脈が失われ、毎回ゼロからのやり直しになります。
- **文脈窓の有限性**: 全履歴を文脈窓に入れるとコスト・精度が劣化します。層分離で必要な情報だけを取得します。
- **記憶肥大**: 無制限に蓄積するとコスト・プライバシーリスク・文脈汚染が増大します。
- **文脈窓の無駄遣い**: 関連の低い情報で文脈窓を消費し、コストが膨張します。
- **低関連情報による精度劣化**: 大量のコンテキストを投入すると、かえって回答精度が下がります（lost in the middle）。

## 選定条件

**採用する場合（When to use）**

- セッション跨ぎ・エージェント跨ぎでデータを引き継ぐ必要がある／継続的な業務支援を行います。
- 情報源が多様で文脈窓のコスト・精度が課題になっています。
- 継続支援・長期伴走型のエージェント、コンテキスト最適化が品質に直結する業務です。

**採用しない場合（When NOT）**

- 一発完結のステートレスタスク（メモリ管理のオーバーヘッドが見合いません）。
- 固定の少量コンテキストで足りる用途（ブローカーの動的組み立てが不要です）。

## 構造

### 4層メモリアーキテクチャ

- **ワーキングメモリ**: 現セッションの会話履歴・中間結果です。セッション終了で揮発します。
- **エピソードメモリ**: 過去セッションの要約・重要イベントです。ユーザー単位で保持し、跨ぎ引き継ぎの基盤となります。
- **セマンティックメモリ**: RAG対象のドキュメント・ナレッジです。ベクタDB等に格納します。
- **組織知識**: 人・部署・プロジェクト・権限の関係を知識グラフで保持します（[P14](p14-semantic-layer.md)）。
- 各層に別の保存先・TTL・ACL（[P10](../identity/p10-context-firewall.md)）を設定します。書き込みは要約・重要度フィルタを通し、不要な情報の蓄積を防ぎます。

### コンテキスト・ブローカー（動的組み立て）

- ユーザーの意図（分類結果）に基づき、関連するメモリ層・ドキュメント・ユーザー属性・組織情報（[P14](p14-semantic-layer.md)）を取得します。
- トークン予算を設定し、優先度に応じて情報を選択・要約・圧縮します。lost in the middle（中間に配置された情報が無視される問題）を避ける配置を行います。
- リランキングで関連度を再評価し、低関連情報を除外します。
- キャッシュ可能な共通コンテキスト（[P24](../reliability/p24-semantic-cache.md)）は再利用します。

## 駆動変数とチューニング（程度）

全記憶を保持すると文脈汚染・コスト増・プライバシーリスクが増大します。重要度スコア＋定期要約＋忘却（TTL/時間減衰）の3つで制御します。エピソードメモリは生ログではなく要約を残します。保持量の起点として「直近N回の要約＋重要イベントのみ」を設定し、ユーザーの利用パターンとコストを実測して調整します。

## 相反における立ち位置

!!! note "相反する仕組みの選定（知識供給方式）"
    頻繁に変化する大量データ → RAG＋ツール呼び出し。形式や専門知識の習得 → ファインチューニング。小さく完結する情報 → 長文脈への直接投入。リアルタイムの真実が必要 → ツール呼び出し（API経由）。実務ではこれらを組み合わせ、コンテキストブローカーが最適な供給方式を選択します。

## ユースケース

- Salesforce の営業エージェントが、過去の商談対応履歴（エピソード）と社内ナレッジ（セマンティック）を組み合わせて、現在の商談（ワーキング）に最適な提案を生成します。
- ServiceNow のITサポートエージェントが、ユーザーの過去の問い合わせ傾向（エピソード）を参照し、類似インシデントの解決策（セマンティック）を優先的に提示します。
- Jira のプロジェクト管理エージェントが、ユーザーの質問に応じて関連チケット・Confluenceドキュメント・Slackの過去議論・チームメンバー情報を動的に収集し、トークン予算8,000トークンを目安に最も関連の高い情報だけを組み立てます。（ブローカー）
- Salesforce の営業エージェントが商談コンテキストを構築する際、顧客情報・過去の対応履歴・競合情報・製品ドキュメントから、いま必要な情報のみをリランキングで選択します。（ブローカー）

## 実装メモ

- **メモリ層**: ベクタDB（Pinecone / Weaviate / pgvector）、要約パイプライン、知識グラフ（Neo4j）、Mem0/Zep、ACLタグ付け。
- **コンテキストブローカー**: リランキング（Cohere Rerank / cross-encoder）、トークン予算管理、コンテキスト圧縮、構造化テンプレート。
