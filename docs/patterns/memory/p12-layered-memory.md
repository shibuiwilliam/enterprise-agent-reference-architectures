# P12 · LAYERED-MEMORY — 階層化メモリ

!!! abstract "一言で"
    寿命と粒度の異なる4層にメモリを分離するパターン。ワーキング（現セッション・揮発）/エピソード（過去要約・ユーザー単位）/セマンティック（RAG対象の知識）/組織知識（人・部署・プロジェクトの知識グラフ）の4層を持ち、各層に適切な保存先・TTL・ACLを割り当てる。

## 解決する問題

- **跨ぎのデータ受け渡し**: セッション間・エージェント間で文脈が失われ、毎回ゼロからのやり直しになる。
- **文脈窓の有限性**: 全履歴を文脈窓に入れるとコスト・精度が劣化する。層分離で必要な情報だけを取得。
- **記憶肥大**: 無制限に蓄積するとコスト・プライバシーリスク・文脈汚染が増大する。

## 選定条件

**採用する場合（When to use）**

- セッション跨ぎ・エージェント跨ぎでデータを引き継ぐ必要がある／継続的な業務支援を行う。
- 継続支援・長期伴走型のエージェント、セッション跨ぎ・エージェント跨ぎのデータ引き継ぎが必要な業務。

**採用しない場合（When NOT）**

- 一発完結のステートレスタスク（メモリ管理のオーバーヘッドが見合わない）。

## 構造

- **ワーキングメモリ**: 現セッションの会話履歴・中間結果。セッション終了で揮発。
- **エピソードメモリ**: 過去セッションの要約・重要イベント。ユーザー単位で保持し、跨ぎ引き継ぎの基盤。
- **セマンティックメモリ**: RAG対象のドキュメント・ナレッジ。ベクタDB等に格納。
- **組織知識**: 人・部署・プロジェクト・権限の関係を知識グラフで保持（[P14](p14-semantic-layer.md)）。
- 各層に別の保存先・TTL・ACL（[P10](../identity/p10-context-firewall.md)）を設定。書き込みは要約・重要度フィルタを通し、不要な情報の蓄積を防ぐ。

## 駆動変数とチューニング（程度）

全記憶を保持すると文脈汚染・コスト増・プライバシーリスクが増大する。重要度スコア＋定期要約＋忘却（TTL/時間減衰）の3つで制御する。エピソードメモリは生ログではなく要約を残す。保持量の起点として「直近N回の要約＋重要イベントのみ」を設定し、ユーザーの利用パターンとコストを実測して調整する。

## ユースケース

- Salesforce の営業エージェントが、過去の商談対応履歴（エピソード）と社内ナレッジ（セマンティック）を組み合わせて、現在の商談（ワーキング）に最適な提案を生成する。
- ServiceNow のITサポートエージェントが、ユーザーの過去の問い合わせ傾向（エピソード）を参照し、類似インシデントの解決策（セマンティック）を優先的に提示する。

## 実装メモ

ベクタDB（Pinecone / Weaviate / pgvector）、要約パイプライン、知識グラフ（Neo4j）、Mem0/Zep、ACLタグ付け。
