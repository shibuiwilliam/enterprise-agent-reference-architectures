# P35 · AGENT-REGISTRY — エージェント・レジストリ / 社内マーケットプレイス

!!! abstract "一言で"
    社内のエージェント・ツール・スキル・プロンプトをカタログ化し、発見・再利用・ガバナンスを実現するパターン。各エージェントにメタデータ（所有者/対象部署/権限/eval結果/コスト/SLA）を付与し、社内マーケットプレイスとして管理する。

## 解決する問題

- **エージェントの乱立・重複開発**: 各チームが類似のエージェントを個別に開発し、リソースが分散する。
- **品質のばらつき**: eval未実施・テスト不十分なエージェントが本番に入る。
- **所有者不在**: 開発者の異動後にメンテナンスされないエージェントが残る。
- **ガバナンスの欠如**: どのエージェントが何の権限でどのデータにアクセスしているか把握できない。

## 選定条件

**採用する場合（When to use）**

- エージェントが乱立し始める／重複開発・所有者不在が課題になっている。
- 全社にエージェント開発が広がるフェーズ、エージェント数が増加し管理が課題になっている組織。

**採用しない場合（When NOT）**

- エージェントが数個の初期段階（レジストリのオーバーヘッドが見合わない）。

## 構造

- 各エージェントにメタデータ（所有者・対象部署・必要権限・eval結果・コスト実績・SLA・バージョン）を登録。
- 利用申請・承認フローを設け、ガバナンスを確保する。
- バージョン管理を行い、変更履歴を追跡可能にする。
- 廃止管理（非推奨化→依存者への通知→停止）のライフサイクルを組み込む。

## ユースケース

- ServiceNow / Backstage 的なサービスカタログに全社のエージェントを登録し、利用者が「営業支援」「チケット分類」等のカテゴリからエージェントを検索・利用申請する。
- 新規エージェントの公開前にeval gate（[P26](../reliability/p26-eval-gate-canary.md)）の通過を必須とし、品質基準を満たさないエージェントの本番投入を防ぐ。

## 実装メモ

サービスカタログ（ServiceNow / Backstage 的）、バージョニング、メタデータ管理、承認ワークフロー。
