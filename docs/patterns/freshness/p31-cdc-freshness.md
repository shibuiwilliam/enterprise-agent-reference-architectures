# P31 · CDC-FRESHNESS — CDC駆動の知識鮮度管理

!!! abstract "一言で"
    ソースシステム（Salesforce / Notion / Box 等）の変更をCDC（Change Data Capture）で捕捉し、RAG索引・キャッシュ・メモリを増分更新/無効化するパターン。古い知識に基づく誤回答を防ぎ、エージェントが常に最新の情報を参照できる状態を維持する。

## 解決する問題

- **陳腐化した知識**: 更新されたポリシーや価格を古い情報のまま回答する。
- **キャッシュの古さ**: セマンティックキャッシュに古い回答が残り、誤情報を返す。
- **削除済み文書の参照**: 削除または非公開になった文書がRAG索引に残り、参照される。

## 選定条件

**採用する場合（When to use）**

- 頻繁に更新される知識（ポリシー/価格/在庫/ドキュメント）を扱う。
- 頻繁に更新される知識を扱うエージェント（ポリシー・価格・在庫・ドキュメント）。

**採用しない場合（When NOT）**

- 静的・更新が稀な情報のみを扱う用途。

## 構造

- ソースのWebhook/CDCイベントを監視し、変更があった該当チャンクのみを再索引（増分更新）する。
- キャッシュ（[P24](../reliability/p24-semantic-cache.md)）の該当エントリを即時無効化する。
- 鮮度要件が高いデータはTTLを短縮し、更新時に即時無効化する。
- 最終更新時刻をメタデータとして保持し、回答に鮮度を明示できるようにする。
- **削除・権限剥奪の伝播は最優先**で処理する。

## 駆動変数とチューニング（程度）

削除・権限剥奪の伝播は最優先。退職者や権限喪失後にデータがRAG索引に残ると漏洩源になるため、即時反映が必須。それ以外のデータは変動速度と誤情報コストに応じてTTLを設定する。価格・在庫など変動が速く誤情報コストが高いデータは短TTL、参考情報的なドキュメントは長めのTTLで運用する。

## ユースケース

- Notion上の社内ポリシーが更新された際、Webhookで変更を検知し、該当チャンクを再ベクタ化してRAG索引を更新。キャッシュの関連エントリも無効化する。
- Salesforceの価格表が変更された際、CDCイベントで営業エージェントが参照する価格情報を即時更新する。

## 実装メモ

CDC（Debezium）、SaaS Webhook（Salesforce Platform Events / Notion Webhook）、増分索引パイプライン、キャッシュ無効化、削除伝播。
