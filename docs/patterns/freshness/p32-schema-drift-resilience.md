# P32 · SCHEMA-DRIFT-RESILIENCE — スキーマ・契約ドリフト耐性

!!! abstract "一言で"
    依存するAPIやSaaSのスキーマ変更（フィールド追加/廃止/型変更）を自動検知し、壊れる前に防御するパターン。SaaSのサイレントな仕様変更でエージェントが暗黙に破綻する問題を、契約テストとドリフト検知で構造的に解消する。

## 解決する問題

- **SaaSのサイレント仕様変更による暗黙の破綻**: APIのレスポンス形式が変わり、エージェントが誤ったデータを処理する。
- **ツール定義と実APIの乖離**: ツール定義が古いままで、実際のAPIと齟齬が生じる。

## 選定条件

**採用する場合（When to use）**

- 多数の外部API/SaaSに依存する。
- 多数の外部API/SaaSに依存するエージェント環境。

**採用しない場合（When NOT）**

- 依存APIが固定・安定していて変更頻度が極めて低い環境。

## 構造

- ツール定義/APIスキーマを「契約（versioned contract）」として固定し、バージョン管理する。
- 定期的に実際のAPIスキーマと契約を突合し、差分（ドリフト）を検知する。
- 後方互換のないドリフト（必須フィールド削除・型変更等）を検出した場合は、アラート＋該当ツールの一時無効化で安全側に倒す。
- MCPツール定義（[P04](../integration/p04-mcp-gateway.md)）の検証にも適用する。

## ユースケース

- Salesforce APIのフィールド構成変更を週次の契約テストで検知し、影響を受けるMCPツールを一時停止。開発チームにアラートを送信し、ツール定義の更新を促す。
- Shopify APIのバージョンアップで応答フォーマットが変わった際、ドリフト検知が差分を検出し、注文処理エージェントの該当ツールを安全に無効化する。

## 実装メモ

契約テスト（Pact等）、スキーマレジストリ、ドリフト検知パイプライン、MCPツール定義検証。
