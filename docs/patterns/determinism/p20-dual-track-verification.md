# P20 · DUAL-TRACK-VERIFICATION — 二重トラック検証

!!! abstract "一言で"
    確率的なエージェント出力を、独立した安価・決定論的な検証器でクロスチェックするパターンです。エージェントの「生成」と別トラックの「検証」を分離し、ハルシネーション・計算ミス・捏造引用を実行前に検出します。

## 解決する問題

- **ハルシネーション**: 事実と異なる情報が生成されます。
- **根拠なき主張**: 出典が示されない、または捏造された引用が含まれます。
- **計算ミス**: LLMが数値計算を誤ることがあります。
- **捏造引用**: 存在しない文書・条項・判例が引用されます。

## 選定条件

**採用する場合（When to use）**

- 数値・事実・引用の正確性が重要な業務（財務/法務/分析/サポート）。
- 事実・数値・引用の正確性が重要な業務（財務報告・法務助言・分析・サポート回答）。

**採用しない場合（When NOT）**

- 創造的・主観的な出力で検証基準が定義できない領域（ブレインストーミング・文章作成等）。

## 構造

- **数値検証**: コードが信頼ソース（DB/API）から再計算し、エージェント出力との一致を確認します。
- **引用検証（grounding）**: 引用が実在すること、引用箇所が主張と一致することをプログラムで照合します。
- **アクション検証**: 出力されたアクションパラメータをスキーマ＋ビジネスルールで事前検証します。
- 不一致が検出された場合は、棄却/再試行/人間へのエスカレーション（[P21](p21-confidence-gated-abstention.md)）のいずれかで対応します。

## 相反における立ち位置

不可逆な副作用を伴う操作 → **事前検証**（実行前に止めます）。情報提供のみの操作 → **事後検証**（生成後にチェック・注記を付与します）。事前検証はレイテンシが増えますが安全性が高く、事後検証は速度を保ちつつ品質を補強します。業務の可逆性に応じて使い分けてください。

## ユースケース

- Salesforceの営業エージェントが「先月の売上は1,200万円」と回答した際、検証トラックがSalesforce APIから実データを取得し再計算します。不一致なら回答を差し止め、正しい数値で再生成します。
- 法務エージェントが契約条項を引用して助言する際、引用照合器が該当文書の実在と引用箇所の正確性をプログラムで検証します。引用が見つからない場合は「確認が必要」と注記します。

## 実装メモ

grounding / 引用照合、ルールエンジン、JSONスキーマ検証、決定論計算との突合。
