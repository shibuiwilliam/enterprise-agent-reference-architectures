# P40 · FEEDBACK-FLYWHEEL — フィードバックループ・フライホイール

!!! abstract "一言で"
    本番のユーザー評価（👍/👎・修正・エスカレーション・成否）を体系的に収集し、evalデータセット化・改善・ルーティング最適化へ還流するパターン。一度デプロイして終わりではなく、運用データを継続的に改善サイクルへ回すフライホイールを構築する。

## 解決する問題

- **本番品質の継続改善**: デプロイ後の品質改善サイクルが回らない。
- **劣化の早期発見**: 品質劣化（モデル変更・データ変化・新パターンの出現）を検知できない。
- **改善の優先順位付け**: どの問題から改善すべきか判断するデータがない。

## 選定条件

**採用する場合（When to use）**

- 継続運用するすべての本番エージェント。

**採用しない場合（When NOT）**

- 使い捨ての実験・PoC。

## 構造

- **暗黙シグナル**（やり直し・離脱・コピー・修正）＋**明示シグナル**（👍/👎・テキストフィードバック・エスカレーション）を収集する。
- 低評価・エスカレーション事例をeval データセット化し、回帰テストに組み込む。
- プロンプト・ツール・ルーティング・RAGの改善に活用し、eval gate（[P26](../reliability/p26-eval-gate-canary.md)）で回帰検証してから再デプロイする。
- 改善の優先順位は、低評価の頻度×影響度で付ける。

## 駆動変数とチューニング（程度）

全件を人手でレビューするのは不可能。低評価・高コスト・エスカレーション・新パターン（未知のカテゴリ）を優先サンプリングし、正常系は薄く抽出する。収集量の起点として「低評価は全件、正常系は5〜10%」を設定し、レビュー負荷と改善効果を実測して調整する。

## ユースケース

- Zendesk の顧客対応エージェントで、ユーザーの👎評価や有人エスカレーション事例を自動収集し、月次で低評価パターンを分析。上位の問題パターンに対してプロンプト改善→eval gate 通過→再デプロイのサイクルを回す。
- 社内ヘルプデスクエージェントで、回答後の「もう一度聞き直す」操作を暗黙シグナルとして検知し、回答品質の低いトピックを特定してナレッジベースを拡充する。

## 実装メモ

フィードバック収集（アプリ内UI・Slack リアクション）、evalデータセット管理、回帰テスト、LLM-as-judge＋人手評価、A/Bテスト。
