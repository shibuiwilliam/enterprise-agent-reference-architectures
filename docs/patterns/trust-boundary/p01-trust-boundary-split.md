# P01 · TRUST-BOUNDARY-SPLIT — 信頼境界の二層分離

!!! abstract "一言で"
    従業員向けと顧客向けで、ネットワーク・データ・モデル・ガードレール・監査の各層を物理的/論理的に
    分離するパターン。オーケストレーション基盤を共有しても、顧客向けは「内部システムへ直接到達できない」
    隔離プレーンに置く。従業員向けと顧客向けは信頼レベル・データ境界・失敗コスト・規模特性が根本的に
    異なるため、同一スタックで一括設計してはならない。

## 解決する問題

- 顧客向け経路からの内部情報漏洩（最も致命的なリスク）。
- 敵対的入力（ジェイルブレイク・間接インジェクション）による暴走。
- 従業員向けの緩い前提を顧客向けにそのまま適用してしまう設計事故。

## 選定条件

**採用する場合（When to use）**

- 利用者に顧客と従業員の**両方**が含まれる／顧客向けと社内向けが同一基盤を共有しようとしている。
- B2C / B2B 両面を持ち、顧客接点と社内業務の双方でエージェントを使う企業。

**採用しない場合（When NOT）**

- 純粋に社内専用の用途（過剰な分離になりコストだけ増える）。

## 構造

| 観点 | 従業員向け | 顧客向け |
|---|---|---|
| 信頼レベル | 高い（雇用・教育・監査） | 低い（敵対的入力前提・乱用） |
| データ境界 | 社内データへ広く（権限内） | 自テナント/自顧客のみ。内部漏洩は致命 |
| 失敗コスト | 業務非効率（多くは可逆） | ブランド毀損・法的責任（不可逆化しやすい） |
| 規模 | 数千〜数万・業務時間集中 | 数万〜数百万・スパイク・24/365 |
| 認証 | 社内IdP（Okta/Entra）SSO | 顧客IdP（Auth0/CIAM）・匿名も |
| KPI | 生産性・処理時間 | CSAT・解決率・封じ込め率 |

- 顧客向けエージェントは DMZ 相当の隔離環境に置く。内部データは「公開可能と明示した投影
  （read model）」のみ参照可能とする。
- 顧客向けの出力は、内部由来情報のリーク検査（DLP）を必須通過させる。
- ガードレールは顧客向けが格段に厳格（ジェイルブレイク検知・トピック制限・トーン制御・拒否方針）。
- 監査・トレースは顧客識別子と紐付けるが、PIIは分類別の保持期間で管理する。
- 共有してよいのはオーケストレーション基盤やモデルゲートウェイ（[P03](../integration/p03-ai-gateway.md)）
  までで、データ到達経路は分離する。

## 相反における立ち位置

社内・高信頼・広いデータ権限が前提なら従業員向けプレーン。外部・敵対前提・隔離・厳格ガードレールが
必要なら顧客向けプレーン。**どちらかに倒さず、両方を別プレーンとして設計する**のが本パターンの要点。
顧客向けで複数テナントが同居するなら [P02](p02-multi-tenant-isolation.md) を併用する。

## ユースケース

- Shopify 連携 EC の顧客向け問い合わせ対応エージェントと、社内の受発注オペレーション支援エージェントを
  同一プラットフォーム上で、別プレーンとして運用する。
- Zendesk 上の顧客向け一次対応エージェントが、社内 Notion / Box の機密ナレッジへ直接到達できないよう、
  公開可能ナレッジの投影だけを参照させる。

## 実装メモ

ネットワークセグメンテーション、CQRS の read model（公開可能データの投影）、DLP（データ漏洩防止）、
ジェイルブレイク/プロンプトインジェクション検知、出力分類器、顧客IdP（Auth0/CIAM）と社内IdP（Okta/Entra）の分離。
