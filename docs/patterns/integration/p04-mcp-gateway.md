# P04 · MCP-GATEWAY — MCPゲートウェイ / ツール・フェデレーション

!!! abstract "一言で"
    エージェントのツール（外部システム操作）をMCP（Model Context Protocol）等の統一規約で集約・仲介するパターンです。各SaaSコネクタをMCPサーバとして束ね、エージェントに統一ツールカタログを提示することで、N（エージェント）×M（SaaS）の統合爆発を解消します。

## 解決する問題

- **ツール定義の乱立**: 各エージェントが独自にツール定義を持ち、重複・不整合が発生します。
- **N×M統合の爆発**: エージェント数×SaaS数の統合コストが指数的に増大します。
- **ツール経由インジェクション**: ツールの入出力を経由した間接プロンプトインジェクション。
- **過剰露出による選択精度劣化**: エージェントに見えるツールが多すぎると、適切なツール選択の精度が低下します。
- **SaaSのサイレント仕様変更による暗黙の破綻**: APIのレスポンス形式が変わり、エージェントが誤ったデータを処理します。
- **ツール定義と実APIの乖離**: ツール定義が古いままで、実際のAPIと齟齬が生じます。

## 選定条件

**採用する場合（When to use）**

- 連携SaaSが10種以上／複数エージェントが共通ツールを使う／N×M統合の複雑さに困っている。
- 多SaaS環境で複数エージェントがツールを共有する組織。
- 多数の外部API/SaaSに依存するエージェント環境で、スキーマ変更による破綻リスクがある。

**採用しない場合（When NOT）**

- ツールが2〜3個固定の単機能エージェント（直結のほうが堅牢でシンプルです）。
- 依存APIが固定・安定していて変更頻度が極めて低い環境（ドリフト検知のオーバーヘッドが不要です）。

## 構造

- システムごとにMCPサーバ（公式または自作）を用意し、ゲートウェイがツールの発見・認可・呼び出し監査・スコープ制御を担います。
- ツール許可リストを主体（部署×エージェント種別）で動的に絞り、エージェントに見えるツールを必要最小限にします。
- 危険ツール（削除・送金・外部送信など不可逆操作）には承認フック（[P09](../identity/p09-dynamic-authz-pdp.md) と連携）を設置します。

### スキーマ・契約ドリフト耐性

ツール定義/APIスキーマを「契約（versioned contract）」として固定し、バージョン管理します。定期的に実際のAPIスキーマと契約を突合し、差分（ドリフト）を検知します。後方互換のないドリフト（必須フィールド削除・型変更等）を検出した場合は、アラート＋該当ツールの一時無効化で安全側に倒します。MCPツール定義の検証にもこのドリフト検知を適用し、SaaS側の仕様変更による暗黙の破綻を防ぎます。

## 駆動変数とチューニング（程度）

エージェントに露出するツールが20〜30を超えると選択精度が低下する目安があります。tool RAG で意図に応じて動的に絞るか、役割別サブエージェントに分割することを検討してください。最適な露出数はタスクの複雑度とモデル性能に依存するため、実測でツール選択の正答率を計測し調整します。

## ユースケース

- Salesforce・ServiceNow・Jira・Slack・Box・Workday の各MCPサーバをゲートウェイ配下に束ね、営業エージェント・ITエージェント・人事エージェントがそれぞれ必要なツールだけを利用する構成です。
- 共通のMCPゲートウェイを通じて、顧客対応エージェントがZendesk・Shopify・社内ナレッジを横断操作する際に、統一的な認可・監査を適用します。
- Salesforce APIのフィールド構成変更を週次の契約テストで検知し、影響を受けるMCPツールを一時停止します。開発チームにアラートを送信し、ツール定義の更新を促します。
- Shopify APIのバージョンアップで応答フォーマットが変わった際、ドリフト検知が差分を検出し、注文処理エージェントの該当ツールを安全に無効化します。

## 実装メモ

MCP（Model Context Protocol）、OpenAPI→ツール自動生成、OAuth 2.1、ツールレジストリ、契約テスト（Pact等）、スキーマレジストリ、ドリフト検知パイプライン、MCPツール定義検証。
