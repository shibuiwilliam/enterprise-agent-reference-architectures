# P04 · MCP-GATEWAY — MCPゲートウェイ / ツール・フェデレーション

!!! abstract "一言で"
    エージェントのツール（外部システム操作）をMCP（Model Context Protocol）等の統一規約で集約・仲介するパターン。各SaaSコネクタをMCPサーバとして束ね、エージェントに統一ツールカタログを提示することで、N（エージェント）×M（SaaS）の統合爆発を解消する。

## 解決する問題

- **ツール定義の乱立**: 各エージェントが独自にツール定義を持ち、重複・不整合が発生する。
- **N×M統合の爆発**: エージェント数×SaaS数の統合コストが指数的に増大する。
- **ツール経由インジェクション**: ツールの入出力を経由した間接プロンプトインジェクション。
- **過剰露出による選択精度劣化**: エージェントに見えるツールが多すぎると、適切なツール選択の精度が低下する。

## 選定条件

**採用する場合（When to use）**

- 連携SaaSが10種以上／複数エージェントが共通ツールを使う／N×M統合の複雑さに困っている。
- 多SaaS環境で複数エージェントがツールを共有する組織。

**採用しない場合（When NOT）**

- ツールが2〜3個固定の単機能エージェント（直結のほうが堅牢でシンプル）。

## 構造

- システムごとにMCPサーバ（公式または自作）を用意し、ゲートウェイがツールの発見・認可・呼び出し監査・スコープ制御を担う。
- ツール許可リストを主体（部署×エージェント種別）で動的に絞り、エージェントに見えるツールを必要最小限にする。
- 危険ツール（削除・送金・外部送信など不可逆操作）には承認フック（[P09](../identity/p09-dynamic-authz-pdp.md) と連携）を設置。
- ツール定義のバージョニングとドリフト検知（[P32](../freshness/p32-schema-drift-resilience.md)）を組み込み、SaaS側の仕様変更による暗黙の破綻を防ぐ。

## 駆動変数とチューニング（程度）

エージェントに露出するツールが20〜30を超えると選択精度が低下する目安がある。tool RAG で意図に応じて動的に絞るか、役割別サブエージェントに分割することを検討する。最適な露出数はタスクの複雑度とモデル性能に依存するため、実測でツール選択の正答率を計測し調整する。

## ユースケース

- Salesforce・ServiceNow・Jira・Slack・Box・Workday の各MCPサーバをゲートウェイ配下に束ね、営業エージェント・ITエージェント・人事エージェントがそれぞれ必要なツールだけを利用する構成。
- 共通のMCPゲートウェイを通じて、顧客対応エージェントがZendesk・Shopify・社内ナレッジを横断操作する際に、統一的な認可・監査を適用する。

## 実装メモ

MCP（Model Context Protocol）、OpenAPI→ツール自動生成、OAuth 2.1、ツールレジストリ。
