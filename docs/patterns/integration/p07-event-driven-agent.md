# P07 · EVENT-DRIVEN-AGENT — イベント駆動エージェント（アンビエント）

!!! abstract "一言で"
    業務イベント（新規リード作成、SLA違反間近、PR作成、契約期限到来など）をトリガにエージェントが自律起動するパターンです。ユーザーの明示的な指示を待たず、イベント発生時点で自動的に対応を開始します。常時稼働・予防的なAI支援を実現します。

## 解決する問題

- **気づく前の予防対応**: SLA違反・異常検知などを人が気づく前にエージェントが対応を開始します。
- **常時カバレッジ（24/365）**: 人間不在の時間帯でも一次対応が途絶えません。
- **一次対応の自動化**: 定型的なトリアージ・分類・初動を自動化し、人間は判断・承認に集中します。

## 選定条件

**採用する場合（When to use）**

- 人の指示前に予防対応したい／24/365カバレッジが必要／定型的な一次対応を自動化したい。
- イベント量が多く一次対応が定型化可能な業務、予防的な対応が価値をもたらす領域。

**採用しない場合（When NOT）**

- 誤起動コストが極端に高い領域（不可逆な操作を即座に実行する構成は避けます）。

## 構造

- イベントバス（Kafka / EventBridge / Webhook 等）を購読し、条件評価を経てエージェントを起動します。
- 処理は**必ず非同期**（[P22](../reliability/p22-async-job-callback.md)）で実行します。
- 冪等キーで重複起動を防止し、同一イベントによる二重処理を回避します。
- 起動後の動作は「人にレビュー依頼」または「自動実行」の段階制御とし、初期は人間承認（[P18](../orchestration/p18-hitl-approval-gate.md)）を挟みます。

## 駆動変数とチューニング（程度）

「自動で下書きまで作成し人間がレビュー（HITL）」から始め、誤り率と影響度を実測してから「自動実行（HOTL）」へ段階的に昇格させます。最初から全自動にすると事故リスクが高くなります。昇格の判断は、一定期間の精度実績（目安として誤り率・エスカレーション率）を計測し、業務オーナーの承認を経て行います。

## ユースケース

- Zendesk に新規問い合わせが到着したイベントを検知し、自動トリアージ（カテゴリ分類・優先度判定）と一次回答ドラフトを作成して担当者にレビュー依頼します。
- Salesforce で商談ステージが変わったイベントを起点に、関連情報を収集しリードの初動対応（メールドラフト・スケジュール提案）を準備します。
- 本番システムの異常検知アラートを受けて、ログ調査・影響範囲の特定・初動手順の提案を自動実行します。

## 実装メモ

Kafka、Amazon EventBridge、Webhook、Temporal / AWS Step Functions、DLQ（Dead Letter Queue）、冪等性キー。
