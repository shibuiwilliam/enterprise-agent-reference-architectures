# P30 · OFFLINE-ONLINE-SPLIT — オフライン・バッチ知性とオンライン知性の分離

!!! abstract "一言で"
    即応が必要な対話的知性（オンライン）と、時間をかけて深い分析を行う知性（オフライン/バッチ）を分離し、それぞれに最適なコスト・モデル・SLAを割り当てるパターン。「全部リアルタイム」のコスト/レイテンシ破綻を防ぐ。

## 解決する問題

- **「全部リアルタイム」のコスト/レイテンシ破綻**: すべての処理をリアルタイムで行うと、コストとレイテンシが共に制御不能になる。
- **重い分析と即応の両立**: 深い分析には時間がかかるが、ユーザーには即座に結果を返したい。

## 選定条件

**採用する場合（When to use）**

- 重い分析処理と即応の対話の両方が必要。
- 重い分析＋即応の両方が必要な業務。

**採用しない場合（When NOT）**

- どちらか一方（リアルタイムのみ or バッチのみ）で足りる場合。

## 構造

- **オンライン**: 低レイテンシ・高可用・キャッシュ重視。軽量モデルで即応。
- **オフライン**: 大規模・高精度・コスト効率重視。バッチ割引・大型モデル・多段推論を活用。夜間/低負荷時間帯に実行。
- 結果ストア（データベース / キャッシュ）を介して連携し、バッチ結果をオンラインエージェントが参照する。

## 相反における立ち位置

即時性・予測不能な質問 → オンライン。大量・重い分析・事前計算可能・コスト効率重視 → オフライン。多くの実務では「バッチで前処理＋オンラインで仕上げ」の組み合わせが最適。

## ユースケース

- 夜間バッチで全顧客の解約予兆スコアを大型モデルで計算し、結果ストアに格納。日中に営業エージェントが「解約リスクの高い顧客は？」と聞かれた際、事前計算済みの結果を即座に参照する。
- 週次で全Jiraチケットの傾向分析・技術的負債の評価をバッチ実行し、オンラインの開発支援エージェントが「最も対処すべき技術的負債は？」に即答する。

## 実装メモ

バッチ処理基盤（Airflow / Step Functions）、結果ストア（Redis / DynamoDB）、バッチ割引API、スケジューラ。
