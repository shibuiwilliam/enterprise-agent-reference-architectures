# P28 · SPECULATIVE-EXECUTION — 投機的・並列・先読み実行

!!! abstract "メタ"
    **カテゴリ**: レイテンシ ／ **対象**: 両方 ／ **成熟度**: 高度 ／ **関連**: [P30](p30-offline-online-split.md), [P13](../memory/p13-context-broker.md)

**トリガ（採用検討の条件）**: 対話的で体感速度が重要な用途。

## 概要

長セッションのレイテンシを推測実行・プリフェッチ・並列処理で体感的に隠すパターン。ユーザーの入力途中で意図を予測しコンテキストを先読みしたり、多分岐を投機的に並列実行して確定後に不要な分岐を破棄したりする。ストリーミングで思考過程を見せることで、待ち時間のUX影響を軽減する。

## 設計

- **プリフェッチ**: 入力途中で意図を予測し、必要なコンテキスト（[P13](../memory/p13-context-broker.md)）を先読み取得する。
- **投機的並列実行**: 多分岐の可能性がある場合、複数の分岐を並列実行し、確定後に不要な分岐を破棄する。
- **ストリーミング**: 思考過程・中間結果を逐次表示し、体感待ち時間を短縮する。
- **安価モデルで下書き→高性能モデルで検証**: 速度重視の初期回答を先に返し、品質検証を非同期で行う。

## 解決する課題

- **高レイテンシのUX影響**: LLMの推論時間が数秒〜数十秒に及び、対話的なUXを損なう。
- **待ち時間の体感**: 何も表示されない待ち時間がユーザーの離脱を招く。

## ユースケース

- Slack上の営業エージェントで、ユーザーが「商談の」と入力した時点で直近の商談データをプリフェッチし、「商談の状況を教えて」と確定した瞬間に即座に回答を開始する。
- ServiceNow のインシデント調査エージェントで、関連ログ・類似インシデント・ナレッジの検索を並列実行し、最も早く得られた結果からストリーミングで表示する。

## 向き / 不向き

- **向き**: 対話的・体感速度が重要な用途、ユーザーの待ち時間がKPIに影響する場面。
- **不向き**: コスト最優先の環境（投機実行は無駄コストを伴う）、予測精度が低い場面。

## 要素技術

ストリーミング応答（SSE）、プリフェッチ、並列実行、投機的実行、意図予測。

!!! warning "程度の判断（投機の積極度）"
    投機実行はレイテンシとコストの交換である。「予測命中率×レイテンシ改善の価値 > 無駄コスト」のときだけ積極化する。命中率が低い場合は無駄なトークン消費が増えるだけなので、予測精度を実測し、費用対効果が見合う範囲で適用する。
