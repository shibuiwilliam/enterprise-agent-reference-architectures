# P29 · BACKPRESSURE-LOAD-SHEDDING — バックプレッシャ / 優先度キュー

!!! abstract "メタ"
    **カテゴリ**: レイテンシ ／ **対象**: 顧客 ／ **成熟度**: 高度 ／ **関連**: [P02](../trust-boundary/p02-multi-tenant-isolation.md), [P22](../reliability/p22-async-job-callback.md)

**トリガ（採用検討の条件）**: 数万〜数百万規模の顧客向けでスパイクのある環境。

## 概要

過負荷時にリクエストを優先度に基づいて制御し、低優先度のリクエストを遅延・拒否・縮退させることで全体の安定性を守るパターン。「全リクエストを平等に処理して全体が崩壊する」のではなく、重要な処理を確実に守る。

## 設計

- **優先度キュー**: リアルタイム対話 > バックグラウンド処理の優先度付け。
- **容量超過時の制御**: バックグラウンド処理を後回し / 縮退モデルに切替 / 「混雑中」応答を即返却。
- **アドミッションコントロール**: 受付段階から過負荷を検知し、受け入れ可能な量に制限する。
- テナント別クォータ（[P02](../trust-boundary/p02-multi-tenant-isolation.md)）と連動し、特定テナントの暴走が他テナントを巻き込まないようにする。

## 解決する課題

- **スパイク時の全体崩壊**: 一時的な負荷急増で全ユーザーのレイテンシが悪化・タイムアウトする。
- **コスト暴走**: スパイク時のLLM呼び出しコストが制御不能になる。
- **重要処理の巻き添え**: 低優先度の大量バッチ処理が、リアルタイム対話の品質を損なう。

## ユースケース

- Shopify連携のECサポートエージェントで、セール開始時のアクセス急増に対し、リアルタイムの顧客対応を最優先し、バックグラウンドのレポート生成は後回しにする。
- マルチテナントの顧客対応サービスで、特定テナントの異常な大量リクエストをレート制限し、他テナントへの影響を防ぐ。

## 向き / 不向き

- **向き**: 大規模・スパイクのある顧客向けサービス、マルチテナント環境。
- **不向き**: 負荷が常に低く一定の環境（制御のオーバーヘッドが見合わない）。

## 要素技術

優先度キュー、アドミッションコントロール、レート制限（Token Bucket / Sliding Window）、プロバイダレート制限との協調。
