# P16 · SUPERVISOR-ROUTER — スーパーバイザ / ルーター

!!! abstract "一言で"
    ユーザーの意図を分類し、適切なドメインエージェント（営業/IT/人事/開発 等）へ委譲する軽量ルーター。[P06](../integration/p06-agent-hub.md) エージェント・ハブの頭脳に相当し、単一巨大プロンプトの破綻を防ぎつつ、専門性を維持するための振り分け機構。

## 解決する問題

- **単一巨大プロンプトの破綻**: すべての機能を1つのプロンプトに押し込むと品質が劣化する。
- **専門性の維持**: ドメインごとに最適化されたプロンプト・ツール・モデルを使い分けられない。
- **コスト最適化**: 全リクエストを最高性能モデルで処理するとコストが破綻する。

## 選定条件

**採用する場合（When to use）**

- 入口で意図を分類してドメインエージェントへ振り分けたい。
- 多様な業務をカバーする全社展開、ドメインエージェントが複数存在する環境。

**採用しない場合（When NOT）**

- 単一ドメインで完結するエージェント（ルーティングが不要）。

## 構造

- 安価・高速モデル（小型分類器）で意図を分類し、専門エージェントへ委譲する。
- 曖昧な入力には聞き返しを行い、意図を確定してから振り分ける。
- 委譲先に権限上限・コスト上限・タイムアウトを引き継ぎ、暴走を防止する。
- フォールバック経路を設け、分類不能な場合はデフォルトエージェントまたは人間へエスカレーションする。

## 駆動変数とチューニング（程度）

全リクエストを最高性能モデルで処理するとコストが破綻する。難易度ルーティングにより、簡単なタスク → 安価・高速モデル、複雑なタスク → 大型・高性能モデルに振り分ける。ただしルーティング誤りのコスト（簡単なタスクに大型モデルを使う無駄、複雑なタスクに小型モデルを使う品質劣化）も見込み、分類精度を実測して調整する。

## ユースケース

- Slack上の全社AIハブで「経費精算の方法を教えて」→人事/総務エージェント、「本番サーバーが遅い」→ITエージェント、「来月の営業見込みは？」→営業エージェントへ自動振り分けする。
- 部署スコープエージェント（[P33](../organization/p33-department-scoped-agent.md)）を束ねるルーターとして、組織構造に沿った委譲を行う。

## 実装メモ

小型分類器（ファインチューニング済み軽量モデル / ルールベース）、ルーティングテーブル、フォールバック経路。
