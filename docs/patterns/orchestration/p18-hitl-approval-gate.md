# P18 · HITL-APPROVAL-GATE — Human-in-the-Loop 承認ゲート

!!! abstract "一言で"
    高リスクアクションの実行前に人間の承認を挟むパターンです。エージェントは提案（何を・なぜ・どう実行するか）を作成し、人間が承認/却下/修正します。ハルシネーション・誤操作の最終防波堤であり、説明責任と規制対応の基盤となります。

## 解決する問題

- **ハルシネーション・誤操作の最終防波堤**: 自動化された処理の中で、致命的な誤りを人間が最終チェックします。
- **説明責任**: 「誰が承認したか」を記録し、意思決定の証跡を残します。
- **規制対応**: 規制上、人間の関与が義務付けられている操作に対応します。

## 選定条件

**採用する場合（When to use）**

- 金銭・契約・顧客接点・人事・本番変更など高リスク操作を含む場合。
- 高リスク操作（金銭・契約・顧客接点・人事・本番変更）がある場合。

**採用しない場合（When NOT）**

- 低リスク・大量・即時性が命の処理（承認がボトルネックとなり形骸化します）。

## 構造

- アクションをリスクスコアリング（金額・影響範囲・可逆性・データ分類など）し、閾値を超えたものを承認キューへ送ります。
- 承認通知はSlack/メール/専用UIで送信し、承認待ちの間はジョブを中断・永続化します（長時間の承認待ちに対応します）。
- 承認/却下/修正の結果と承認者情報を監査ログに記録します。
- PDP（[P09](../identity/p09-dynamic-authz-pdp.md)）と連携し、ポリシーベースで承認要否を動的に判定します。

## 駆動変数とチューニング（程度）

全操作を承認対象にすると形骸化します（機械的にクリックするだけになります）。影響度×可逆性×実績精度の3軸で対象を絞り、信頼の蓄積に応じて自動化率を段階的に上げる設計にしましょう。初期は広めに承認を求め、精度実績が一定水準を超えた操作から順次自動化（HITL→HOTL→全自動）に移行していきます。

## ユースケース

- Salesforceの営業エージェントが100万円超の見積書を作成した際、Slack経由で上長に承認依頼を送り、承認後に初めて顧客へ送付します。
- ServiceNowのITエージェントが本番環境のサーバー設定変更を提案した際、変更内容・影響範囲・ロールバック手順を提示し、SRE担当の承認を経て実行します。
- Zendesk の顧客対応エージェントが返金処理を行う際、金額と理由を提示して承認を求めます。

## 実装メモ

承認ワークフロー（Slack Bolt / カスタムUI）、永続ジョブ状態（Temporal / Step Functions）、リスクスコアリング、監査ログ。
