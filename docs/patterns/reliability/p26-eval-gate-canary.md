# P26 · EVAL-GATE-CANARY — 評価ゲート / シャドー / カナリア

!!! abstract "一言で"
    エージェントの変更（モデル切替・プロンプト変更・ロジック修正）を本番投入前にevalで検証し、本番ではカナリア/シャドー展開で段階的にリリースするパターンです。LLMはバージョンアップで挙動が変わる特性を持つため、ソフトウェアのCI/CDに相当する品質ゲートをAI特有の評価軸で構築します。

## 解決する問題

- **サイレントな品質劣化**: モデルアップグレードやプロンプト変更で、テストされない品質劣化が本番に入ります。
- **モデルアップグレードのリスク**: 新モデルが特定のタスクで前バージョンより劣化する可能性があります。
- **プロンプト変更の副作用**: あるケースの改善が別のケースの品質を損なうことがあります。

## 選定条件

**採用する場合（When to use）**

- モデル/プロンプト/エージェントを変更する場合（LLM交換で挙動が変わる特性への備え）。
- モデル/プロンプト/エージェントの変更を行うすべての本番環境。

**採用しない場合（When NOT）**

- 特になし（本番運用であれば変更時に原則適用）。

## 構造

- **eval gate（CI組込）**: ゴールデンデータセット＋自動評価（正確性・安全性・形式遵守・トーン）をCIパイプラインに組み込み、回帰を検知します。
- **シャドー展開**: 新モデル/プロンプトを裏で並走させ、既存の出力と比較します（ユーザーには影響なし）。
- **カナリア展開**: 少量のトラフィック（目安として1〜5%を起点）を新版に流し、メトリクスを監視します。
- 回帰を検知したらロールバックします。全面展開はカナリア期間のメトリクスが基準を満たした後に行います。

## 相反における立ち位置

高速・大量・回帰検知 → LLM-as-judge（自動評価）。最終品質・安全性の判定 → 人手評価。併用が前提で、LLM評価は人手評価との相関を定期的に検証し、乖離が大きくなった場合は評価基準を再調整します。

## ユースケース

- Zendesk の顧客対応エージェントのプロンプトを改善する際、eval gate で100件のゴールデンデータセットに対する正確性・安全性スコアが閾値を満たすことをCI で確認してからデプロイします。
- 全社AIゲートウェイのモデルをGPT-4oからClaude Sonnetに切り替える際、シャドー展開で1週間並走させ、出力品質を比較してからカナリア展開に進みます。

## 実装メモ

Promptfoo等のeval フレームワーク、LLM-as-judge＋人手評価、特徴フラグ（LaunchDarkly等）、A/B・シャドー基盤。
