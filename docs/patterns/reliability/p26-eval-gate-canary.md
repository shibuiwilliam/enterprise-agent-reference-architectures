# P26 · EVAL-GATE-CANARY — 評価ゲート / シャドー / カナリア

!!! abstract "メタ"
    **カテゴリ**: 信頼性/運用 ／ **対象**: 両方 ／ **成熟度**: 標準 ／ **関連**: [P37](../governance/p37-prompt-supply-chain.md), [P40](../regulation/p40-feedback-flywheel.md)

**トリガ（採用検討の条件）**: モデル/プロンプト/エージェントを変更する（LLM交換で挙動が変わる特性への備え）。

## 概要

エージェントの変更（モデル切替・プロンプト変更・ロジック修正）を本番投入前にevalで検証し、本番ではカナリア/シャドー展開で段階的にリリースするパターン。LLMはバージョンアップで挙動が変わる特性を持つため、ソフトウェアのCI/CDに相当する品質ゲートをAI特有の評価軸で構築する。

## 設計

- **eval gate（CI組込）**: ゴールデンデータセット＋自動評価（正確性・安全性・形式遵守・トーン）をCIパイプラインに組み込み、回帰検知する。
- **シャドー展開**: 新モデル/プロンプトを裏で並走させ、既存の出力と比較する（ユーザーには影響なし）。
- **カナリア展開**: 少量のトラフィック（目安として1〜5%を起点）を新版に流し、メトリクスを監視する。
- 回帰を検知したらロールバックする。全面展開はカナリア期間のメトリクスが基準を満たした後に行う。

## 解決する課題

- **サイレントな品質劣化**: モデルアップグレードやプロンプト変更で、テストされない品質劣化が本番に入る。
- **モデルアップグレードのリスク**: 新モデルが特定のタスクで前バージョンより劣化する可能性。
- **プロンプト変更の副作用**: あるケースの改善が別のケースの品質を損なう。

## ユースケース

- Zendesk の顧客対応エージェントのプロンプトを改善する際、eval gate で100件のゴールデンデータセットに対する正確性・安全性スコアが閾値を満たすことをCI で確認してからデプロイする。
- 全社AIゲートウェイのモデルをGPT-4oからClaude Sonnetに切り替える際、シャドー展開で1週間並走させ、出力品質を比較してからカナリア展開に進む。

## 向き / 不向き

- **向き**: モデル/プロンプト/エージェントの変更を行うすべての本番環境。
- **不向き**: 特になし（本番運用であれば変更時に原則適用）。

## 要素技術

Promptfoo等のeval フレームワーク、LLM-as-judge＋人手評価、特徴フラグ（LaunchDarkly等）、A/B・シャドー基盤。

!!! note "相反する仕組みの選定（LLM-judge vs 人手評価）"
    高速・大量・回帰検知 → LLM-as-judge（自動評価）。最終品質・安全性の判定 → 人手評価。併用が前提で、LLM評価は人手評価との相関を定期的に検証し、乖離が大きくなった場合は評価基準を再調整する。
