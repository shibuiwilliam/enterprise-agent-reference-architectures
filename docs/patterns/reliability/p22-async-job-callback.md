# P22 · ASYNC-JOB-CALLBACK — 非同期ジョブ＋負荷制御

!!! abstract "一言で"
    長時間・高コストのエージェント処理をリクエスト/レスポンスから切り離し、ジョブとして非同期に実行するパターン。即時にジョブIDを返却し、完了時にWebhookや通知で結果を届ける。HTTPタイムアウトやUXブロッキングの問題を構造的に解消するとともに、優先度キュー・バックプレッシャ・オフライン/オンライン分離で負荷を包括的に制御する。

## 解決する問題

- **長セッションのHTTPタイムアウト**: 数十秒〜数分のエージェント処理でHTTP接続がタイムアウトする。
- **UXブロッキング**: ユーザーが処理完了を画面で待ち続ける体験の悪化。
- **スパイク時の全体崩壊**: 一時的な負荷急増で全ユーザーのレイテンシが悪化・タイムアウトする。
- **コスト暴走**: スパイク時のLLM呼び出しコストが制御不能になる。
- **重要処理の巻き添え**: 低優先度の大量バッチ処理が、リアルタイム対話の品質を損なう。
- **「全部リアルタイム」のコスト/レイテンシ破綻**: すべての処理をリアルタイムで行うと、コストとレイテンシが共に制御不能になる。

## 選定条件

**採用する場合（When to use）**

- 処理が数十秒を超える／バッチ的な処理／結果を後で受け取れる構成が可能。
- 処理が数十秒超、バッチ的処理、イベント駆動（[P07](../integration/p07-event-driven-agent.md)）、大量並列処理。
- 数万〜数百万規模の顧客向けでスパイクのある環境、マルチテナント環境。
- 重い分析処理と即応の対話の両方が必要な業務。

**採用しない場合（When NOT）**

- 会話的・数秒で完結する対話（ジョブ化のオーバーヘッドが体験を損なう）。
- 負荷が常に低く一定の環境（バックプレッシャ制御のオーバーヘッドが見合わない）。
- どちらか一方（リアルタイムのみ or バッチのみ）で足りる場合（オフライン/オンライン分離が不要）。

## 構造

- リクエスト受信時にジョブIDを即時返却し、処理をバックグラウンドキューに投入する。
- 進捗をSSE/WebSocketでストリーム通知、またはポーリングAPIで公開する。
- 完了時にWebhook/Slack通知/メールでコールバックする。
- タイムアウト・リトライ・DLQ（Dead Letter Queue）で障害耐性を確保する。

### バックプレッシャと負荷制御

- **優先度キュー**: リアルタイム対話 > バックグラウンド処理の優先度付け。
- **容量超過時の制御**: バックグラウンド処理を後回し / 縮退モデルに切替 / 「混雑中」応答を即返却。
- **アドミッションコントロール**: 受付段階から過負荷を検知し、受け入れ可能な量に制限する。
- テナント別クォータ（[P02](../trust-boundary/p02-multi-tenant-isolation.md)）と連動し、特定テナントの暴走が他テナントを巻き込まないようにする。

### オフライン・バッチ知性とオンライン知性の分離

- **オンライン**: 低レイテンシ・高可用・キャッシュ重視。軽量モデルで即応。
- **オフライン**: 大規模・高精度・コスト効率重視。バッチ割引・大型モデル・多段推論を活用。夜間/低負荷時間帯に実行。
- 結果ストア（データベース / キャッシュ）を介して連携し、バッチ結果をオンラインエージェントが参照する。

## 相反における立ち位置

会話的・数秒・往復が価値 → 同期。数十秒超・バッチ・イベント駆動・大量並列 → 非同期。実務では「同期ストリーミングで体感を隠す＋裏で長処理はジョブ化」の併用が多い。

!!! note "相反する仕組みの選定（オンライン vs オフライン）"
    即時性・予測不能な質問 → オンライン。大量・重い分析・事前計算可能・コスト効率重視 → オフライン。多くの実務では「バッチで前処理＋オンラインで仕上げ」の組み合わせが最適。

## ユースケース

- Salesforceの全顧客データを横断分析するエージェントタスクをジョブ化し、完了後にSlackで結果サマリを通知する。ユーザーは待機不要で他の業務を続けられる。
- Zendesk の過去チケット一括分析（数千件のカテゴリ再分類）をバッチジョブとして投入し、進捗をダッシュボードで確認する。
- Shopify連携のECサポートエージェントで、セール開始時のアクセス急増に対し、リアルタイムの顧客対応を最優先し、バックグラウンドのレポート生成は後回しにする。
- マルチテナントの顧客対応サービスで、特定テナントの異常な大量リクエストをレート制限し、他テナントへの影響を防ぐ。
- 夜間バッチで全顧客の解約予兆スコアを大型モデルで計算し、結果ストアに格納。日中に営業エージェントが「解約リスクの高い顧客は？」と聞かれた際、事前計算済みの結果を即座に参照する。
- 週次で全Jiraチケットの傾向分析・技術的負債の評価をバッチ実行し、オンラインの開発支援エージェントが「最も対処すべき技術的負債は？」に即答する。

## 実装メモ

メッセージキュー（SQS / RabbitMQ / Kafka）、ワークフローエンジン（Temporal / Step Functions）、SSE/WebSocket、DLQ、優先度キュー、アドミッションコントロール、レート制限（Token Bucket / Sliding Window）、プロバイダレート制限との協調、バッチ処理基盤（Airflow / Step Functions）、結果ストア（Redis / DynamoDB）、バッチ割引API、スケジューラ。
