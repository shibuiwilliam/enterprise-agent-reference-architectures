# P22 · ASYNC-JOB-CALLBACK — 非同期ジョブ＋コールバック

!!! abstract "一言で"
    長時間・高コストのエージェント処理をリクエスト/レスポンスから切り離し、ジョブとして非同期に実行するパターン。即時にジョブIDを返却し、完了時にWebhookや通知で結果を届ける。HTTPタイムアウトやUXブロッキングの問題を構造的に解消する。

## 解決する問題

- **長セッションのHTTPタイムアウト**: 数十秒〜数分のエージェント処理でHTTP接続がタイムアウトする。
- **UXブロッキング**: ユーザーが処理完了を画面で待ち続ける体験の悪化。
- **スパイク時の安定性**: 大量リクエストを直接処理せずキューで平準化する（[P29](../latency/p29-backpressure-load-shedding.md) と連動）。

## 選定条件

**採用する場合（When to use）**

- 処理が数十秒を超える／バッチ的な処理／結果を後で受け取れる構成が可能。
- 処理が数十秒超、バッチ的処理、イベント駆動（[P07](../integration/p07-event-driven-agent.md)）、大量並列処理。

**採用しない場合（When NOT）**

- 会話的・数秒で完結する対話（ジョブ化のオーバーヘッドが体験を損なう）。

## 構造

- リクエスト受信時にジョブIDを即時返却し、処理をバックグラウンドキューに投入する。
- 進捗をSSE/WebSocketでストリーム通知、またはポーリングAPIで公開する。
- 完了時にWebhook/Slack通知/メールでコールバックする。
- タイムアウト・リトライ・DLQ（Dead Letter Queue）で障害耐性を確保する。

## 相反における立ち位置

会話的・数秒・往復が価値 → 同期。数十秒超・バッチ・イベント駆動・大量並列 → 非同期。実務では「同期ストリーミングで体感を隠す＋裏で長処理はジョブ化」の併用が多い。

## ユースケース

- Salesforceの全顧客データを横断分析するエージェントタスクをジョブ化し、完了後にSlackで結果サマリを通知する。ユーザーは待機不要で他の業務を続けられる。
- Zendesk の過去チケット一括分析（数千件のカテゴリ再分類）をバッチジョブとして投入し、進捗をダッシュボードで確認する。

## 実装メモ

メッセージキュー（SQS / RabbitMQ / Kafka）、ワークフローエンジン（Temporal / Step Functions）、SSE/WebSocket、DLQ。
