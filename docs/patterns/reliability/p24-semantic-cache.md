# P24 · SEMANTIC-CACHE — セマンティック / プロンプトキャッシュ

!!! abstract "一言で"
    意味的に類似した過去のリクエストに対して、生成済みの結果を再利用するパターンです。入力をベクタ化し類似度検索でキャッシュヒットを判定する「セマンティックキャッシュ」と、LLMプロバイダが提供する共通プレフィクス再利用の「プロンプトキャッシュ」の両方を活用します。

## 解決する問題

- **「1リクエストが高い」コスト特性**: LLM呼び出しのトークンコストが高く、定型的な繰り返しで費用が嵩みます。
- **定型反復のレイテンシ**: 毎回LLMを呼び出すことで不要な待ち時間が発生します。

## 選定条件

**採用する場合（When to use）**

- 定型質問の繰り返しが多い／コスト・レイテンシ削減が課題の場合。
- FAQ的・定型的な質問が多い環境、コスト・レイテンシ削減が優先される場面。

**採用しない場合（When NOT）**

- 高度に個別化された回答が必要な場面、リアルタイム性が命の用途（誤ヒットのリスクが高くなります）。

## 構造

- 入力を埋め込みベクタに変換し、閾値以上の類似度でヒットした場合にキャッシュ応答を返します。
- プロバイダのプロンプトキャッシュ（共通システムプロンプトのプレフィクス再利用）を併用し、トークンコストを削減します。
- 鮮度要件に応じたTTL（Time To Live）を設定し、古い回答の再利用を防ぎます。
- マルチテナント環境では必ずテナント名前空間で分離してください（[P02](../trust-boundary/p02-multi-tenant-isolation.md)）。

## 駆動変数とチューニング（程度）

類似度閾値が緩いと別の質問に同じ答えを返す誤ヒットが発生します。機密データ・個別化された回答にはキャッシュを無効化するか閾値を厳格にし、公開FAQ的な質問には緩めに設定しましょう。鮮度が重要な情報はTTLを短縮してください。マルチテナント環境ではテナント名前空間で分離し、クロステナント漏洩を防ぎます（[P02](../trust-boundary/p02-multi-tenant-isolation.md)）。

## ユースケース

- Zendesk の顧客対応エージェントで「返品方法を教えてください」「配送状況を確認したい」等のFAQ的な定型質問にキャッシュ応答を返し、LLM呼び出しコストを削減します。
- 社内ヘルプデスクで「VPN接続方法」「経費精算の手順」等の頻出質問に対し、検証済みの回答をキャッシュから即時返却します。

## 実装メモ

セマンティックキャッシュ（GPTCache / Redis + ベクタ検索）、プロンプトキャッシュ（プロバイダ提供機能）、埋め込みモデル、TTL管理。
