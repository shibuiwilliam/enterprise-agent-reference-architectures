# P10 · CONTEXT-FIREWALL — コンテキスト・ファイアウォール＋データ主権

!!! abstract "一言で"
    取り込んだ各情報断片のACL（アクセス制御リスト）を保持し、合流点で最も厳しい権限に縮退させるパターン。出力時に現在のユーザーが閲覧できない由来の内容を遮断する。さらに、データの所在・処理リージョン・利用するLLMモデルを地域別に分離・固定し、越境規制への準拠と機密データの学習利用リスクの排除を構造的に実現する。データ分類およびデータ主権に関する設計判断は人間のレビューが必須。

## 解決する問題

- **最弱リンク漏洩**: 複数ソースを横断集約した際に、最も権限の弱いソース経由で機密情報が漏れる。
- **間接インジェクションの無害化**: 取り込んだ外部文書内の悪意ある指示が、他ソースの機密データを引き出すことを防ぐ。
- **ACL不整合**: ソースごとに異なるアクセス制御を、集約後も一貫して維持できない。
- **越境規制違反**: EU居住者のPIIを日本/米国のLLMで処理し、GDPR違反になる。
- **機密データの学習利用リスク**: モデルプロバイダの学習データに機密情報が含まれるリスク。

## 選定条件

**採用する場合（When to use）**

- 複数システムのデータを混ぜてコンテキスト/メモリにする／機密・規制データを含む。
- 複数ソースを集約する全本番環境、機密・規制データを含むコンテキスト。
- 越境規制（APPI / GDPR 等）に該当する／顧客PIIを扱う／グローバル展開している。
- グローバル展開・規制業種（金融・医療・公共）・PIIを扱うエージェント。

**採用しない場合（When NOT）**

- 単一権限境界で完結する用途（フィルタのオーバーヘッドが不要）。
- 単一国内・非個人データのみの用途（データ主権の地域分離は不要）。

## 構造

### ACL継承付き集約

- 各チャンク（文書断片・データレコード）にソース・データ分類・ACLメタデータを付与する。
- リトリーバル（RAG検索）では閲覧可能な範囲で**事前フィルタ**を行う（post-filter はフィルタ前のデータがモデルに到達するため漏洩リスクがある）。
- 機密分類に応じて送信先モデル（社内専用/外部API）を分岐する。
- 出力前に引用元の閲覧可否を最終チェックし、権限のない由来情報を含む回答を遮断する。

### データ主権・地域分離

- データの地域属性（ユーザーの居住国・データの生成国）に応じて処理リージョンとLLMエンドポイント（同一リージョン・越境なし契約）を選択する。
- 機密データにはzero data retention（ZDR）設定を適用し、モデルプロバイダ側での学習利用を防ぐ。
- 地域別の保持/削除ポリシーを適用する（GDPR の忘れられる権利等）。
- ゲートウェイ（[P03](../integration/p03-ai-gateway.md)）で地域ルーティングを実装する。

## 相反における立ち位置

コンテキストの豊かさと安全性は相反する。判断軸はデータ分類。公開〜社内一般のデータはリッチに集約し、機密〜規制データは厳格に事前フィルタして外部モデルへの送信を禁止する。一律に厳格にすると有用性が損なわれ、一律に緩くすると漏洩リスクが増大するため、**データ分類に応じて段階的に制御**する。

## ユースケース

- 営業エージェントがSalesforce・Box・Slack・社内Wikiのデータを横断的に参照する際、ユーザーがBox上で閲覧権限のないフォルダの内容は検索結果にも回答にも含まれないようにする。
- Zendesk の顧客対応エージェントが社内ナレッジベースを参照する際、「社内限定」ラベルの情報を顧客向け回答に含めないようフィルタする。
- グローバル企業でEU顧客のPIIを含む問い合わせをEUリージョンのLLMエンドポイントでのみ処理し、日本拠点の従業員が参照する際もEUリージョンで推論を実行する。
- 金融機関で顧客の取引データをZDR設定のモデルAPIでのみ処理し、データ保持ポリシーに準拠する。

## 実装メモ

ベクタDBのメタデータフィルタ（行レベルセキュリティ）、データ分類（Microsoft Purview 等）、PII検出/マスキング、入力サニタイズ、引用必須化、リージョナルデプロイ、ZDR（Zero Data Retention）設定、地域ルーティング（ゲートウェイ [P03](../integration/p03-ai-gateway.md)）。
