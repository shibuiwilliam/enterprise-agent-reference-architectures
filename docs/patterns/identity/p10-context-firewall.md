# P10 · CONTEXT-FIREWALL — コンテキスト・ファイアウォール（ACL継承付き集約）

!!! abstract "メタ"
    **カテゴリ**: ID/認可 ／ **対象**: 両方 ／ **成熟度**: 標準 ／ **関連**: [P08](p08-token-exchange-obo.md), [P12](../memory/p12-layered-memory.md), [P38](../regulation/p38-data-sovereignty.md)

**トリガ（採用検討の条件）**: 複数システムのデータを混ぜてコンテキスト/メモリにする／機密・規制データを含む。

## 概要

取り込んだ各情報断片のACL（アクセス制御リスト）を保持し、合流点で最も厳しい権限に縮退させるパターン。出力時に現在のユーザーが閲覧できない由来の内容を遮断する。横断コンテキストの最大リスクである「最弱リンク漏洩」をアーキテクチャレベルで防ぐ。データ分類に関する設計判断は人間のレビューが必須。

## 設計

- 各チャンク（文書断片・データレコード）にソース・データ分類・ACLメタデータを付与する。
- リトリーバル（RAG検索）では閲覧可能な範囲で**事前フィルタ**を行う（post-filter はフィルタ前のデータがモデルに到達するため漏洩リスクがある）。
- 機密分類に応じて送信先モデル（社内専用/外部API）を分岐する。
- 出力前に引用元の閲覧可否を最終チェックし、権限のない由来情報を含む回答を遮断する。

## 解決する課題

- **最弱リンク漏洩**: 複数ソースを横断集約した際に、最も権限の弱いソース経由で機密情報が漏れる。
- **間接インジェクションの無害化**: 取り込んだ外部文書内の悪意ある指示が、他ソースの機密データを引き出すことを防ぐ。
- **ACL不整合**: ソースごとに異なるアクセス制御を、集約後も一貫して維持できない。

## ユースケース

- 営業エージェントがSalesforce・Box・Slack・社内Wikiのデータを横断的に参照する際、ユーザーがBox上で閲覧権限のないフォルダの内容は検索結果にも回答にも含まれないようにする。
- Zendesk の顧客対応エージェントが社内ナレッジベースを参照する際、「社内限定」ラベルの情報を顧客向け回答に含めないようフィルタする。

## 向き / 不向き

- **向き**: 複数ソースを集約する全本番環境、機密・規制データを含むコンテキスト。
- **不向き**: 単一権限境界で完結する用途（フィルタのオーバーヘッドが不要）。

## 要素技術

ベクタDBのメタデータフィルタ（行レベルセキュリティ）、データ分類（Microsoft Purview 等）、PII検出/マスキング、入力サニタイズ、引用必須化。

!!! note "相反する仕組みの選定（豊かさ vs 安全性）"
    コンテキストの豊かさと安全性は相反する。判断軸はデータ分類。公開〜社内一般のデータはリッチに集約し、機密〜規制データは厳格に事前フィルタして外部モデルへの送信を禁止する。一律に厳格にすると有用性が損なわれ、一律に緩くすると漏洩リスクが増大するため、**データ分類に応じて段階的に制御**する。
