# P09 · DYNAMIC-AUTHZ-PDP — 動的認可（PDP / PBAC）

!!! abstract "一言で"
    各アクションを実行直前に外部ポリシーエンジン（PDP: Policy Decision Point）で評価するパターンです。主体×アクション×リソース×文脈の組み合わせで「許可/拒否/要承認」を判定し、リスクに応じた細粒度の認可を実現します。認可ポリシーの設計は人間のレビューが必須です。

## 解決する問題

- **権限のハードコード散在**: アプリケーションコードに散在する認可ロジックの管理が困難になります。
- **リスク応分の制御**: 全操作を一律に許可/拒否するのではなく、影響度に応じた制御ができません。
- **規制証跡の欠如**: 「なぜその操作が許可されたか」を後から説明できません。

## 選定条件

**採用する場合（When to use）**

- 操作の影響度が読み取り〜送金まで幅広い／規制対応の証跡が要る。
- 操作の影響度が幅広い組織、規制対応で証跡が必要な環境。

**採用しない場合（When NOT）**

- 全アクションが等価で低リスクな閉じた用途（ポリシー管理のオーバーヘッドが見合いません）。

## 構造

- エージェントのツール呼び出しを横取りするPDP（OPA / Cedar）を配置し、実行前にポリシー評価を行います。
- policy-as-code で「金額 > 10万円は要承認」「機密ラベルのデータは外部モデルに送信禁止」などを宣言的に定義します。
- 判定結果は「許可/拒否/要承認（[P18](../orchestration/p18-hitl-approval-gate.md) へエスカレーション）」の3値です。
- 判定ログは不変の監査証跡として記録し、規制対応に活用します。

## ユースケース

- Salesforce上の営業エージェントが商談金額を更新する際、金額が一定額を超える場合のみ上長承認を要求するポリシーをOPAで定義します。低額の更新は即時実行されます。
- ServiceNowのITエージェントが本番環境の変更操作を行う際、「業務時間外」「変更影響範囲が広い」等の文脈を加味して承認要否を動的に判定します。

## 実装メモ

OPA（Rego）、Amazon Verified Permissions（Cedar）、ABAC/ReBAC、OpenFGA/SpiceDB（組織階層・所属の表現に強い）。
