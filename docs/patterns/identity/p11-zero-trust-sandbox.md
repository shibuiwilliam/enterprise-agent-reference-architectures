# P11 · ZERO-TRUST-SANDBOX — ゼロトラスト・メッシュ＋サンドボックス実行

!!! abstract "一言で"
    エージェントが生成したコードやツール連鎖を、ネットワーク・ファイルシステム・外部送信を制限したサンドボックス内で実行するパターンです。エージェント間の通信も相互認証を行い、信頼を暗黙に仮定しません。生成コードによる破壊・データ持ち出しを物理的に不可能にします。

## 解決する問題

- **生成コードによる破壊**: ハルシネーションや悪意あるプロンプトにより、破壊的なコードが実行されます。
- **データ持ち出し**: 生成コードが機密データを外部エンドポイントへ送信します。
- **サプライチェーン的リスク**: エージェント間の連鎖呼び出しで、ある1つの侵害が全体に波及します。

## 選定条件

**採用する場合（When to use）**

- エージェントにコード実行や自動アクション（ファイル操作・API呼び出し・スクリプト実行）を許可します。
- コード実行・自動アクション（ファイル操作・スクリプト実行・外部API呼び出し）を許可するエージェント。

**採用しない場合（When NOT）**

- 検索・要約のみのエージェント（サンドボックスのオーバーヘッドが見合いません）。

## 構造

- 生成コードは隔離コンテナ（egress許可リスト付き）で実行し、許可されたエンドポイント以外への外部送信を遮断します。
- エージェント間呼び出しはmTLS＋ワークロードID（SPIFFE）で相互認証します。マルチエージェント構成（[P15](../orchestration/p15-single-vs-multi.md)）では、各エージェントの権限を分離します。
- 外部送信は明示許可制とし、デフォルトは全遮断です。
- ファイルシステムは一時領域のみ書き込み可能で、永続ストレージへの書き込みはAPIを経由させます。

## ユースケース

- データ分析エージェントがPythonコードを生成して社内データを集計する際、gVisorコンテナ内で実行し、結果のみをユーザーに返します。外部への通信は一切できません。
- 複数エージェントが連携する開発支援システムで、コード生成エージェントとテスト実行エージェントがそれぞれ独立したサンドボックスで動作し、mTLSで認証された経路でのみ連携します。

## 実装メモ

gVisor、Firecracker、egressプロキシ（Envoy / Istio）、SPIFFE/SPIRE、Kubernetes ネットワークポリシー。
