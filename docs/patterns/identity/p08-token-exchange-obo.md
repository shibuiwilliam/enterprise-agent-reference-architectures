# P08 · TOKEN-EXCHANGE-OBO — エージェントID とトークン交換（On-Behalf-Of）

!!! abstract "メタ"
    **カテゴリ**: ID/認可 ／ **対象**: 両方 ／ **成熟度**: 基盤 ／ **関連**: [P03](../integration/p03-ai-gateway.md), [P09](p09-dynamic-authz-pdp.md), [P10](p10-context-firewall.md)

**トリガ（採用検討の条件）**: 複数SaaSをユーザー権限で操作する（ほぼ全本番に該当）。

## 概要

エージェントは自分自身のID（ワークロードID）と代理ユーザーのIDの両方を持ち、各SaaSへはユーザー権限に縮退したスコープ付きトークンで到達する。「誰の代理で・どのスコープで」を全コールに伝播し、混乱した代理人問題を防ぐ。認証・認可設計は人間のレビューが必須。

## 設計

- 社内IdP（Okta / Entra ID）でSSO→ゲートウェイ（[P03](../integration/p03-ai-gateway.md)）が受領→下流SaaSへは **OAuth 2.0 Token Exchange（RFC 8693）/ OBO** で最小スコープのトークンを取得。
- エージェント自身の操作（ログ記録・内部通信等）はワークロードIDで分離し、ユーザー操作と混同しない。
- 「誰の代理で・どのスコープか」をトークンクレームとして全コールに伝播し、下流でも検証可能にする。
- 短命トークンを基本とし、長時間ジョブでは自動更新で最小権限を維持する。

## 解決する課題

- **混乱した代理人問題**: エージェントが自身の権限で操作し、ユーザーの意図を超えたアクセスが発生する。
- **過剰権限**: 「サービスアカウント全権」で全SaaSにアクセスするアンチパターン。
- **監査での主体の曖昧さ**: 「誰の指示で・誰の権限で」が不明確で監査に耐えない。

## ユースケース

- Slack上のエージェントハブに「Salesforceの商談を更新して」と依頼すると、ユーザーのOkta認証→Token Exchange→Salesforce APIへユーザー権限で商談を更新する。エージェント自身はSalesforceの管理者権限を持たない。
- Workdayの人事データ参照エージェントが、依頼者の閲覧権限に縮退したトークンで給与情報にアクセスし、本人に見えないデータは取得できない。

## 向き / 不向き

- **向き**: ほぼ全本番環境。複数SaaSをユーザー権限で操作するすべてのケース。
- **不向き**: 内部・読み取り専用・非機密で、ここまで厳格な権限委譲が不要な場合がある（ただし本番では原則適用）。

## 要素技術

OAuth 2.0 Token Exchange（RFC 8693）、OIDC、SPIFFE/SPIRE（ワークロードID）、Okta/Entra ID/Auth0 の OBO フロー、短命トークン。

!!! warning "程度の判断（トークン寿命）"
    短命トークンは長セッションと相反する。セッションが数十分〜数時間に及ぶ場合、短命トークン×自動更新で対応する。トークン寿命は「目安として5〜15分」を起点に、ジョブの平均処理時間と更新オーバーヘッドを実測して調整する。長時間ジョブでもセッション全体で単一の長寿命トークンを使うのではなく、ステップごとに必要最小限のスコープで再取得する設計が望ましい。
