# 設計プロトコル（コーディングエージェント向け）

コーディングエージェントは、エンタープライズの要件を受け取ったら **この順序で** 設計し、
[提案テンプレート](proposal-template.md) を埋めて人間に提案してください。

```text
STEP 1  要件抽出
  以下の軸を、ユーザーへの質問または既存情報から確定します。
  欠けている軸は「未確定」として記録し、後で人間に確認してください。
    a. 利用者種別 ........ 従業員 / 顧客 / 両方        → P01,P02
    b. 既存システム ...... 連携対象のSaaS・独自システム
    c. データ分類 ........ 公開 / 社内 / 機密 / 規制(PII等) → P10,P25
    d. 副作用の可逆性 .... 読み取りのみ / 可逆 / 不可逆      → P17,P18,P20
    e. 頻度・反復性 ...... 一回性 / 高頻度・形式安定         → P17
    f. レイテンシ要件 .... 対話即応 / 数十秒許容 / バッチ    → P22
    g. 規模・負荷特性 .... 同時実行数・スパイク有無          → P02,P22
    h. 規制・地域 ........ 越境規制・業界規制               → P10,P25

STEP 2  信頼境界の確定（最優先・例外なく最初に行います）
  利用者種別から P01(信頼境界の二層分離) を適用します。顧客向け要素があれば
  P02(マルチテナント分離) を必須候補に入れてください。これが後続の全判断の制約になります。

STEP 3  候補パターンの選定
  各パターンの「トリガ」を要件と突合し、合致したものを候補に追加します。
  パターン索引を一次フィルタに使ってください。

STEP 4  相反の解消
  候補に競合（同期 vs 非同期、シングル vs マルチ等）があれば、
  「相反する仕組みの選定基準」で判断軸を引き、選定理由を記録してください。
  選定理由は提案にそのまま転記します（人間が検証できるようにするためです）。

STEP 5  程度（パラメータ）の初期設定
  「程度の選定基準」で、タイムアウト・リトライ・ログ粒度・自律レベル等の
  初期値を設定します。「実測で調整する前提」を必ず明記してください（断定しないでください）。

STEP 6  アーキテクチャ組み立て
  参照アーキテクチャの層構造（信頼境界→体験→オーケストレーション→ツール/データ
  →ID/認可→運用→ガバナンス）に候補を配置し、層の抜けを点検してください。

STEP 7  提案生成
  提案テンプレートを埋めてください。各パターンはIDで参照します。

STEP 8  人間へのエスカレーション（必須）
  以下は自動で確定せず、提案内で明示し人間の承認を求めてください。
    - 認証・認可・トークン委譲の設計（P08,P09）
    - 機密/規制データの取り扱いと外部モデル送信可否（P10）
    - 不可逆な副作用を伴う自動化（P17,P18）
    - カタログ外の独自構成（extension として根拠とリスクを併記してください）
```

## エージェントが守る規約

- パターンは必ず `P0X` または `SLUG` で引用してください。出典が曖昧な主張はしないでください。
- カタログに無い設計は `extension:` と明記し、なぜ既存パターンで足りないかを書いてください。
- 「程度」は断定せず、**初期値 ＋ 計測指標 ＋ 調整方針** の3点で書いてください。
- セキュリティ・不可逆操作・規制に関わる判断は、自信があっても人間レビューへ回してください。
- 不確実・未確定の要件は隠さず「未確定」として提案に列挙してください。
