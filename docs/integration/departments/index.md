# 部門別適用例

エンタープライズAIエージェントを実際の業務で動かすとき、「どのパターンを組み合わせるか」は部門ごとに大きく変わる。リスクの種類、扱うデータの機密度、連携する SaaS の特性、承認の構造——これらはすべて部門によって異なるからだ。

たとえば営業部門のエージェントは商談金額や競合情報という高感度データを扱い、CRM への書き込みには本人権限の厳密な適用が求められる。一方で人事部門のエージェントは給与・評価・懲戒という法的リスクを伴うデータを扱い、二者承認や DLP（データ損失防止）が不可欠になる。カスタマーサポートは顧客面と社内面の境界分離が最優先であり、エンジニアリングはコード実行サンドボックスとインシデント対応が重要になる。

このセクションでは、各部門に「なぜそのパターンが必要か」を具体的な適用場面とともに解説する。パターン単体の説明は各パターンページを参照し、ここでは部門文脈での組み合わせ理由に集中する。

## 部門一覧

| 部門 | 概要 | 主要パターン |
|---|---|---|
| [Sales Agent](sales.md) | 商談管理・顧客情報・見積承認を横断。権限忠実性と CRM 操作の安全性が核心。 | ID-2, ID-4, IN-2, KM-5, RT-5, RT-4 |
| [HR Agent](hr.md) | 給与・評価・異動という極めて機密度の高いデータを扱う。二者承認と DLP が必須。 | KM-4, KM-6, RT-4, GV-4, RT-6 |
| [Customer Support Agent](customer-support.md) | 顧客面と社内面の完全分離が大前提。自動回答と要承認操作の段階的自律が鍵。 | ID-1, RT-3, KM-1, RT-7, RT-9 |
| [Engineering Agent](engineering.md) | コード実行・CI/CD・本番デプロイを扱う。サンドボックス隔離とポリシー自動チェックが重要。 | IN-1, ID-6, RT-8, OB-1, GV-9, ID-7 |
| [Executive Agent](executive.md) | 経営横断の KPI・財務・人事情報へのアクセス。厳格な監査と権限付きフェデレーションが必要。 | KM-3, KM-2, KM-6, GV-8, GV-7, OB-2 |

## パターンを選ぶ視点

部門に合ったパターン束を選ぶとき、以下の4つの軸で考えると整理しやすい。

**データ機密度**：給与・評価・経営情報など個人情報保護法・会社法上のリスクが高いデータを扱う部門ほど、DLP・スコープ分離・監査のパターンを厚くする。

**書き込みリスク**：読み取りだけでなく CRM・ERP・HR システムへの書き込みを行う場合、Command Envelope（RT-5）や SoR Write Boundary（RT-6）で操作を構造化し、Human Approval（RT-4）で承認を挟む。

**顧客接点の有無**：顧客と直接やり取りするエージェント（カスタマーサポート等）は、社内従業員向けエージェントと ID・権限・ログを完全分離する（ID-1）。

**実行環境のリスク**：コード実行や本番インフラ操作を伴うエンジニアリング用途では、Zero-Trust PDP/PEP（ID-6）とサンドボックス（IN-1 相当）による強制隔離が前提となる。
