# エンタープライズAIエージェント アーキテクチャパターン

数万人規模の従業員・顧客と、多様なSaaS（Salesforce / ServiceNow / Workday / Okta / Slack /
M365 / Box / Sansan / Google Workspace / AWS / Linear / Notion / Jira / Zendesk / Shopify /
Auth0 等）・独自開発システムが併存する企業を前提に、**AIエージェントを本番システムへ
組み込むためのアーキテクチャパターン（全30）**を体系化したカタログです。

!!! info "想定読者は2種類"
    - **人間**：アーキテクチャを学び、判断する。
    - **コーディングエージェント**：要件を受け取り、このカタログを引用しながら設計し、人間に提案する。
      → まず [設計プロトコル](usage/design-protocol.md) を読んでください。

## なぜ「普通の統合」では破綻するのか ― 4つの摩擦

AIエージェントは従来のエンタープライズ統合に4つの「異質さ」を持ち込みます。全パターンは
この摩擦への応答です。

1. **確率的主体が決定論的システムを操作する** — ハルシネーション・出力の揺らぎが不可逆操作
   （送金・契約・削除・送信）に届くとリスク化。→ 検証・承認・冪等性・補償トランザクション・確率→決定論変換。
2. **自然言語が新しい攻撃面・権限境界になる** — プロンプトインジェクション・間接インジェクション・
   混乱した代理人問題。→ アイデンティティとコンテキストの境界制御。
3. **セッションが長く・高価・不安定** — 数秒〜数十分、桁違いのコスト、上流LLMの可用性変動。
   → 非同期・ジョブ化・キャッシュ・フォールバック・コストガバナンス。
4. **横断コンテキストは価値だがそれ自体がリスク** — 複数SaaSを束ねると賢くなるが、ACLを無視すると
   最弱リンクの権限まで漏れる。→ ACL継承付きの集約。

## 横断的な設計原則

1. エージェントIDとユーザーIDを分離し、かつ常に紐付ける。
2. 権限はエージェントでなく「アクション×文脈」に対して評価する（PBAC）。
3. 読み取りと副作用ある書き込みを物理的に別経路にする（System of Record / System of Action）。
4. コンテキストは最も厳しいACLを継承する（合流点で必ず縮退）。
5. すべてのエージェント行動はトレース可能・再現試行可能にする。
6. モデルは交換される前提で抽象化する（ゲートウェイ＋eval gate で守る）。

## 設計時に最初に答えるべき5つの問い

1. 誰のための、どの信頼境界か？（従業員/顧客 → [P01](patterns/trust-boundary/p01-trust-boundary-split.md), [P02](patterns/trust-boundary/p02-multi-tenant-isolation.md)）
2. 副作用は可逆か不可逆か？（不可逆 → P17 ＋ P18 ＋ P20）
3. 頻度は高いか一回性か？（高頻度 → P17、一回性 → ランタイムLLM）
4. どのデータ分類に触れるか？（機密/規制 → P10 ＋ P25）
5. どの「程度」を、何の指標で計測して調整するか？（→ [程度の選定基準](selection/degree-criteria.md)）
