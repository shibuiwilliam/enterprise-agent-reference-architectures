# RT-D1 単一 vs マルチエージェントと分業

## 意思決定の問い

エージェントを単一構成にすべきか、マルチエージェント構成にすべきか。マルチにする場合、組織階層に沿ったHub&Spoke（RT-1）とRACIに基づく責任分担型（RT-2）のどちらの分業方式を採用するかを決めます。

「処理が複雑だからマルチエージェントにしよう」は過剰設計の入り口です。マルチにすればコストはN倍になり、レイテンシは加算され、障害点も増えます。マルチ化が正当化される唯一の基準は「企業内の責任分担が複数の部門・役割に分かれるから」です。

## 選択肢／程度

| 選択肢 | 概要 | 適用条件 |
|---|---|---|
| 単一エージェント | 1つのエージェントが全処理を担当 | 責任主体が単一部門に収まる。低遅延・低コスト要件 |
| Hub&Spoke（RT-1） | ハブが意図をルーティングし、ドメインスポークが専門処理を実行。権限は親→子で縮退 | 複数部門をまたぐ意図ルーティングが必要。組織階層に沿ったエスカレーションが必要 |
| RACIマルチエージェント（RT-2） | RACI（責任・説明責任・協議・情報共有）の構造をエージェントトポロジに写像 | 複数部門が関与し、各ステップで説明責任者が異なる業務。監査証跡でR/A/C/Iの記録が必須 |
| 段階的分離 | 単一で開始し、責任分割点を観測してからサブエージェントを分離 | プロトタイプ段階。責任境界が未確定 |

## 判断軸

- **責任分担の有無**：マルチ化の判断基準は処理の複雑さではなく、組織上の責任が複数の主体に分かれているかどうかです。RACIマトリクスを先に定義し、それに対応するエージェント構成を導出する順序を守ります。
- **コスト・レイテンシの許容度**：マルチエージェント構成ではコストがN倍、レイテンシが加算されます。対話型UIでリアルタイム応答が必要な場合は単一を維持すべきです。
- **運用経験**：マルチエージェント運用経験が乏しく障害対応コストが見通せない段階では単一を維持します。
- **ドメイン数とツールセット**：部門ごとに異なる権限境界・SaaS連携・ベクトルDBが存在する場合はHub&Spokeの価値が高くなります。
- **監査要件**：R/A/C/Iの記録が必要な規制対応業務ではRACIマルチエージェントが適合します。

## 推奨と既定値

まず単一エージェントで開始します。責任分担の境界が明確になった箇所のみ段階的に分離し、最終的にHub&SpokeまたはRACIマルチエージェントへ拡張します。

Hub&SpokeとRACIは排他ではなく補完関係にあります。Hub&Spokeのスポーク間調整にRACIの責任割り当てを組み合わせると、ドメイン間の責任境界が明確になります。

## 必要な構成要素

- **RT-1 Hub & Spoke**：ハブが意図分類とルーティングのみを担い、各スポークは自分の担当SaaSだけを扱います。権限はHub→Spokeの方向にのみ減衰して渡されます。スポークはサマリを返すだけなので、ハブのコンテキストウィンドウに全ドメインの生データが蓄積されません。要素技術＝Semantic Router、LangGraph、AutoGen、CrewAI、Pinecone、Weaviate、pgvector、Capability Registry。落とし穴＝単一メガエージェント化（全ツール・全ポリシーを1エージェントに持たせる）、セマンティックルーターの精度不足（低信頼度時のフォールバック未設計）、スポーク間の暗黙的な権限依存（ハブを経由せず直接呼び出す設計は権限縮退の一貫性を破壊します）、従業員面・顧客面の混在（ID-1違反）。 → 機械詳細は building-blocks.json[RT-1]

- **RT-2 RACI Multi-Agent**：RACIマトリクスを先に定義し、それに対応するエージェント構成を導出します。Accountable（A）は常に人間が担います。Consultedエージェントの関与は1ラウンドに限定して無限ループを防ぎます。デシジョンログは各フェーズの開始・終了時点でリアルタイムに記録します。要素技術＝LangGraph、AutoGen、Semantic Kernel、YAML/JSON RACI definition、OpenTelemetry。落とし穴＝「複雑だからマルチエージェント」という設計根拠の欠如、Accountableの空席（エージェントにAを割り当てるとミス時の責任者不在）、Cフィードバックの無限ループ、デシジョンログの事後補完。 → 機械詳細は building-blocks.json[RT-2]

## 落とし穴・アンチパターン

**「複雑だからマルチエージェント」**。タスクの難しさだけを理由にエージェントを増やすと、責任境界のないアーキテクチャが生まれます。マルチエージェントへの移行は必ずRACIマトリクスの定義から始めてください。

**単一メガエージェント**。「とりあえず1エージェントに全ツール・全ポリシーを持たせる」構成は、コンテキスト汚染・権限過多・変更影響の広域化を引き起こします。規模が小さいうちは問題が見えにくいですが、ドメイン数・ツール数が増えるにつれて破綻します。

**ケイパビリティレジストリの放棄**。スポークが増えるにつれ、どのスポークがどのツールを持つかの管理が散漫になります。レジストリを中央管理しGV-2 Agent Catalogと統合すべきです。

**Accountableの空席**。マルチエージェントシステムでAを別のエージェントに担わせたくなる誘惑があります。Aは常に人間が担います。エージェントにAを割り当てればミス発生時の責任者が不在になります。

**Cフィードバックの無限ループ**。Consultedエージェント同士が追加意見を要求し合い終わらない状態に陥ります。Cの関与を1ラウンドに限定し、Rが集約する責任を明示的に設計してください。
