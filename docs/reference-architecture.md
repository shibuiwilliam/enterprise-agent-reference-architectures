# 参照アーキテクチャ・導入順序

## 層構造（エージェントはこの順で配置を点検します）

1. **信頼境界**：P01（従業員/顧客分離）、P02（顧客はマルチテナント隔離）。全判断の最上位制約です。
2. **体験層**：P06ハブ ＋ P07イベント駆動。同じ下層を共有します。
3. **オーケストレーション層**：P16が P33 へ委譲します。複雑タスクのみ P15マルチを使います。
   副作用は P17 ＋ P18 ＋ P20、自信不足は P21 で対応します。
4. **ツール・データ層**：P04 MCP ＋ P10 ファイアウォール ＋ P12メモリ ＋ P14セマンティックレイヤー。
   鮮度は P31 で管理します。
5. **ID・認可（横断）**：P08トークン交換 ＋ P09 PDP ＋ P11サンドボックスで構成します。
6. **運用（横断）**：P03ゲートウェイに P23フォールバック・P24キャッシュ・P25トレース・P26 eval gate・
   P27コスト・P22非同期/負荷制御を集約します。
7. **ガバナンス（横断）**：P36ライフサイクル ＋ P37プロンプト管理 ＋ P35レジストリ ＋ P40フィードバックで構成します。

## 導入順序の現実解

- **フェーズ0（基盤）**：P01 ＋ P03 ＋ P08 ＋ P25。ここを飛ばすとシャドーAI・監査不能・漏洩で破綻します。
- **フェーズ1（価値）**：P06を高滞在の1業務に適用します。読み取り中心・低リスクから始めましょう。
- **フェーズ2（横断）**：P04 ＋ P10 ＋ P31でデータ統合の難所に取り組みます。
- **フェーズ3（自動化）**：P17 ＋ P18 ＋ P20 ＋ P07で副作用ある業務へ進みます。HITL→HOTLへ段階昇格していきます。
- **フェーズ4（統制）**：P09 ＋ P26 ＋ P27 ＋ P36 ＋ P35 ＋ P40でスケールに耐える組織化を行います。

## まとめ（エージェント・人間共通の指針）

これらは「正解」でなく**設計の語彙**です。自社制約（規制・既存SaaS・組織の自律性・リスク許容度）を
[程度の選定基準](selection/degree-criteria.md)・[相反する仕組みの選定基準](selection/tension-criteria.md)
に当てはめ、各パターンの「程度」を実測でチューニングすることが本質です。

PoCを本番に昇格させる分水嶺は次の3点に集約されます。

1. アイデンティティとコンテキストの境界
2. 確率的主体と不可逆操作の分離
3. 程度の継続的な計測

設計は [設計プロトコル](usage/design-protocol.md) から着手し、
信頼境界 → 可逆性 → 頻度 → データ分類 → 計測 の順で具体化していきましょう。
